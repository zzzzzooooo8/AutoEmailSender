# 全局消息提示设计

## 背景

当前前端没有统一的全局消息提示机制。成功、警告、报错和占位提示分散在多个页面与组件中，主要表现为以下几类：

- 页面内部通过局部 `message`、`error`、`feedback` 状态渲染提示文案。
- 表单字段级与区域级错误直接显示在页面或表单附近。
- 少量交互仍使用 `window.alert(...)`。
- 弹窗打开时，局部消息提示无法保证始终处于可见层级。

这导致当前体验存在几个明显问题：

- 消息样式和出现位置不统一。
- 页面内提示容易被滚动区域、弹窗层级或局部布局影响。
- 表单一次出现多条错误时，提示分散，难以形成统一反馈。
- 用户想复制、点击或仔细阅读报错时，现有形式不够稳定。

本次设计目标是在前端应用内建立统一的右下角全局消息系统，用于承接所有反馈类消息，并替换现有零散实现。

## 已确认的产品决策

- 全部反馈类消息统一迁移为右下角全局消息，不再保留字段级、区域级或页面内常驻提示作为主要反馈出口。
- 表单一次性校验出多条错误时，合并为 1 条消息显示。
- 这条聚合消息的标题固定为类似“请检查表单”，正文列出多条错误。
- 表单提交失败后不自动滚动到第一个错误字段，也不自动聚焦字段。
- 全局消息同时最多显示最近 4 条。
- 超出 4 条时，直接移除最老的一条，不做历史汇总入口，不做折叠中心。
- 消息显示顺序为上旧下新；新消息进入时位于最下方，把旧消息向上顶。
- 自动消失时长按消息重要程度和字数共同决定。
- 用户一旦与某条消息发生交互，这条消息就不再自动消失，必须手动关闭。
- 即使用户正在弹窗页面中操作，全局消息也必须正常显示。

## 目标与非目标

### 目标

- 在前端应用根部提供统一的全局消息提示出口。
- 让成功、警告、错误、表单校验失败和现有 `window.alert(...)` 占位提示都使用同一套机制。
- 保证消息在普通页面和弹窗场景下都可见。
- 让表单多错误场景可以以单条聚合消息稳定展示。
- 让用户可以复制、点击、停留查看消息，而不会被自动消失打断。
- 在不引入后端依赖的前提下，完成前端全站范围内的消息反馈收口。

### 非目标

- 不将确认类对话框改造成通知。
- 不新增“消息中心”“历史记录抽屉”或“查看更多消息”入口。
- 不在本次改动中加入自动跳转到错误字段、自动滚动到表单区或自动聚焦字段。
- 不把业务详情面板强行改造成通知，例如连接诊断结果面板、模型列表面板等。
- 不改动后端接口结构，也不新增后端消息推送机制。

## 方案比较

### 方案 A：自研全局消息中心

特点：

- 使用前端应用根部 Provider 管理消息队列。
- 使用 Portal 将消息容器挂到 `document.body`。
- 通过统一接口从各页面和组件发出消息。

优点：

- 能完整实现本次要求中的所有细节：右下角堆叠、最新在下、交互后锁定、跨弹窗显示、聚合表单错误。
- 与当前项目的 React 结构最匹配。
- 迁移路径明确，后续维护成本可控。

缺点：

- 需要自己实现消息队列、定时器、关闭动画和交互锁定。

### 方案 B：引入第三方 Toast 库并二次封装

特点：

- 利用第三方现成的通知组件做基础能力。

优点：

- 初始搭建较快。

缺点：

- 本次需求定制度高，最终仍然需要大量二次封装。
- 会引入新的依赖与适配成本。
- 对“交互后永久转手动关闭”这类规则未必天然匹配。

### 方案 C：事件总线式轻量通知服务

特点：

- 在 React 组件树外维护通知入口，通过全局服务或事件总线发消息。

优点：

- 任意位置调用都比较直接。

缺点：

- 对当前项目而言偏重，类型边界和调试路径更分散。
- 与现有 React 状态结构整合成本高于 Provider 方案。

## 推荐方案

推荐采用方案 A：自研全局消息中心。

选择原因：

- 它最贴合本次已确认的交互要求。
- 它可以最小范围改造现有页面，而不增加新的外部依赖。
- 它允许我们把“反馈消息”与“业务详情面板”“确认弹窗”清晰分层，避免后续实现过度扩散。

## 整体架构

### 根部挂载

在应用根部新增 `NotificationProvider`，并让它包裹需要使用消息能力的整棵前端树。

推荐层级为：

- `BrowserRouter`
- `NotificationProvider`
- `SelectionProvider`
- 现有页面与路由内容

这样做的原因是：

- `SelectionContext` 本身就会产生全局加载与切换模式错误，也需要使用通知。
- 页面切换后通知仍可保留，不会因为局部组件卸载而丢失。

### 展示层

新增 `NotificationViewport` 负责渲染所有可见消息。

职责边界：

- 只负责展示与动画。
- 不负责业务调用。
- 使用 Portal 挂到 `document.body`。
- 固定在右下角。
- 容器内部按“上旧下新”垂直排列，使最新消息始终贴近右下角。

### 调用接口

对业务代码暴露统一调用接口，最小集合为：

- `notifySuccess(...)`
- `notifyWarning(...)`
- `notifyError(...)`
- `notifyFormErrors(...)`
- `dismissNotification(id)`

其中：

- 页面和组件不再使用局部 `message`、`error`、`feedback` 状态去渲染反馈条。
- 表单校验类错误统一通过 `notifyFormErrors(...)` 输出单条聚合消息。
- 删除、保存、导入、恢复、接口报错、占位提醒等都走通知接口。

## 数据模型

每条消息建议至少包含以下字段：

- `id`
- `level`
- `title`
- `description`
- `details`
- `createdAt`
- `durationMs`
- `interactiveLocked`
- `closing`

字段含义：

- `level` 表示消息级别，至少支持 `success`、`warning`、`error`。
- `title` 用于快速概括消息性质。
- `description` 用于展示简短正文。
- `details` 用于承载多条表单错误等结构化内容。
- `durationMs` 为自动关闭时长。
- `interactiveLocked` 表示该消息是否已因用户交互转入手动关闭模式。
- `closing` 用于退出动画阶段。

其中表单聚合消息建议统一归类为 `error` 级别。

## 生命周期与自动关闭规则

### 级别与时长区间

三类消息的目标停留区间固定如下：

- 普通通知或成功消息：2 到 3 秒
- 警告消息：3 到 6 秒
- 错误消息：5 到 8 秒

### 计算公式

自动关闭时间不采用随机值，而采用“基础时长 + 字数补偿 + 区间截断”的确定性计算。

推荐公式如下：

- 通知类：`2200ms + 字数 * 45ms`，最终限制在 `2000ms` 到 `3000ms`
- 警告类：`3200ms + 字数 * 55ms`，最终限制在 `3000ms` 到 `6000ms`
- 错误类：`5200ms + 字数 * 65ms`，最终限制在 `5000ms` 到 `8000ms`

字数统计范围包括标题、正文和错误列表文本。

这样做的原因是：

- 同一条消息每次显示时长一致，便于调试和验证。
- 文字越多，可读时间越长。
- 表单聚合错误自然会落入更长的可读区间。

### 自动关闭前提

只有当一条消息未被交互锁定时，才允许自动关闭。

一旦用户发生以下任一行为，该消息立即转入手动关闭模式：

- 鼠标移入消息
- 在消息内选中文本
- 点击消息内部按钮或链接
- 在消息上按下鼠标，准备复制、点击或其他操作

转入手动关闭模式后：

- 不再恢复自动倒计时
- 只能通过点击关闭按钮移除

本次设计刻意采用偏保守的规则，以避免用户在阅读、复制、点击时被消息自动消失打断。

## 队列与堆叠规则

- 同时最多显示最近 4 条消息。
- 消息在视觉上按照时间顺序自上而下排列，顶部最旧，底部最新。
- 新消息插入时位于底部，已有消息整体向上移动。
- 当新消息插入后总数超过 4 条，立即移除最老的一条。
- 不做消息折叠，不做“还有 X 条”入口，不保留独立历史列表。
- 页面跳转时不主动清空现有消息，由消息自己根据超时或手动关闭结束生命周期。

## 表单错误聚合规则

### 聚合方式

当一个表单在同一轮校验中出现多条错误时：

- 只显示 1 条消息
- 标题固定为类似“请检查表单”
- 正文或详情区域按列表形式列出所有错误

### 不再保留的行为

本次迁移后，以下模式不再作为主要反馈方式：

- 字段旁边的局部错误提示
- 表单区域顶部的常驻报错条
- 页面内独立的成功或失败横幅

### 不做的增强

表单校验失败后：

- 不自动滚动到第一个错误字段
- 不自动聚焦字段
- 不自动展开额外错误详情面板

## 文案与内容结构

每条消息的视觉内容统一由以下部分组成：

- 级别图标与颜色标识
- 标题
- 正文
- 右上角关闭按钮

对于不同类型的消息，文案遵循如下约定：

- 成功类标题偏结果型，例如“保存成功”“导入完成”
- 警告类标题偏提醒型，例如“请注意配置”
- 错误类标题偏行动型，例如“操作失败”“请检查表单”

正文要求：

- 普通成功、警告、错误使用一段简短说明
- 表单聚合错误使用列表形式逐条列出问题
- 正文允许文本选择，以支持复制长报错或错误列表

## 层级与视觉约束

### 挂载与层级

消息容器通过 Portal 直接挂到 `document.body`，并使用高于当前弹窗与确认框的层级。

它需要覆盖当前已知的弹窗层级实现，例如：

- `frontend/src/pages/ProfilePage.tsx`
- `frontend/src/pages/ProfessorsPage.tsx`
- `frontend/src/components/atoms/ConfirmDialog.tsx`

### 布局要求

- 桌面端固定在右下角，保留安全边距。
- 移动端仍固定在右下角，但缩窄宽度，避免遮挡主要交互区域。
- 最新消息贴近右下角。
- 单条消息呈稳定卡片样式，而不是脆弱的瞬时 hover 提示。

### 交互约束

- 消息正文允许选择文本。
- 如果后续某些消息需要包含可点击链接或按钮，也允许在正文区域中承载。
- 一旦发生点击、按下、选择或悬停，该消息立即锁定为手动关闭模式。

## 迁移策略

### 需要迁移的反馈类型

- 全局上下文加载与切换模式错误
- 页面级 `error`、`message`、`feedback`
- 表单提交前校验失败
- 表单提交失败
- 导入、保存、删除、恢复、测试连接等结果提示
- 当前所有 `window.alert(...)` 占位提示

### 不迁移的内容

- `useConfirmDialog` 这类确认型对话框继续保留
- 资料页中的连接诊断结果面板、模型列表面板等业务详情面板继续保留

换句话说，本次迁移的是“反馈消息出口”，不是把所有业务信息块都压缩成通知。

### 页面改造原则

- 页面本身只保留业务状态，不再自行渲染成功/失败横幅。
- 表单验证逻辑可以继续产出错误集合，但不再直接把它们绑定到字段提示 UI 上。
- 所有反馈类出口统一调用通知接口。
- 改动应保持外科手术式，不顺手重构与本次需求无关的页面结构。

## 验证策略

### 自动验证

至少覆盖以下验证：

1. 与消息时长计算相关的纯函数验证
2. 与表单错误聚合相关的纯函数验证
3. 前端构建通过
4. 前端静态检查通过

如果当前前端已有适合承载这类验证的测试基础设施，则优先补最小测试；如果没有，则不为本次功能额外引入过重测试框架。

### 手动验证

至少覆盖以下场景：

- 连续触发多条消息时，右下角最多显示 4 条，且最新在最下方
- 第 5 条消息出现时，最老的一条被移除
- 成功、警告、错误消息的自动关闭时长分别落在对应区间
- 字数更多的消息显示更久
- 鼠标移入消息后，消息不再自动消失
- 选中文本后，消息不再自动消失
- 点击消息内部交互后，消息不再自动消失
- 点击关闭按钮后，消息被正常移除
- 在弹窗打开时触发消息，消息仍可见
- 表单一次出现多条错误时，只出现 1 条“请检查表单”消息，正文列出所有错误
- 当前页面里的 `window.alert(...)` 占位提示已被统一消息替换

## 实现注意事项

- 不要把确认类弹窗和反馈类消息混在同一机制里。
- 不要为了消息系统顺手重构与需求无关的页面结构。
- 不要新增历史消息中心或消息抽屉。
- 不要恢复字段旁局部错误提示作为主反馈出口。
- 不要加入自动滚动、自动聚焦或自动跳转到错误字段的行为。
- 保持中文文案与当前项目整体风格一致。

## 成功标准

当以下条件同时满足时，视为本次设计目标达成：

- 前端应用拥有统一的右下角全局消息系统
- 最新消息显示在最下方，旧消息会被向上顶
- 同时最多显示最近 4 条消息
- 成功、警告、错误消息的停留时长符合既定区间，并随字数增长
- 用户一旦悬停、点击或选中文本，消息转为手动关闭模式
- 弹窗打开时消息仍然可见
- 表单字段级、区域级、页面内常驻提示已统一迁移为全局消息
- 现有成功、警告、报错和 `window.alert(...)` 入口已收口到统一调用接口
