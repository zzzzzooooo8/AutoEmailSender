# HTML 模板预览切换设计

## 背景

身份页的“默认模板正文（HTML，可保留格式）”目前只提供源码 textarea。
用户在导入 `.docx/.html` 模板后，能够拿到 HTML 正文，但只能阅读和编辑源码，无法直接查看渲染效果。

本次目标是在现有编辑区内增加一个长期存在的“渲染预览 / 原 HTML”切换，默认显示渲染后的预览，同时保持实现范围最小，不改后端接口和模板导入协议。

## 已确认的产品决策

- 切换长期存在于“默认模板正文（HTML，可保留格式）”编辑区。
- 交互采用同一区域双标签切换。
- 默认进入“渲染预览”。
- 模板变量如 `{{name}}`、`{{sender_name}}` 在预览中保持原样显示，不做示例值替换。
- “渲染预览”模式必须只读；只有切到“原 HTML”后才能编辑源码。
- 预览前需要做前端清洗，避免危险 HTML 直接进入页面。

## 目标与非目标

### 目标

- 让用户在身份页模板弹窗中直接查看 HTML 正文的渲染效果。
- 保持现有源码编辑能力不变。
- 让导入模板文件后的默认体验更自然：导入完成后直接看到预览。
- 将安全风险限制在前端预览层，不影响保存到后端的 HTML 原文。

### 非目标

- 不修改后端 API、数据库字段或导入返回结构。
- 不替换模板变量为示例数据。
- 不在创建任务页、工作台或其他页面同步实现该切换。
- 不引入并排双栏预览或复杂富文本编辑器。
- 不在本次改动中重构身份页弹窗的其他表单结构。

## 范围

仅修改身份页模板弹窗内的 HTML 正文编辑区，具体落点是 `OutreachTemplateModal` 所在文件。

预期受影响区域：

- `frontend/src/pages/ProfilePage.tsx`
- 新增一个轻量 HTML 预览辅助文件：`frontend/src/lib/htmlPreview.ts`
- 新增一个前端 HTML 清洗依赖

## 推荐方案

采用“同一区域双标签切换 + 预览前清洗 HTML”的方案。

原因：

- 最符合用户已经确认的 A 方案。
- 对现有状态和布局改动最小。
- 不会把弹窗高度进一步拉高。
- 能把危险 HTML 限制在预览层，不污染用户实际编辑和保存的源码。

## 交互设计

### 区域结构

在“默认模板正文（HTML，可保留格式）”标题右侧增加一个双标签切换：

- `渲染预览`
- `原 HTML`

标签风格应与现有弹窗视觉语言一致，保持轻量，不引入新的大型卡片层级。

### 默认行为

- 弹窗每次打开时，默认选中 `渲染预览`。
- 导入模板文件成功后，保持 `渲染预览` 为当前模式。
- 如果用户已经切到 `原 HTML` 并持续编辑，局部切换状态只在当前弹窗打开周期内有效；关闭弹窗后重置为默认的 `渲染预览`。

### 渲染预览模式

- 显示一个只读预览面板。
- 预览面板渲染当前 `outreach_template_body_html` 的清洗结果。
- 不显示 HTML textarea。
- 在预览面板下方显示简短说明：预览仅用于检查排版，如需修改请切换到 `原 HTML`。

### 原 HTML 模式

- 显示现有 HTML textarea。
- 保留当前 placeholder、字体、输入方式和状态写回逻辑。
- 不在该模式下额外显示渲染预览。

### 空态与异常态

- 若 HTML 为空：在预览面板中显示空态文案，提示用户切到 `原 HTML` 粘贴或编辑模板。
- 若 HTML 非空但清洗后无可显示内容：显示提示文案，说明当前 HTML 预览为空，请检查源码内容。

## 安全与渲染策略

### 依赖选择

前端新增轻量 HTML 清洗依赖，推荐使用 `dompurify`。

选择理由：

- 在 React/Vite 场景中成熟、稳定、心智负担低。
- 比手写清洗规则更安全。
- 集成成本低，适合本次小范围功能扩展。

### 数据流

- 用户输入或导入的 HTML 原文继续保存在 `identityForm.outreach_template_body_html`。
- 预览层使用单独的清洗函数，例如 `sanitizeTemplateHtmlForPreview(rawHtml)`。
- 该清洗结果只用于 `dangerouslySetInnerHTML` 的渲染。
- 清洗不会反写源码，不会改变保存到后端的 HTML 原文。

### 保留与移除原则

预览清洗需满足以下明确规则：

- 去掉 `script` 等危险标签。
- 去掉内联事件属性，例如 `onclick`。
- 去掉危险 URL，例如 `javascript:`。
- 保留邮件模板常见结构标签与安全的行内样式；如安全性与还原度冲突，以安全优先。

## 状态设计

在模板弹窗内部新增一个局部状态：

- `htmlViewMode: "preview" | "source"`

该状态特征：

- 只存在于前端弹窗内，不进入后端。
- 不进入全局状态。
- 弹窗打开时重置为 `preview`。
- 当 HTML 内容变化时，预览自动基于最新源码重新计算，不增加“刷新预览”按钮。

## 组件边界建议

实现阶段应把这部分拆成两个非常小的单元：

- 一个纯函数：负责预览 HTML 的清洗与空值归一化。
- 一个局部 UI 片段或小组件：负责切换标签、预览面板与 textarea 的显示切换。

不要为了这次改动把整个 `ProfilePage` 拆大重构。改动应保持外科手术式。

## 验证策略

### 自动验证

优先级从高到低：

1. 为 HTML 预览辅助函数补最小单元测试。
2. 至少保证 `frontend` 构建通过。

考虑到当前前端仓库未体现现成测试框架，本次实现阶段可以采用以下策略：

- 如果补一个低成本纯函数测试不会引入明显额外复杂度，则补上。
- 如果引入完整测试框架会明显扩大范围，则不为本次功能额外搭建整套测试基础设施。

### 手动验证

至少覆盖以下场景：

- 打开模板弹窗时默认显示 `渲染预览`。
- 点击 `原 HTML` 后出现源码 textarea，且可编辑。
- 在 `原 HTML` 中修改内容，再切回 `渲染预览`，预览随之更新。
- 导入模板文件后默认停留在 `渲染预览`。
- 占位符在预览中保持原样显示。
- 空 HTML 时展示空态。
- 带危险标签或属性的 HTML 不会在预览中直接执行。

## 影响文件

- `frontend/src/pages/ProfilePage.tsx`
- `frontend/src/lib/htmlPreview.ts`
- `frontend/package.json`

## 实现注意事项

- 不要改动现有 `outreach_template_body_html` 字段的保存语义。
- 不要顺手把此功能扩展到别的页面。
- 不要把“预览安全”演变成“保存前强制清洗”；本次只处理预览层。
- 保持现有中文文案风格和弹窗视觉语言。

## 成功标准

当以下条件同时满足时，视为本次设计完成：

- 用户可以在身份页模板弹窗中长期使用 `渲染预览 / 原 HTML` 切换。
- 弹窗默认显示渲染预览。
- 预览模式只读，源码模式可编辑。
- 模板变量原样显示。
- 预览使用前端清洗结果渲染。
- 改动仅限身份页模板弹窗范围，没有扩散到其他页面或后端接口。
