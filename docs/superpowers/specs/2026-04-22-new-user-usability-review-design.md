# 新用户首次上手易用性评审设计

## 1. 背景

本次评审面向「第一次接触系统的新用户」，并且把首次初始化也纳入范围，包括：

- 本地启动前后端
- 首次配置身份、模型、材料、模板
- 导入导师
- 创建第一批任务
- 进入工作区完成匹配、草稿、审核与发送

本次评审不以「现有高频用户的效率优化」为主，而是聚焦两个问题：

- 术语和按钮含义不直观
- 操作步骤过碎、页面切换过多

## 2. 现状概述

从当前实现来看，新用户首次使用系统时，实际会经历如下路径：

1. 阅读 `README.md`，启动前后端。
2. 进入首页，但首页默认是导师看板，隐含前提是用户已经完成身份、模型、材料和导师数据准备。
3. 如果顶部没有选中身份和模型，系统提示去个人页配置。
4. 在个人页中，用户需要同时面对身份、SMTP、IMAP、材料、模板、模型、默认项与测试能力。
5. 配置后，用户还需要去导师管理页导入导师。
6. 回到首页选择导师，再进入创建任务页。
7. 在创建任务页理解任务模式、发送方式、材料选择、模板配置等概念。
8. 创建任务后，再进入工作区执行匹配分析、草稿生成、人工修改和发送。

这条路径的问题不在于功能缺失，而在于系统把内部结构直接暴露给新用户，缺少一条明确的首次上手主线。

## 3. 核心问题判断

### 3.1 入口顺序与用户目标不一致

新用户的目标通常是「先跑通一封邮件」，但当前系统要求用户先理解：

- 身份
- 模型
- 材料
- 模板
- 导师
- 任务
- 工作区

也就是说，用户还没开始做事，就先被迫理解系统内部组织结构。

### 3.2 术语使用偏系统语言

以下术语对于系统设计是准确的，但对新用户理解成本较高：

- 模板润色
- 固定模板
- 默认材料
- 任务快照
- 当前上下文
- dry run / live

尤其在创建任务页和工作区，这些术语直接决定用户是否敢继续下一步。

### 3.3 关键动作跨页分散

当前关键动作分布在多个页面：

- 个人页：配置身份、模型、材料、模板
- 导师管理页：导入或维护导师
- 首页：筛选导师、发起任务
- 创建任务页：定义任务配置
- 工作区：匹配、草稿、审核、发送
- 任务页：查看批量任务状态

单个页面内部逻辑基本成立，但从新手视角看，用户很难稳定回答两个问题：

- 我现在做到哪一步了？
- 下一步应该去哪？

### 3.4 必要项与可选项没有清晰分层

首次使用时，并非所有配置同等重要。例如：

- SMTP 是首次跑通发送闭环的高优先项
- IMAP 属于增强项，用于回信检测
- `live` 模式属于高风险项，不应与普通配置并列处理
- 发送限制和高级参数不是首次上手的必经步骤

但当前页面展示容易让新用户误判为「必须先把所有东西都配置完整，才能开始使用」。

## 4. 设计目标

本次设计不追求重做整个产品结构，而是基于现有页面体系，建立一条更适合首次使用者的引导路径。

目标如下：

- 让新用户一进入系统就知道还差什么
- 让新用户始终知道下一步该去哪里
- 把内部术语翻译成动作语言和结果语言
- 在不推翻现有页面结构的前提下，降低跨页来回的心理负担
- 优先帮助用户跑通 `dry run` 首次闭环，再逐步引导到 `live` 与 IMAP 等进阶能力

## 5. 方案选择

在本次头脑风暴中，对比过 3 种方向：

1. 轻量修补型：主要改文案和提示。
2. 新手引导型：保留现有页面，重建首次上手顺序和引导。
3. 流程重组型：大幅调整页面职责，向向导式流程收敛。

最终选择 **新手引导型**，原因如下：

- 能直接命中当前最突出的问题：术语难懂、步骤过碎。
- 不需要推翻现有页面结构，实施风险相对可控。
- 可以优先改善新用户首次体验，同时保留老用户现有使用习惯。

## 6. 设计方案

### 6.1 首页：从导师看板升级为首次上手工作台

首页应承担两种模式：

#### 模式 A：首次上手模式

当系统检测到关键前置条件未完成时，不应默认展示导师列表主视图，而应优先展示一张首次上手引导卡。

引导卡应清楚回答三个问题：

- 你现在处于哪个阶段
- 还缺哪些必要项
- 下一步应该点击哪里

建议使用明确的顺序表达：

1. 创建发件身份
2. 配置模型
3. 上传材料与默认模板
4. 导入导师
5. 创建第一批任务

#### 模式 B：正常工作模式

当最低必要项已经满足后，首页恢复为当前的导师筛选与任务入口页面。

这样首页既能兼顾新用户引导，也能保留熟练用户的操作效率。

### 6.2 个人页：从大而全配置页改为阶段化准备页

个人页保留现有能力，但信息组织方式需要调整为更符合首次使用顺序的结构。

建议分为 4 个层级：

#### 第一层：基础必填

- 身份名称
- 发件邮箱
- SMTP 配置

目标是先完成最小可发送闭环。

#### 第二层：写信基础

- 默认材料
- 默认模板

目标是保证后续匹配和草稿生成可以继续。

#### 第三层：AI 基础

- 模型配置
- 模型连通性测试

目标是让用户明确 AI 能否工作。

#### 第四层：增强和高级设置

- IMAP 回信检测
- 发送限制
- 默认项切换
- 删除等低频操作

目标是避免首次使用时把高风险、低频能力与基础流程混在一起。

### 6.3 创建任务页：用动作语言替代系统术语

创建任务页是新用户理解负担最重的页面之一，建议优先做术语翻译与解释重写。

建议替换如下：

- 「模板润色」→「AI 辅助写信」
- 「固定模板」→「直接套用模板」
- 「默认材料」→「用于分析匹配的材料」
- 「随信材料」→「随邮件附带的文件」

说明文案也应从机制解释改为结果解释。

例如，不强调「任务快照」这一内部实现，而改为：

> 你这次填写的内容只作用于本次任务，之后修改默认设置，不会影响已经创建的任务。

此外，创建任务页应更加明确地说明这一步完成后的后续动作，例如：

- 创建后会进入任务列表
- 下一步通常是进入工作区生成草稿并确认发送

### 6.4 工作区：从工具集合改为下一步驱动

工作区当前功能足够，但新用户仍然需要自己判断顺序。

建议在工作区内始终明确当前阶段与下一步，例如：

1. 先分析匹配
2. 再生成草稿
3. 然后人工修改
4. 最后发送或定时发送

工作区中的按钮文案也建议改为更贴近用户目标的表达，例如：

- 「先看匹配」→「分析这位导师是否值得联系」
- 「生成新草稿」→「生成一版邮件草稿」

核心原则是：系统不只是提供按钮，而是明确告诉用户此刻应该做什么。

### 6.5 顶部全局栏：从配置控件改为安全状态条

顶部的身份、模型与发送模式很重要，但对新用户来说，它们更像系统状态，而不是普通筛选器。

建议将其强化为状态表达：

- 当前正在使用哪个身份
- 当前正在使用哪个模型
- 当前是否处于 `dry run`
- 如果是 `live`，需要明显强调风险

这能降低新用户对全局状态的误操作概率，也能减少首次使用阶段频繁切换上下文的心智负担。

## 7. 优先级建议

### 7.1 第一优先级

这些改动直接决定新用户能否顺利完成首次上手：

1. 首页增加首次上手引导卡
2. 个人页按必填、可选、高级分层重排
3. 创建任务页术语翻译与文案重写
4. 工作区加入阶段提示与下一步引导

### 7.2 第二优先级

这些改动会继续提升体验，但不必作为第一轮阻塞项：

1. 顶部全局栏改成更强的状态提示
2. 各页增加「下一步去哪」的回流提示
3. README 与运行手册补一份真正面向新手的首次使用说明

## 8. 非目标

为了控制范围，本轮设计明确不包括：

- 重做整套路由结构
- 把所有流程强行合并成单页向导
- 优先做纯视觉风格重构
- 修改任务状态机和后端核心业务流程

本轮目标是先建立清晰的首次上手主线，而不是全面重构产品。

## 9. 验收标准

本次设计如果落地，应该至少满足以下标准：

- 新用户首次进入首页时，能在 10 秒内看懂下一步该做什么
- 新用户无需理解系统内部术语，也能完成第一批任务创建
- 用户在个人页、导师管理页、首页、创建任务页、工作区之间切换时，不会反复询问「下一步去哪」
- 新用户能先跑通 `dry run` 闭环，再逐步理解 `live` 与 IMAP 等增强能力

## 10. 最终结论

这套系统当前最需要的，不是新增功能，而是为第一次使用的人建立一条清晰、低术语、低来回跳转的上手主线。

具体来说，系统应先回答：

- 你现在离开始还差什么
- 下一步应该去哪里
- 这个按钮点下去会发生什么

当这些问题被清楚回答后，现有能力才能真正转化为用户可用的体验。
