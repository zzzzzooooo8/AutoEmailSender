# 移除 dry_run 并引入测试写信页设计

## 1. 背景

当前系统把发送能力分成全局 `dry_run / live` 两种模式：

- `dry_run`：批准发送后不真实发信，但会写入任务状态和发送日志
- `live`：真实调用 SMTP 发信，并参与后续 IMAP 回复关联

这套设计最初是为了提供安全护栏，但在当前产品定位下已经出现明显问题：

- 产品表达不直观，用户需要先理解模式概念，再理解真实动作
- `dry_run` 不是完整业务模式，只是安全分支，却被放成高存在感的全局开关
- 用户会被引导去“模拟发信”，但模拟结果并不等于真实邮件效果
- 真正高风险的地方其实是正式发送和定时发送，而不是是否存在一个全局模式

本次设计目标是一步到位移除 `dry_run`，并用更符合直觉的能力替代它。

## 2. 设计目标

- 删除 `dry_run / live` 这一组产品概念和全局模式切换
- 保留真实发信能力，并让用户明确知道每次动作的后果
- 提供一个独立的“测试写信页”，用于真实发一封邮件给自己，验证整条发信链路
- 让工作区只处理真实导师任务，不再混入“本地演练”语义
- 对立即发送、定时发送、批量发送增加明确的风险提示
- 一步到位清理前端、后端、数据库、测试和文档中的 `dry_run` 残留

## 3. 方案选择

本次讨论中实际对比了以下方向：

1. 保留 `dry_run`，只改名称和文案
2. 保留底层 `dry_run` 能力，但在前台弱化展示
3. 彻底移除 `dry_run`，用“测试写信页 + 风险提示”替代

最终选择第 3 种方案，原因如下：

- 用户真正需要的是“确认真实邮件长什么样”，而不是“体验一次不发出的模拟流程”
- 风险应该在高风险动作前直说，而不是要求用户先理解系统模式
- 独立测试写信页比工作区里的“本地演练”更符合产品语义
- 一步到位可以避免后续长期兼容两套发送概念

## 4. 最终方案

### 4.1 产品概念调整

系统不再暴露全局发送模式，也不再使用 `dry_run / live` 这组术语。

发送体系改成两类动作：

- 正式导师发送：发生在工作区和批量任务流中，永远是真实发送
- 测试发送：发生在独立的测试写信页中，收件人固定为当前身份自己的邮箱

### 4.2 个人页入口

“发送测试邮件”入口放在个人页配置区域的最下方，属于身份配置完成后的收尾动作。

入口职责：

- 帮用户在完成 SMTP、IMAP、模板、材料等配置后，进入测试发信流程
- 明确告知这是发给自己的测试邮件，不会进入导师任务流
- 不把测试入口放进工作区，避免把测试行为混入真实导师任务

### 4.3 独立测试写信页

新增一页语义独立的“测试写信页”，它不是导师工作区，也不依附任何导师或批量任务。

页面能力与真实写信体验尽量一致，包括：

- 使用当前选中的身份
- 使用当前选中的 LLM 配置
- 使用当前身份下的默认模板能力
- 支持生成草稿
- 支持手动编辑主题和正文
- 支持选择附件
- 支持立即发送给自己
- 保留测试发送历史

页面不支持：

- 绑定导师
- 创建或复用 `email_task`
- 定时发送
- 批量发送
- 回复检测
- 正式任务状态流转

### 4.4 正式工作区的职责收敛

工作区继续只承担真实导师任务流：

- 匹配分析
- 草稿生成
- 人工修改
- 立即发送
- 定时发送
- 查看真实往来与结果

工作区中不再出现：

- 本地演练
- 发送模式快照
- 顶部全局模式依赖

### 4.5 风险提示替代模式开关

删除模式开关后，风险控制改为动作前确认。

#### 单封立即发送

在真正发送前弹出轻量确认，明确说明：

- 收件人是谁
- 将附带哪些附件
- 这是一封真实邮件
- 发送后会进入正式通信记录

#### 单封定时发送

在真正提交前弹出更强确认，明确说明：

- 这是未来会自动发出的真实邮件
- 到点后不会再弹第二次确认
- 当前确认的主题、正文和附件会按本次版本执行

#### 批量发送与批量排程

在批量任务关键节点给出集中风险提示，重点强调：

- 这是对真实导师的正式发送
- 如果是排程批量，后续会按计划持续自动发出
- 风险提示不再借助 `dry_run/live` 术语表达

## 5. 信息架构与数据边界

### 5.1 正式任务流

正式任务流仍以现有 `email_tasks`、`email_logs`、批量任务体系为中心。

删除 `dry_run` 后，正式任务流的核心变化是：

- 不再读取全局发送模式
- 正式发送永远按真实 SMTP 路径执行
- 回复检测继续只针对正式发出的真实邮件

### 5.2 测试写信流

测试写信流需要独立于导师任务流。

建议新增一套最小独立数据结构，用于保存：

- 测试写信会话
- 当前身份和当前 LLM 的快照引用
- 测试主题、正文、HTML 内容
- 附件选择
- 每次测试发送历史
- 真实 SMTP 发送结果和错误信息

设计原则：

- 不复用 `email_task`
- 不伪造“测试导师”
- 不让测试邮件污染导师通信记录
- 保持与正式任务流分离，但复用已有的模板渲染、LLM 草稿生成、SMTP 发信能力

## 6. 页面与交互设计

### 6.1 个人页

个人页底部新增“发送测试邮件”入口区块，展示：

- 当前身份邮箱
- 简短说明：进入独立测试写信页，发信对象是自己
- 按钮：进入测试写信页

### 6.2 测试写信页

测试写信页建议复用现有工作区中的成熟编辑体验，但语义上独立。

页面建议包含以下区域：

- 页面头部：说明这是测试写信页，收件人为当前身份自己的邮箱
- 主编辑区：主题、正文、HTML 状态提示
- 草稿辅助区：草稿方式、生成草稿、重新生成
- 附件区：从当前身份材料中勾选附件
- 发送区：仅保留“立即发送测试邮件”
- 历史区：展示测试发送历史，不与导师历史混合

### 6.3 工作区

工作区移除所有与发送模式相关的 UI：

- 顶部模式显示
- 模式选择控件
- 与“本地演练”有关的状态提示

保留并强化：

- 发送前确认
- 排程前确认
- 真实发送结果说明

## 7. 后端设计

### 7.1 发送服务

现有发送服务保留，但发送决策逻辑调整为：

- 正式工作区发送：始终走真实 SMTP
- 测试写信页发送：同样走真实 SMTP，但收件人固定为当前身份自己的邮箱

测试发信应尽量复用正式发送时的邮件构建逻辑，确保用户看到的效果尽量接近真实正式发信。

### 7.2 系统设置

删除系统级 `mail_delivery_mode` 概念，包括：

- 默认配置项
- 读取与更新接口
- 前端全局上下文中的发送模式状态

### 7.3 任务与日志

正式任务流中删除 `delivery_mode` 相关读写和展示逻辑。

如果正式发送日志中仍需要区分来源，应该使用更准确的新字段或新事件类型表达，例如：

- 正式导师发送日志
- 测试写信发送日志

不再延续 `dry_run/live` 的历史表达。

## 8. 数据迁移策略

一步到位移除 `dry_run` 需要处理现有数据库结构和历史数据。

迁移原则如下：

- 删除 `app_settings.mail_delivery_mode`
- 删除 `email_tasks.delivery_mode`
- 删除 `email_logs.delivery_mode`
- 删除依赖这些字段的接口响应和前端类型

对于已有历史数据：

- 允许历史库在迁移前存在 `dry_run` 记录
- 迁移后不再依赖该值驱动任何业务逻辑
- 如需保留审计信息，可在迁移时把旧日志内容折叠进更通用的备注或 provider payload；如果当前没有明确业务价值，则可直接移除字段，不做额外兼容展示

## 9. 错误处理

### 9.1 测试写信页

需要明确处理以下失败：

- 当前身份没有邮箱，不能发送测试邮件
- SMTP 配置缺失或认证失败
- 主题或正文为空
- 附件不存在或不可读取
- LLM 草稿生成失败

错误展示原则：

- 清楚说明是“测试发送失败”，不要混成正式任务失败
- 保留失败历史，便于用户回看
- 不因为测试失败污染正式任务状态

### 9.2 正式任务流

正式任务流继续保留当前发送失败处理，但移除与模式切换相关的错误语义。

用户在确认发送前应看见明确后果，在发送失败后应看见明确原因，不再出现“是不是因为还停留在 dry_run”这种额外判断。

## 10. 测试策略

### 10.1 前端

至少覆盖：

- 个人页测试入口展示与跳转
- 测试写信页的主题、正文、附件编辑
- 测试写信页调用草稿生成
- 测试写信页发送成功与失败提示
- 工作区移除模式 UI 后仍可立即发送与定时发送
- 立即发送与定时发送的风险确认交互

### 10.2 后端

至少覆盖：

- 删除 `mail_delivery_mode` 后系统设置接口与默认初始化行为
- 正式发送始终走真实 SMTP 路径
- 测试写信发送给当前身份自己的邮箱
- 测试写信历史独立存储
- 回复检测不受测试写信历史干扰
- 旧 `dry_run` 相关字段移除后的迁移正确性

### 10.3 回归重点

重点回归以下闭环：

1. 个人页配置完成后，进入测试写信页并成功收到测试邮件
2. 正式工作区生成草稿后立即发送，真实发给导师
3. 正式工作区定时发送后，到点自动真实发出
4. IMAP 回复检测仍然只关联正式导师邮件

## 11. 范围边界

本次设计不包含以下内容：

- 重做整个个人页结构
- 重做工作区整体布局
- 新增测试写信页的定时发送
- 把测试写信页与正式导师工作区合并
- 增加多测试收件人能力

## 12. 成功标准

本次改动完成后，应达到以下结果：

- 用户不再需要理解 `dry_run / live`
- 用户可以在个人页完成配置后，进入独立测试写信页发一封真实测试邮件给自己
- 工作区只承担正式导师发送
- 发送风险通过动作前确认表达，而不是通过全局模式表达
- 前后端、数据库、测试和文档中不再残留 `dry_run` 作为正式产品能力
