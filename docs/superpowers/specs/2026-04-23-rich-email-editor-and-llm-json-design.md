# 富文本邮件编辑与 LLM 结构化草稿设计

## 背景

当前系统已经同时保存邮件正文的纯文本和 HTML 版本，但用户界面仍然把“纯文本正文”和“HTML 正文”拆开维护。测试写信页和正式写信区也主要使用纯文本 textarea，导致用户看到的编辑体验和最终 HTML 邮件之间不一致。

目标是统一邮件内容模型：用户编辑富文本，系统以 HTML 作为主内容存储和发送，同时自动派生纯文本。LLM 不直接生成 HTML，而是生成受控富文本 JSON，由系统校验并渲染为安全 HTML 和纯文本。

## 目标

- 个人页默认模板、测试写信页、正式工作区写信区使用同一个富文本编辑体验。
- `.docx`、`.html`、`.htm`、`.txt`、`.md` 模板导入后都进入富文本编辑器显示。
- 数据库和接口继续保留 `body_html` / `body_text` 双字段，但 HTML 是事实来源，纯文本是派生字段。
- LLM 生成草稿时输出受控富文本 JSON，后端负责校验、渲染 HTML、派生纯文本。
- 邮件发送只使用经过系统清洗和渲染的 HTML，避免模型直接输出不可控 HTML。

## 非目标

- 不实现完整 Word/邮箱客户端级排版能力。
- 不允许用户在主流程里直接编辑 HTML 源码。
- 不支持图片内嵌、任意字体、任意颜色等高兼容风险格式。
- 不把 LLM 输出 JSON 作为长期主存储格式；JSON 只是生成过程的中间格式。

## 内容模型

富文本编辑器对外输出：

- `body_html`: 主内容，保存和发送使用。
- `body_text`: 从富文本内容派生，用于后端校验、纯文本 fallback、摘要、搜索和 LLM 上下文。

后端继续兼容已有请求：如果调用方只传 `body_text`，后端仍可生成基础 HTML；新前端应优先同时提交 `body_html` 和 `body_text`。

## LLM 输出结构

LLM 生成草稿时输出 JSON，而不是 HTML。第一版支持这些节点：

- `paragraph`
- `bullet_list`
- `numbered_list`
- `link`
- `strong`
- `emphasis`
- `line_break`

建议结构：

```json
{
  "subject": "邮件主题",
  "blocks": [
    {
      "type": "paragraph",
      "children": [
        { "type": "text", "text": "老师您好，" }
      ]
    }
  ]
}
```

后端新增富文本模块负责：

- 校验 JSON schema。
- 丢弃未知节点和不安全属性。
- 限制链接协议为 `http`、`https`、`mailto`。
- 渲染为邮件安全 HTML。
- 派生纯文本。

## 前端组件

新增共用组件 `RichEmailEditor`。

职责：

- 提供富文本编辑区和基础工具栏。
- 支持段落、加粗、斜体、链接、无序列表、有序列表。
- 对外提供 `html`、`text`、`onChange`。
- 可选展示“发送预览”和“纯文本预览”。

接入位置：

- 个人页默认模板弹窗：替换“默认模板正文（纯文本）”和“默认模板正文（HTML）”两个输入。
- 测试写信页：替换邮件正文 textarea。
- 工作区正式写信区：替换邮件正文 textarea，并去掉“改正文会切回普通文本”的提示。

## 模板导入

现有导入逻辑已经可将 `.docx` 转为 HTML，并抽取纯文本。新流程保持后端导入能力，但前端显示方式改为：

- `.docx`: 后端转 HTML，前端直接加载到富文本编辑器。
- `.html` / `.htm`: 后端规范化 HTML，前端加载到富文本编辑器。
- `.txt` / `.md`: 后端转基础 HTML，前端加载到富文本编辑器。

导入成功提示不再强调“自动生成纯文本正文”，改为强调“已导入为可编辑富文本模板”。

## 后端调整

- 新增富文本校验和渲染模块。
- LLM 草稿生成改为要求模型输出受控 JSON。
- 生成失败或 JSON 不合法时返回明确错误，不静默发送。
- 保存模板、测试草稿、正式草稿时，后端对 HTML 再做一次清洗和纯文本派生。
- 保留旧字段兼容，降低迁移风险。

## 测试

后端测试：

- 受控 JSON 能渲染出预期 HTML 和纯文本。
- 未知节点、危险链接、脚本内容会被拒绝或清洗。
- LLM 生成草稿使用 JSON 渲染结果写入 `body_html` 和 `body_text`。
- 模板导入 `.docx` 后保留 HTML 正文。

前端测试：

- 个人页导入模板后富文本编辑器显示正文。
- 个人页保存时提交 HTML 和派生纯文本。
- 测试写信页保存/发送使用富文本编辑器输出。
- 工作区正式写信区保存/发送使用富文本编辑器输出。

## 分阶段实施

第一阶段：

- 建立后端富文本 JSON 校验和渲染模块。
- 建立前端 `RichEmailEditor` 最小可用版本。
- 接入个人页模板编辑。

第二阶段：

- 接入测试写信页和正式工作区写信区。
- 调整保存和发送请求，确保提交 HTML 和派生纯文本。

第三阶段：

- 改造 LLM 生成草稿输出为受控富文本 JSON。
- 补齐异常处理和端到端回归测试。
