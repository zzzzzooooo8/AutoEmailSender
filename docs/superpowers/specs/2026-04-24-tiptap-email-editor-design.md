# Tiptap 邮件编辑器统一替换设计

## 背景

当前系统已经在测试写信页和正式工作区接入了一套自研富文本邮件编辑器，个人页模板弹窗仍使用“纯文本正文 + HTML 编辑/预览”混合方案。最近的调试表明，这套基于 `contentEditable + execCommand + HTML 清洗` 的实现会持续暴露以下问题：

- 编辑阶段与最终邮件 HTML 不一致。
- 样式保真脆弱，尤其是表格和内联字体样式。
- 浏览器原生撤销/重做体验不稳定。
- 随着功能扩大到字体、段落、表格操作，维护成本会快速上升。

用户已经确认：

- 富文本编辑器第一版选择 `Tiptap`。
- 三处入口一次性统一替换：
  - 个人页模板弹窗
  - 测试写信页
  - 正式工作区写信区
- 第一版能力范围：
  - 字体：字体、字号、加粗、斜体、下划线
  - 段落：对齐、行距、首行缩进
  - 列表：有序、无序
  - 链接
  - 表格：创建、增删行列、合并/拆分单元格
  - HTML 预览

## 目标

- 用 `Tiptap` 替换现有自研邮件编辑器。
- 三个入口共享同一个 `EmailTemplateEditor` 组件。
- 用户编辑时以富文本文档状态为主，不再要求用户维护纯文本字段。
- 仍以 HTML 作为数据库和发送链路的主内容。
- 纯文本继续作为系统派生字段保留，用于后端校验、搜索、纯文本 fallback 和部分 LLM 场景。
- 保持与当前后端 `body_html` / `body_text` 契约兼容。

## 非目标

- 第一版不做图片上传、图片缩放、浮动布局。
- 不支持颜色、背景色、高级段落样式面板。
- 不开放 HTML 源码直接编辑作为主流程。
- 不修改当前“LLM 输出受控 JSON，后端渲染为 HTML/纯文本”的总体架构。

## 总体方案

前端新增一个共用组件 `EmailTemplateEditor`，基于 `Tiptap` 构建，内部维护 Tiptap 文档状态，外部暴露：

- `html`
- `text`
- `onChange`

三个入口全部改为使用这一组件：

1. 个人页模板弹窗  
   替换“默认模板正文（纯文本）”和“默认模板正文（HTML，可保留格式）”。

2. 测试写信页  
   替换当前富文本邮件正文编辑区，保留主题、附件、发送逻辑。

3. 工作区写信区  
   替换当前正式发送正文编辑区，保留主题、草稿生成、发送/定时发送逻辑。

## 技术选择

基于 `Tiptap` React 生态，组合官方扩展和少量自定义扩展。

优先使用：

- `@tiptap/react`
- `@tiptap/starter-kit`
- `@tiptap/extension-link`
- `@tiptap/extension-underline`
- `@tiptap/extension-text-style`
- `@tiptap/extension-table`
- `@tiptap/extension-table-row`
- `@tiptap/extension-table-cell`
- `@tiptap/extension-table-header`

自定义扩展或命令层：

- `fontFamily`
- `fontSize`
- `lineHeight`
- `firstLineIndent`

选择 Tiptap 的理由：

- 表格能力成熟，适合后续扩展工具栏和上下文菜单。
- 与 React 集成良好，适合做共用编辑器组件。
- 文档模型稳定，适合和当前后端“结构化富文本 JSON → HTML/纯文本”路线对齐。
- 比继续手写 `contentEditable` 更容易维护。

## 数据模型

### 前端

编辑器对外始终输出：

- `html`: 邮件 HTML 主内容
- `text`: 从当前编辑内容派生的纯文本

### 后端

继续保留现有字段：

- `body_html`
- `body_text`

规则改为：

- `body_html` 是事实来源。
- `body_text` 是由前端或后端派生的辅助字段。
- 后端保存/发送前仍做最终清洗和派生，作为最后防线。

## 与 LLM 的关系

当前后端已经在推进：

- LLM 输出 `rich_body` 受控 JSON
- 后端渲染成 `body_html` / `body_text`

这一设计保持不变。

接入方式：

- Tiptap 编辑器只接收后端最终产出的 `body_html`
- 用户后续编辑该 HTML
- 保存/发送时再回传新的 `body_html` / `body_text`

因此：

- LLM 的中间 JSON 不是前端编辑主格式
- 前端编辑主格式仍是 HTML
- 后端继续负责 JSON → HTML 的可信渲染

## 编辑器能力设计

### 工具栏

第一版工具栏分组：

- 字体组：字体、字号
- 文本样式组：加粗、斜体、下划线
- 段落组：左对齐、居中、右对齐、两端对齐
- 段落样式组：行距、首行缩进
- 列表组：有序列表、无序列表
- 链接组：插入/编辑链接、取消链接
- 表格组：插入表格
- 预览组：HTML 预览切换

### 工具栏交互方向

第一版采用 `Word / WPS` 风格的双层工具栏，不使用原生 `<select>`。

目标：

- 用户一眼就知道这是“能认真排版邮件模板”的编辑器。
- 所有格式控件都具备编辑器语义，而不是表单语义。
- 当前值显示必须直观，避免“字体”和“黑体”并列、或“1.5”看不出是行距这种情况。

双层布局：

第一行：

- 字体
- 字号
- 行距
- 首行缩进
- 对齐（左 / 中 / 右 / 两端）

第二行：

- 加粗
- 斜体
- 下划线
- 链接
- 无序列表
- 有序列表
- 插入表格
- HTML 预览

交互规则：

- 字体、字号、行距、首行缩进都使用自定义弹出菜单按钮。
- 按钮本身直接显示当前值，而不是只显示占位词。
- 选项菜单中当前值带勾选状态。

默认文案：

- 字体：`宋体` / `微软雅黑` / `Times New Roman`
- 字号：`12pt` / `14pt` / `16pt` / `18pt`
- 行距：`1.5 倍行距` / `1.75 倍行距` / `2 倍行距`
- 首行缩进：`无缩进` / `首行缩进 2 字符`

### 表格操作

表格相关命令建议通过选中单元格后的浮动工具条或右上角上下文区提供：

- 插入行
- 删除行
- 插入列
- 删除列
- 合并单元格
- 拆分单元格

不在第一版内做拖拽调宽或复杂表格属性面板。

### 表格交互方向

当前的主要问题不是“没有插入表格按钮”，而是用户不知道怎么继续编辑表格内容。

第一版交互：

1. 插入表格默认生成 2x2。
2. 插入后光标自动落到第一个单元格，直接可输入。
3. 当光标位于表格单元格内时，工具栏右侧出现表格操作组：
   - 插入行
   - 删除行
   - 插入列
   - 删除列
   - 合并单元格
   - 拆分单元格
4. 表格外层有清晰焦点边框，避免“选不中”的感觉。
5. 单元格内部仍然按普通文本区编辑，不嵌套额外控件。

不在第一版做：

- 拖拽列宽
- 复杂右键菜单
- 单元格背景色和边框样式面板

## 测试写信页重排

当前测试写信页的主要问题是：

- 顶部说明区太像普通卡片页，不像编辑器页面。
- 编辑器不是视觉中心。
- 附件、历史和正文在同一层级上抢注意力。

新的页面布局：

### 顶部信息区

用紧凑头部替换现在的大说明卡：

- 页面标题：`测试写信`
- 当前身份
- 当前模型
- 测试收件邮箱
- 一句短说明：这是发给自己的真实测试邮件

### 主编辑区

双栏布局：

- 左侧（主）：主题 + 双层工具栏 + 正文编辑器
- 右侧（辅）：附件、发送历史、发送说明

编辑器区域必须是页面视觉中心，不再被右侧信息卡抢走重心。

### 底部动作区

底部单独一条动作栏，固定三类操作：

- 次级：`生成测试草稿`
- 次级：`保存草稿`
- 主操作：`发送测试邮件`

动作不再散在标题旁边，减少视觉噪声。

### 视觉一致性

测试写信页要和工作区共享同一种视觉语言：

- 白底主编辑器面板
- 更高密度的工具栏
- 右侧辅助信息区弱化
- 操作按钮层级明确

这样它看起来会像“测试场景下的邮件编辑工作台”，而不是“普通页面里放了个编辑器”。

### 样式保真策略

编辑阶段优先保留现有 HTML 中的邮件样式，特别是：

- `table` 结构
- `tbody/tr/td/th`
- `font-family`
- `font-size`
- `line-height`
- `text-indent`
- 基本对齐相关样式

发送前由后端继续做 HTML 白名单收敛，避免不安全标签和属性混入。

## 三个入口的具体调整

### 1. 个人页模板弹窗

替换当前双正文输入模型：

- 删除“默认模板正文（纯文本）”
- 删除“默认模板正文（HTML，可保留格式）”
- 改为一个“默认模板正文”编辑器

导入 `.docx/.html/.txt/.md` 后：

- 后端输出 `body_html` 和 `body_text`
- 前端只把 `body_html` 灌入编辑器
- 页面文案改为“已导入为可编辑富文本模板”

### 2. 测试写信页

保留：

- 主题输入
- 生成测试草稿
- 保存草稿
- 发送测试邮件

替换：

- 正文区域改为 `EmailTemplateEditor`

### 3. 工作区写信区

保留：

- 主题输入
- 草稿生成
- 发送与定时发送
- 附件选择

替换：

- 正文编辑区改为 `EmailTemplateEditor`
- 删除“手动改正文会切回普通文本”的旧提示

## 测试策略

### 前端

- `EmailTemplateEditor` 基础渲染与 `onChange`
- 字体、字号、段落样式命令会产出对应 HTML
- 表格插入与基础行列操作
- 三个入口都能正确加载已有 HTML
- 编辑后不会在输入一个字符后丢失表格或字体样式
- 测试写信页和工作区发送 payload 包含 `body_html` / `body_text`

### 后端

- 保持现有 `rich_text` 清洗与纯文本派生测试
- 确认新的 HTML 输出不会破坏现有发送链路

## 风险

### 1. Tiptap 默认 HTML 输出与邮件场景不完全一致

需要在扩展或导出阶段约束 HTML 输出，避免生成不适合邮件客户端的结构。

### 2. 表格操作 UI 容易膨胀

第一版只做最核心表格操作，不做复杂属性面板。

### 3. 三处一起替换范围较大

通过共用组件和聚焦测试降低风险；先统一编辑器内核，再分别接入三处页面。

## 实施顺序

1. 引入 Tiptap 依赖并建立 `EmailTemplateEditor`
2. 接入个人页模板弹窗
3. 接入测试写信页
4. 接入工作区写信区
5. 跑前端关键测试、lint、build
6. 回归发送链路

## 验收标准

- 三个入口都不再使用当前自研邮件编辑器
- 用户可以设置字体、字号、对齐、行距、首行缩进
- 用户可以插入和编辑表格
- 表格和字体样式在编辑一次后不会立即丢失
- 测试写信页和正式工作区发送链路保持可用
- 前端关键测试、lint、build 全部通过
