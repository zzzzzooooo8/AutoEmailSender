# 联系任务状态模型重构设计

## 1. 背景

当前系统把 3 类不同语义揉进了同一个 `EmailTask.status`：

- 导师是否值得联系
- 单封邮件任务执行到哪一步
- 批量任务是否还在继续处理

这导致了几类明显问题：

- 匹配度低会把任务自动变成 `skipped`，等于系统在用户已经选中导师后再次替用户否决。
- 批量任务停止后，未完成子任务也会进入 `skipped`，用户在工作区很难理解这代表“批量停止”还是“系统建议不联系”。
- 首页、工作区、任务页分别消费同一组底层状态，但需要的语义层级完全不同，结果是页面文案和交互都不自然。

本次重构的核心原则是：

**匹配度只帮助用户选择，不参与任务执行裁决。**

用户一旦把导师加入任务，系统就默认这位导师应被联系；系统只负责提供分数、解释和执行能力，不再做“低分自动跳过”。

## 2. 设计目标

- 让匹配度退出任务执行链路，只保留展示、筛选、排序、解释能力。
- 让 `EmailTask.status` 只表达单封任务的执行进度，不再夹带“系统建议不联系”的语义。
- 让首页只展示用户能理解的导师关系状态，而不是底层任务状态。
- 让批量任务停止后的导师仍可在工作区继续联系，并通过新建手动任务承接上下文。
- 为历史 `skipped` 数据提供明确迁移规则，避免老数据继续污染新语义。

## 3. 非目标

- 本次不重做导师管理页的数据结构。
- 本次不引入新的批量编排模式或复杂调度策略。
- 本次不改变 SMTP、IMAP、LLM 调用能力本身。
- 本次不新增“自动跟进邮件”或多轮会话自动化。

## 4. 现状问题拆解

### 4.1 匹配度与执行状态耦合

当前匹配计算结束后，如果 `match_score < identity.match_threshold`，任务会被直接写成 `skipped`。这意味着：

- 匹配度从“建议信息”变成了“执行拦截器”。
- 用户在首页或创建任务页已经选中导师，但系统仍可能自动取消执行。
- 工作区里的“已跳过”既像系统建议，又像终态，用户难以判断能否继续写信。

### 4.2 批量停止与跳过语义混淆

当前批量任务停止时，所有未完成且未发送的子任务都会被改成 `skipped`。这让同一个状态同时表示：

- 系统认为不建议联系
- 这批任务被人工停止

两者对用户来说完全不是一回事。

### 4.3 不同页面消费了不适合自己的状态粒度

- 首页需要的是“这位导师现在整体处于什么阶段”。
- 工作区需要的是“这封邮件现在能做什么动作”。
- 任务页需要的是“这批自动流程还在不在运行”。

当前三者都依赖同一组底层任务状态，结果是首页太细、工作区太绕、任务页和工作区边界不清。

## 5. 方案选择

本次重构实际对比了 3 个方向：

1. 只修 `skipped` 逻辑，不动整体状态模型。
2. 保留现有状态枚举，但通过文案和映射弱化“自动跳过”的问题。
3. 彻底重定义状态语义：匹配度退出执行链路，任务状态只表达执行进度，首页显示导师关系状态，批量停止改为取消，继续联系时新建手动任务。

最终选择第 3 种方案，原因如下：

- 只修 `skipped` 仍会保留“系统替用户改决定”的深层问题。
- 如果不拆开首页、工作区、任务页的状态语义，前端映射会持续变复杂。
- 新建手动任务承接批量停止后的继续联系，能保持统计和历史记录的边界清晰。

## 6. 最终方案

### 6.1 匹配度重新定位

匹配度只保留 4 个用途：

- 展示：在首页和工作区展示 `match_score`、`match_reason`、`fit_points`、`risk_points`
- 筛选：按匹配分过滤导师
- 排序：按匹配分排序
- 解释：帮助用户理解为什么联系或暂缓联系

匹配度不再承担以下职责：

- 不再自动跳过导师
- 不再自动取消任务
- 不再默认从创建任务结果中剔除低分导师
- 不再阻止工作区生成草稿或发送邮件

### 6.2 邮件任务状态

`EmailTask.status` 调整为只表达单封邮件的执行进度，建议定义为：

- `discovered`：任务已创建，尚未形成有效草稿
- `matched`：已完成匹配分析
- `review_required`：草稿已生成，等待人工审核
- `approved`：已批准，等待立即发送
- `scheduled`：已排程，等待未来发送
- `sent`：已发送
- `send_failed`：发送失败
- `reply_detected`：已检测到导师回复
- `canceled`：该封任务明确被取消，不再继续执行

说明：

- `draft_generated` 可以并入 `review_required`，避免“已生成草稿”和“待审核”两层语义重叠。
- `skipped` 从前台业务语义中移除，不再作为正常执行状态存在。

### 6.3 首页导师关系状态

首页不再直接显示底层任务状态，而是展示派生后的导师关系状态：

- `未开始`
- `准备中`
- `待发送`
- `已联系`
- `已回复`
- `需处理`

推荐派生规则：

- 当前或最近有效任务存在 `reply_detected`：`已回复`
- 当前有效任务为 `approved` 或 `scheduled`：`待发送`
- 当前有效任务为 `discovered`、`matched`、`review_required`：`准备中`
- 当前有效任务为 `send_failed` 或 `canceled`：`需处理`
- 存在已发送历史但当前无待处理任务：`已联系`
- 否则：`未开始`

首页不再显示：

- `已跳过`
- `暂不联系`
- `发送失败`
- `批量停止`

这些细节只在工作区解释。

### 6.4 批量任务停止语义

批量任务停止时：

- `BatchTask.status = stopped`
- 所有未完成、未发送、未回复的子任务改为 `canceled`
- 同时记录 `cancellation_reason = batch_stopped`

这样用户看到的是“这批流程停止了”，而不是“这些导师被系统跳过了”。

### 6.5 工作区继续联系

当用户打开一个 `canceled` 且 `cancellation_reason = batch_stopped` 的任务时，工作区提供：

- 说明：这封邮件属于一个已停止的批量任务，当前未继续处理
- 主动作：`作为单独联系继续`
- 次动作：保留取消状态

点击主动作后，不复用原批量子任务，而是新建一个手动任务。

### 6.6 新建手动任务承接规则

新任务要求：

- `source = manual`
- `batch_task_id = null`
- `parent_task_id = 原批量子任务 id`
- 复制身份、模型、默认材料、已选附件、模板快照、匹配结果

新任务的初始状态按上下文恢复：

- 有有效草稿：`review_required`
- 无草稿但有匹配结果：`matched`
- 尚未算匹配：`discovered`

这样用户不会丢失上下文，批量统计也不会被后续人工操作污染。

## 7. 数据模型调整

### 7.1 `email_tasks`

新增字段：

- `source: TEXT`
  值：`batch` / `manual`
- `parent_task_id: INTEGER NULL`
  指向承接来源任务
- `cancellation_reason: TEXT NULL`
  推荐值：`batch_stopped` / `user_canceled` / `superseded`

调整字段：

- `status` 枚举移除业务语义上的 `skipped`

保留字段：

- `match_score`
- `match_reason`
- `fit_points`
- `risk_points`
- `match_keywords`

这些字段继续为工作区和首页提供解释信息。

### 7.2 `identity_profiles`

`match_threshold` 进入废弃流程：

- 保留数据库字段用于兼容历史数据和迁移
- 前端个人页不再展示和编辑
- 后端不再在任务执行时读取此字段做跳过判断

后续如需彻底删除，可在后续迁移中移除。

## 8. 后端行为调整

### 8.1 匹配计算

匹配计算结束后，无论分数高低都只更新：

- `match_score`
- `match_reason`
- `fit_points`
- `risk_points`
- `match_keywords`

然后把任务状态落到 `matched`。

不再根据 `match_threshold` 写入 `skipped`。

### 8.2 生成草稿

草稿生成的前置条件仍然是：

- 当前任务存在
- 模板配置有效
- 若模式需要默认材料，则当前任务存在可用默认材料

草稿生成完成后统一进入 `review_required`。

### 8.3 发送

发送行为只受以下因素影响：

- 是否有可发送主题和正文
- SMTP 配置是否有效
- 当前任务是否处于允许发送的执行状态

批量任务停止后，原批量子任务不允许继续派发；用户如果要继续联系，必须通过“作为单独联系继续”创建新手动任务。

### 8.4 工作区能力返回

建议后端逐步提供显式能力字段，而不是让前端到处手写状态判断：

- `can_calculate_match`
- `can_generate_draft`
- `can_send`
- `can_schedule`
- `can_retry`
- `can_continue_manually`
- `can_write_follow_up`
- `blocked_reason`

第一阶段可以先不新增接口字段，但后续实现应逐步把能力判断沉到后端。

## 9. 前端行为调整

### 9.1 首页

- 首页状态筛选改为导师关系状态，不再消费原始任务状态。
- 匹配度继续显示，支持筛选和排序，但不触发自动排除。
- `需处理` 的具体原因不在首页展开，只在工作区说明。

### 9.2 创建任务页

- 不再存在“按默认匹配阈值自动筛除低分导师”的入口或默认行为。
- 用户能看到匹配度，但最终由自己决定哪些导师进入任务。

### 9.3 工作区

工作区对不同状态的处理建议如下：

- `matched`：提示先生成草稿
- `review_required`：提示检查后发送
- `approved` / `scheduled`：提示确认发送或调整时间
- `send_failed`：展示失败原因，允许修改后重试
- `canceled` 且原因是 `batch_stopped`：提供“作为单独联系继续”
- `sent`：不再编辑原邮件，提供“写跟进邮件”
- `reply_detected`：展示回复内容，提供“回复 / 跟进”

## 10. 历史数据迁移

对已有 `skipped` 数据执行一次性迁移：

- 若 `status = skipped` 且所属 `batch_task.status = stopped`
  则迁移为 `canceled`，`cancellation_reason = batch_stopped`
- 若 `status = skipped` 且不是因批量停止产生
  则迁移为 `matched`

迁移后：

- 首页不再出现 `已跳过`
- 工作区不会再看到“匹配低所以不能继续”的旧语义

## 11. 测试策略

### 11.1 后端

- 匹配分低于旧阈值时，任务仍落到 `matched`
- 生成草稿后任务进入 `review_required`
- 停止批量任务后，未完成子任务进入 `canceled`
- 从 `batch_stopped` 的取消任务创建手动继续任务时：
  - 新任务 `source = manual`
  - 新任务 `batch_task_id = null`
  - 新任务 `parent_task_id` 指向旧任务
  - 草稿和匹配上下文按规则继承

### 11.2 前端

- 首页状态筛选只显示 `未开始 / 准备中 / 待发送 / 已联系 / 已回复 / 需处理`
- 创建任务页不再有“默认匹配阈值自动筛除”行为
- 工作区 `canceled + batch_stopped` 时显示“作为单独联系继续”
- 工作区 `send_failed` 时显示可重试动作
- 工作区 `sent / reply_detected` 时不再允许编辑原邮件，而是转向跟进动作

## 12. 风险与兼容性

- 旧逻辑里 `match_threshold` 可能已被用户当成“自动筛除规则”使用，移除后需要在更新说明中明确：匹配度仍可筛选，但不再自动跳过。
- `skipped -> matched` 的历史迁移会改变部分导师在首页上的显示状态，需要同步调整首页文案。
- “继续联系时新建手动任务”会让同一位导师在数据库中出现父子任务链，但这比污染原批量任务更清晰，也更利于后续统计。

## 13. 实施分期

### 第一期：语义改正

- 去掉匹配低自动跳过
- 引入 `canceled`
- 为 `email_tasks` 增加 `cancellation_reason`
- 批量停止改为取消未完成子任务
- 首页映射切换到导师关系状态

### 第二期：工作区动作改正

- 为 `email_tasks` 增加 `source`、`parent_task_id`
- 增加“作为单独联系继续”
- 新建手动任务并承接上下文
- 调整 `send_failed / sent / reply_detected / canceled` 的工作区动作

### 第三期：模型清理

- 从个人页移除 `match_threshold`
- 视迁移结果决定是否彻底删除数据库中的 `match_threshold`
