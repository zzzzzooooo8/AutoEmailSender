# 智能抓取加载策略设计

## 背景

最新一次北京邮电大学计算机学院智能抓取暴露出 3 个问题：

- 教师主页所在 host 已经被 HTTP 直连判定为反爬阻断后，后续详情页仍然逐个尝试 HTTP，请求成本高且失败噪声大。
- 详情页浏览器抓取复用了全局 `css:table` 等待条件，而北邮教师主页是 `div`、`ul`、`li` 结构，没有 `table`，导致浏览器已经能拿到页面时仍被判定失败。
- 列表页也不应把 `table` 当成唯一可用结构。不同学校的导师列表可能是表格、卡片、列表或链接集合。

## 目标

- 在同一个抓取任务内记住 host 的 HTTP 阻断状态，避免重复尝试已知会失败的 HTTP 抓取。
- 让浏览器等待策略由页面意图驱动，而不是使用全局固定选择器。
- 详情页不等待 `table`，列表页也不依赖 `table` 作为唯一成功条件。
- 保持改动集中在抓取工具层和详情补全调用点，不引入跨任务持久化策略。

## 非目标

- 不把 host 阻断状态写入数据库。
- 不引入复杂的站点策略配置后台。
- 不改变候选导师保存、审核、去重和 LLM 补全的数据模型。
- 不解决所有网站反爬，只处理当前系统内可控的抓取策略问题。

## 方案

### 任务内 Host 策略记忆

在 `CrawlToolContext` 中增加任务内状态，例如 `http_blocked_hosts`。它只存在于当前 worker 运行过程：

- HTTP 抓取遇到 403、412、429，或明确反爬错误时，将当前 host 标记为 blocked。
- 后续同 host URL 进入 `crawl_page_with_crawl4ai` 时，跳过 HTTP，直接进入浏览器抓取。
- 任务结束后状态自然释放，不影响下一次任务。

这个设计能减少重复失败请求，也避免把某次网络环境下的阻断误判长期固化。

### 页面意图驱动等待策略

新增页面意图参数，例如 `CrawlPageIntent`：

- `directory`：导师列表页。
- `profile`：导师详情页。
- `generic`：普通调查页。

不同入口使用不同意图：

- agent 的普通 `crawl_page` 默认使用 `generic`，避免误判页面结构。
- 明确知道目标是导师列表页的内部调用才使用 `directory`。
- `investigate_with_browser` 继续支持 goal，但底层不能只靠 goal 决定等待条件，默认仍使用 `generic`。
- 详情补全调用明确传 `profile`。

详情页策略：

- 首选等待 `css:body`。
- 抓取成功后通过正文非空、正文长度、邮箱、姓名、院系、研究方向等信号判断是否可用于补全。
- 不要求存在 `table`。

列表页策略：

- 首选等待 `css:body`。
- 不把 `table` 当作加载成功门槛。
- 抓取成功后保留 HTML 和链接，让 agent 或后续解析逻辑识别 `table`、`ul/li`、卡片、链接集合等结构。

### 等待失败降级

浏览器抓取如果因为等待条件失败，不应直接结束。策略如下：

1. 使用页面意图对应的首选配置抓取。
2. 如果失败原因包含 `Wait condition failed`，用更宽松配置重试一次。
3. 降级配置使用 `css:body` 或无 `wait_for`，确保页面 HTML 有机会返回。
4. 只有降级后仍然失败，才记录浏览器抓取失败。

这样可以防止某个学校页面结构和预期不一致时整页失败。

### 成功标准分离

抓取成功和信息补全成功分开判断：

- 抓取成功：浏览器或 HTTP 能返回可解析 HTML，并提取到非空正文。
- 补全成功：从正文中抽取或由 LLM 补齐邮箱、院系、研究方向、近期论文等字段。

详情页不应因为没有某种 DOM 结构而阻止补全阶段运行。

## 数据流

1. 调用方请求抓取 URL，并传入页面意图。
2. `crawl_page_with_crawl4ai` 检查当前 host 是否已被任务内状态标记为 HTTP blocked。
3. 如果未标记，先走 HTTP；如果 HTTP 命中阻断状态，记录快照并标记 host。
4. 如果已标记或 HTTP 结果需要浏览器兜底，进入浏览器抓取。
5. 浏览器按页面意图选择等待策略，必要时执行一次降级重试。
6. 返回 `PageSnapshot`。
7. 详情补全仅在 `PageSnapshot` 成功且正文非空时进入抽取和 LLM 补全。

## 错误处理

- HTTP 阻断状态仍然记录为 `crawl_pages` 中的 HTTP failed 快照，便于排查第一次阻断原因。
- 跳过 HTTP 的后续请求不再写入重复 HTTP failed 快照，只记录浏览器抓取结果。
- 浏览器首选等待失败但降级成功时，最终返回成功快照；必要时可在调试日志里保留首选失败原因。
- 浏览器降级后仍失败时，保留最终错误信息。

## 测试计划

新增或调整后端单元测试：

- 当某 host 首次 HTTP 返回 412 后，当前任务内后续同 host URL 直接进入浏览器抓取。
- 不同 host 不共享 HTTP blocked 状态。
- 详情页抓取使用 `profile` 意图时不传 `css:table`。
- 列表页抓取不把 `table` 作为唯一等待条件。
- 浏览器因 `Wait condition failed` 失败时，会使用宽松配置重试。
- 降级重试成功时，返回成功 `PageSnapshot`，并允许详情补全继续执行。

## 自检

- 没有跨任务持久化，符合「只在当前抓取任务内生效」的约束。
- 没有把 `table` 替换成另一个全局硬编码选择器。
- 方案同时覆盖 HTTP 重复失败、详情页等待错误、列表页结构不确定 3 个问题。
- 改动范围集中在抓取工具层和详情补全调用点，不扩大到候选导师数据模型。
