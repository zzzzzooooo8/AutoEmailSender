# AI 生成草稿研究方向约束设计

## 背景

当前工作区已经支持手动生成草稿、分析匹配度、Tiptap 富文本编辑、导师研究方向字段，以及草稿日志中的 token 用量记录。现有问题是 AI 草稿生成没有把“导师必须有研究方向”作为硬前置条件，同时用户需要 AI 在生成草稿时结合导师研究方向做轻微改写，并在界面显示 token 消耗。

本设计聚焦工作区内导师联系任务的 AI 草稿生成，不改变真实发送、模板发送、批量任务调度和导师抓取流程。

## 目标

- AI 草稿生成必须要求导师 `research_direction` 非空。
- 近期论文不能替代研究方向。即使 `recent_papers` 存在，只要 `research_direction` 为空，就不能使用 AI 生成草稿。
- 模板模式不受此规则限制，因为模板模式不调用 LLM、不消耗 token。
- AI 草稿生成要以现有套磁信模板为基础，只做小幅个性化改写。
- 改写时必须结合导师研究方向，避免泛泛重写。
- 尽量保留原格式，包括段落结构、加粗、斜体、链接等现有受控富文本能力。
- 记录并在工作区显示最近一次 AI 草稿生成的 token 用量。

## 非目标

- 不扩展受控富文本 schema 到完整表格结构。
- 不重做 Tiptap 编辑器。
- 不修复历史草稿日志缺失 token 的情况。
- 不让模板模式显示为 token 消耗行为。
- 不改变匹配度分析的现有逻辑。

## 方案

采用前后端同时收口。

后端作为硬约束，在 `llm` 草稿生成路径中检查导师 `research_direction`。如果为空，直接返回明确错误，不调用 LLM，也不写入草稿日志。前端同步禁用「生成草稿」按钮，并显示原因，减少用户误操作。

这种方案比单纯前端禁用更可靠，也比单纯后端拦截体验更清楚。

## 后端设计

### 前置条件

在 `generate_task_draft()` 的 LLM 模式分支中增加研究方向检查：

- `task.professor.research_direction` 去除首尾空白后为空时，抛出 `ValueError("请先补充导师研究方向，再使用 AI 生成草稿")`。
- 该检查发生在调用 `llm_runtime.generate_draft_content()` 之前。
- 该检查只适用于 `outreach_generation_mode == "llm"`。
- 模板模式继续按原逻辑渲染模板。

### Prompt 调整

调整 `SYSTEM_DRAFT_PROMPT` 与 `build_draft_prompt()` 的任务要求：

- 明确“必须围绕导师研究方向进行个性化改写”。
- 明确“只做轻微修改，不要从零重写”。
- 明确“保留模板整体结构、段落顺序和主要话术风格”。
- 明确“保留可表达的富文本标记，例如加粗、斜体、链接”。

现有 `rich_body` schema 只支持 `paragraph`、`bullet_list`、`numbered_list` 和常见内联节点，因此这次不会完整保留表格。若模板正文包含表格，AI 生成结果应尽量保留表格中的语义内容和顺序，但输出仍以现有受控富文本结构渲染。

### Token 记录

沿用现有 `EmailLog.provider_payload.usage`：

- `prompt_tokens`
- `completion_tokens`
- `total_tokens`

LLM 成功返回 usage 时写入日志；服务商不返回 usage 时字段保持空值。

## 前端设计

### 按钮启用规则

工作区 `canGenerateDraft` 在 LLM 模式下增加研究方向判断：

- 必须存在当前任务。
- 当前任务不能处于已发送、已回信、取消等阻塞状态。
- 必须有模板正文。
- 必须有默认材料。
- 必须有导师研究方向。

模板模式仍只要求模板已配置。

### 提示文案

生成草稿区域的限制提示按优先级显示：

1. 模板缺失：提示先在身份页补充套磁信模板。
2. 默认材料缺失：提示选择用于匹配的材料。
3. 导师研究方向缺失：提示先补充导师研究方向。

研究方向缺失时，「生成草稿」按钮置灰，用户能在按钮附近看到原因。

### Token 显示

在工作区写信区的「生成草稿」区域显示 AI 草稿 token 信息：

- 有最近一次实际用量时，显示“上次消耗：输入 X / 输出 Y / 总计 Z”。
- 只有估算值时，显示“预计上限：输入 X / 输出最多 Y / 总计最多 Z”。
- 没有实际用量和估算值时，显示“暂无 token 记录”。
- 模板模式显示“不消耗 token”。

数据来源使用现有 `WorkspaceTaskSummaryDTO` 字段：

- `last_draft_prompt_tokens`
- `last_draft_completion_tokens`
- `last_draft_total_tokens`
- `estimated_prompt_tokens`
- `estimated_completion_tokens_upper_bound`
- `estimated_total_tokens_upper_bound`

## 格式保留策略

短期策略是遵循现有富文本能力：

- 保留段落顺序。
- 保留列表结构。
- 尽量保留加粗、斜体、链接。
- 不主动改写整封邮件。
- 不新增不必要的段落或复杂结构。

表格属于当前 schema 不支持的格式。若模板包含表格，本次只保证表格中的文本语义尽量保留，不承诺邮件草稿仍是表格。完整表格保留需要扩展后端受控富文本 schema、渲染器和前端预览，应作为后续独立需求。

## 错误处理

- 研究方向缺失：后端返回 400，错误信息为“请先补充导师研究方向，再使用 AI 生成草稿”。
- LLM 调用失败：沿用现有 `LLMRuntimeError` 处理，记录到 `task.last_error`。
- 服务商未返回 usage：草稿仍可生成，token 显示为暂无实际记录。

## 测试计划

后端测试：

- LLM 模式下导师 `research_direction` 为空时，生成草稿接口返回 400。
- 上述失败路径不会调用 LLM，不会新增 draft 日志。
- LLM 模式下导师 `research_direction` 非空时，正常生成草稿，并把 usage 写入 draft 日志。
- Workspace DTO 能返回最近一次草稿的 token 用量。
- 模板模式在导师研究方向为空时仍可生成模板草稿。

前端验证：

- `npm run lint` 通过。
- 导师研究方向为空时，LLM 模式「生成草稿」按钮禁用并显示原因。
- 导师研究方向非空且模板、默认材料齐备时，按钮可用。
- 成功生成 AI 草稿后，写信区显示最近一次 token 消耗。

## 规格自检

- 无占位符、TODO 或未定事项。
- 前后端规则一致：AI 草稿必须有导师研究方向，模板模式不受影响。
- 范围可由一个实现计划覆盖。
- 表格保留边界已明确：本次不扩展 schema，只尽量保留语义和现有富文本格式。
