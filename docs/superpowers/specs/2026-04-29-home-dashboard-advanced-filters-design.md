# 首页导师看板高级筛选设计

## 背景

首页导师看板当前已有关键词、学校、状态和排序能力，但筛选粒度不足。用户需要在创建批量任务前更精确地缩小导师范围，尤其是按同一学校内的学院、系所和教师职称筛选。

现有数据条件：

- 首页通过 `listProfessors` 一次加载 `ProfessorDashboardItemDTO[]`。
- `ProfessorDashboardItemDTO` 已包含 `university`、`school`、`department`、`title`、`match_score`、`sent_count`、`status`。
- 导入模板语义中，`school` 是“学院名称”，示例为“人工智能学院”；`department` 是“院系或系所”，示例为“计算机科学系”。
- 首页排序已在前端本地完成，本期筛选增强沿用同一思路。

## 目标

增强首页筛选区，采用“基础筛选 + 高级筛选”的布局：

- 基础区保持轻量，提供关键词、排序、高级筛选入口和重置。
- 高级区提供学校、学院、系所、职称、状态的下拉多选筛选。
- 高级区提供最低匹配度数字输入，筛选 `match_score >= 输入值` 的导师。
- 所有筛选都作用于当前已加载导师列表，并与排序组合使用。

## 非目标

- 不新增后端筛选参数。
- 不改 `/api/professors` 返回结构。
- 不新增分页或持久化筛选偏好。
- 不调整导师管理页筛选逻辑。
- 不改变批量分析、创建任务、打开工作区等业务动作。

## 字段语义

- 学校：对应 `university`。
- 学院：对应 `school`。
- 系所：对应 `department`。
- 职称：对应 `title`。
- 状态：对应 `status`，显示文案沿用首页当前状态标签。
- 最低匹配度：对应 `match_score`。

`school` 与 `department` 同时纳入高级筛选，界面文案必须区分为“学院”和“系所”，避免用户混淆。

## 交互设计

基础筛选区包含：

- 关键词输入。
- 排序下拉。
- “高级筛选”按钮，显示当前高级筛选数量。
- “重置”按钮，清空关键词、高级筛选和排序选择，恢复默认结果。

高级筛选区包含：

- 学校：下拉多选。
- 学院：下拉多选。
- 系所：下拉多选。
- 职称：下拉多选。
- 状态：下拉多选。
- 最低匹配度：数字输入。
- “清空高级筛选”按钮，只清空高级筛选项，不影响关键词和排序。

高级筛选默认收起。点击“高级筛选”展开，再次点击收起。已有高级筛选生效时，即使收起，也要通过按钮计数让用户知道存在筛选条件。

## 筛选规则

所有筛选在前端本地完成。

关键词筛选：

- 继续使用模糊匹配。
- 匹配字段扩展为姓名、学校、学院、系所、职称、研究方向。

枚举多选筛选：

- 同一筛选组内为“或”关系。例如选择两个学校时，导师属于任一学校即可命中。
- 不同筛选组之间为“且”关系。例如学校命中且职称命中且状态命中才展示。
- 未选择某组时，该组不过滤。
- 空值不生成可选项。

最低匹配度筛选：

- 输入为空时不过滤匹配度，未计算匹配度的导师继续保留。
- 输入 0 到 100 的数字时，只展示 `match_score` 不为空且大于等于该值的导师。
- 输入值应约束在 0 到 100；非数字输入不应进入筛选状态。

结果顺序：

- 筛选先执行，排序后执行。
- “全选当前结果”选择筛选并排序后的当前结果集合。
- 清空选择行为不变。

## 选项生成

高级筛选选项从当前 `professors` 列表派生：

- 学校选项来自非空 `university`。
- 学院选项来自非空 `school`。
- 系所选项来自非空 `department`。
- 职称选项来自非空 `title`。
- 状态选项使用现有 `PROFESSOR_DASHBOARD_STATUS_OPTIONS`。

选项按中文环境 `localeCompare('zh-CN')` 排序。选项值使用原始字符串，展示也使用原始字符串。

本期学院选项不强制随学校联动。原因是多选学校与多选学院组合时，联动规则容易隐藏用户想要选择的学院；先保持全量学院选项，用“不同组之间且关系”实现精确筛选。

## 实现边界

建议新增一个首页筛选 feature：

- `frontend/src/features/home-dashboard/client/filterDashboardProfessors.ts`
- `frontend/src/features/home-dashboard/client/buildDashboardFilterOptions.ts`
- `frontend/src/features/home-dashboard/client/filterDashboardProfessors.test.ts`

首页页面保留为编排层：

- 持有筛选状态。
- 渲染基础区和高级区。
- 调用筛选 helper 得到筛选结果。
- 调用现有排序 helper 得到最终可见结果。

预计修改：

- `frontend/src/pages/HomePage.tsx`

可选新增 UI 小组件：

- `frontend/src/components/molecules/MultiSelectFilter.tsx`

如果直接在 `HomePage.tsx` 内写多选下拉会让页面继续变大，建议抽出一个小组件，职责只处理多选下拉的展示、展开、勾选和清空。

## 测试与验证

自动化测试：

- 覆盖关键词匹配新增字段。
- 覆盖学校、学院、系所、职称、状态多选筛选。
- 覆盖同组“或”、不同组“且”的组合规则。
- 覆盖最低匹配度为空、有效数字和未计算匹配度的处理。
- 覆盖筛选函数不修改输入数组。

验证命令：

```bash
cd frontend
npm run test -- src/features/home-dashboard/client/filterDashboardProfessors.test.ts
npm run lint
npm run build
```

人工验证：

- 默认进入首页时结果不被额外筛选。
- 展开高级筛选后，可以多选学校、学院、系所、职称和状态。
- 输入最低匹配度后，只展示匹配度大于等于该值且已计算匹配度的导师。
- 清空高级筛选后，关键词和排序保持不变。
- 重置后恢复默认筛选和默认排序。
- 高级筛选收起时，按钮仍显示已启用筛选数量。

## 风险与约束

当前筛选基于一次性加载的导师列表，适合现有首页数据流。如果未来导师数量增大并引入分页，需要把筛选迁移到服务端。

学院不随学校联动会让选项数量更多，但规则更稳定，也不会因为多选学校而隐藏可选学院。如果将来需要联动，应在新的规格中明确下拉搜索或按学校分组选项的交互。
