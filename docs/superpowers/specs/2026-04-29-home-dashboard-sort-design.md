# 首页导师看板排序设计

## 背景

首页导师看板当前支持关键词、学校、状态筛选，但缺少排序能力。用户需要在筛选结果中按常用维度调整导师展示顺序，以便更快挑选要分析或创建任务的导师。

现有首页数据流：

- `HomePage.tsx` 通过 `listProfessors` 一次加载导师看板数据。
- 前端本地按关键词、学校、状态过滤。
- 后端 `/api/professors` 默认按导师 `created_at desc` 返回，相当于“最新导入”优先。
- `ProfessorDashboardItemDTO` 当前没有暴露 `created_at` 字段。

## 目标

在首页导师看板筛选区增加一个排序下拉框，让用户可以对当前筛选结果排序。

排序选项：

- 最新导入
- 匹配度高到低
- 发送次数高到低
- 姓名 A-Z

默认排序为“最新导入”，保持当前页面初始展示顺序。

## 非目标

- 不改后端 `/api/professors` 接口参数。
- 不新增分页、服务端排序或持久化排序偏好。
- 不调整导师管理页筛选逻辑。
- 不改变导师选择、批量分析、创建任务等现有业务动作。

## 交互设计

首页顶部筛选区新增“排序”下拉框，与“关键词”“学校”“状态”并列。

用户切换排序后：

- 只改变列表展示顺序。
- 不清空关键词、学校、状态筛选。
- 不清空已选择导师。
- “全选当前结果”仍选择当前筛选后的所有导师，排序不改变选择集合。

## 排序规则

排序在前端本地完成，作用于筛选后的结果。

规则如下：

- 最新导入：保持后端返回顺序，不额外排序。
- 匹配度高到低：`match_score` 数值越高越靠前；未计算匹配度的导师排到最后。
- 发送次数高到低：`sent_count` 数值越高越靠前。
- 姓名 A-Z：按 `name.localeCompare` 排序，空值兜底为空字符串。

为避免原数组被原地修改，排序前应复制筛选结果。

## 实现边界

预计只修改：

`@frontend/src/pages/HomePage.tsx`

建议在 `HomePage` 中新增排序状态和排序选项类型。列表渲染、数量统计和“全选当前结果”都使用排序后的派生数组，保持用户看到的结果和操作对象一致。

## 测试与验证

实现后至少执行：

```bash
cd frontend && npm run lint
```

人工验证：

- 默认进入首页时顺序保持为最新导入。
- 切换“匹配度高到低”后，已计算匹配度的导师按分数降序排列，未计算的排后面。
- 切换“发送次数高到低”后，发送次数多的导师排前面。
- 切换“姓名 A-Z”后，导师按姓名升序排列。
- 关键词、学校、状态筛选与排序可以组合使用。
- “全选当前结果”和“清空选择”行为不变。

## 风险与约束

当前前端 DTO 没有 `created_at`，所以“最新导入”必须依赖后端现有默认顺序。如果未来后端默认顺序变化，首页默认排序也会随之变化；若需要完全稳定，应后续在接口中暴露创建时间或增加服务端排序参数。
