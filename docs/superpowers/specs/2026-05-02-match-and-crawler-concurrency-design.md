# 匹配分析与智能抓取并发设计

## 背景

当前系统已经分别具备「分析匹配度」和「智能抓取」两条能力链路，但两者的并发能力不对称：

- 匹配分析前端已经有 warm-up + 固定并发池，能在批量场景下缩短等待时间，但后端缺少同一任务防重入保护。
- 智能抓取运行在异步后台 worker 中，但当前 worker 默认只有 1 个；单个抓取任务内部也基本按串行执行。
- 抓取侧最近已经补上保存失败熔断、字段归一化和运行指标持久化，说明系统已经开始建立稳定性边界，具备继续引入受控并发的基础。

如果直接把两条链路都改成“尽量多开并发”，最容易放大的问题不是速度，而是重复运行、重复写入、站点限流、数据库争用和错误定位困难。

本设计选择在现有架构上增加分层受控并发，让系统先具备可验证的并发边界，再逐步提高吞吐。

## 目标

- 匹配分析支持更快的批量执行，同时避免同一 `email_task` 被重复分析。
- 智能抓取同时提升系统吞吐和单任务完成速度，但不牺牲候选保存正确性。
- 默认失败策略为「继续跑完其他子任务，最后汇总失败项」。
- 并发参数可配置，并提供保守默认值，方便逐步放量。
- 日志、指标和错误摘要能够说明「哪些步骤并发了、哪些失败了、失败后系统做了什么」。

## 非目标

- 不把匹配分析改造成独立的后台队列系统。
- 不把智能抓取重构成分布式任务平台。
- 不在本次设计中引入 Redis、分布式锁或跨进程消息队列。
- 不追求所有站点都使用同一套最高并发参数。
- 不改变当前「抓取候选审核后才导入正式导师」的业务流程。

## 设计原则

### 1. 分层并发，不做全链路无界并发

不同阶段采用不同并发值：

- 匹配分析：请求级受控并发。
- 抓取任务：任务级并发。
- 抓取任务内部：详情页富化并发。
- 数据库写入和状态变更：保持单写入通道或单条事务边界。

### 2. 幂等优先于吞吐

并发的首要目标不是“同时跑更多”，而是“同一输入不会因为并发产生重复结果或脏状态”。

### 3. 失败隔离

单个导师、单个详情页、单次 LLM 调用失败，不应拖垮整批任务；只有终止级错误才结束当前任务。

### 4. 可观测

每一层并发都必须能回答三个问题：

- 当前最大活跃数是多少？
- 哪些子任务成功、失败、跳过？
- 失败是可重试、不可重试，还是终止级？

## 方案选择

本次采用「方案 B：分层受控并发」。

不选方案 A 的原因：

- 只增加 worker 数量虽然安全，但对单个抓取任务的完成时间改善有限。
- 匹配分析如果只停留在前端并发，而后端没有防重入，仍然存在重复分析风险。

不选方案 C 的原因：

- 全链路高并发会同时放大站点限流、重复保存和状态一致性风险。
- 现有项目尚未引入足够的锁、队列和调度基础设施来支撑激进方案。

## 总体架构

### 匹配分析

采用「前端有限并发 + 后端防重入」结构：

1. 前端继续负责批量触发与结果汇总。
2. 后端负责同一任务防重入和最终审计落库。
3. 同一 `email_task` 同时最多存在 1 个运行中的匹配分析。

### 智能抓取

采用「任务级并发 + 任务内分阶段并发 + 单写入通道」结构：

1. 多个后台 worker 可以同时领取不同抓取任务。
2. 单个抓取任务内部只对适合并发的详情页富化阶段放开并发。
3. 候选保存、运行状态变更和最终统计仍由单协程或单事务串行收口。

## 匹配分析并发设计

### 现状

当前批量匹配已经通过 `runWarmupThenConcurrent` 实现「先 warm-up 第一个，再并发跑剩余项」。这一层解决了前端等待体验，但没有覆盖两个风险：

- 同一导师被重复点击或多个页面同时触发时，后端会重复运行。
- 后端无法明确区分「正在运行」和「已完成但失败」。

### 推荐方案

#### 前端并发模型

- 保留 warm-up 机制，不改交互语义。
- 批量默认并发值为 `3`。
- 前端配置项命名建议为 `match_analysis_concurrency`。
- 允许后续通过设置页或环境变量将其调整到 `5`，但默认不超过 `5`。

warm-up 规则：

1. 第一个可分析导师串行执行。
2. 其余导师进入固定并发池。
3. 最终仍只弹出 1 条批量结果汇总消息。

#### 后端防重入模型

在 `match_analysis_runs` 上补充运行状态字段，而不是复用 `email_tasks.status`。

新增字段：

```text
status            // running | succeeded | failed
started_at
finished_at
error_kind        // llm_runtime | unexpected
```

新增约束：

- 对 `match_analysis_runs(email_task_id)` 建立「同一时刻最多 1 条 running 记录」的唯一约束。

请求流程：

1. `POST /calculate-match` 进入后，先执行默认材料、研究证据等前置校验。
2. 前置校验通过后，尝试创建一条 `status=running` 的 run。
3. 如果唯一约束冲突，说明该任务已在分析中，接口返回 `409 Conflict`。
4. 前端将这类错误展示为「该任务正在分析中」，不再发起第二次请求。
5. 分析成功后，将 run 更新为 `succeeded`，并写入 usage、得分和 `finished_at`。
6. 分析失败后，将 run 更新为 `failed`，写入错误摘要和 `finished_at`。
7. 前置校验失败不创建 run，直接按 skipped 或请求错误处理。

选择该方案的原因：

- 防重入依赖数据库约束，跨请求、跨线程仍然成立。
- 审计表天然能记录运行中、成功和失败，不需要再引入独立锁表。
- 不污染现有 `email_tasks` 业务状态。

#### 匹配分析失败策略

匹配分析只分三类结果：

- 成功：返回 usage，计入 success。
- 跳过：缺少默认材料、缺少研究证据等前置校验失败，计入 skipped。
- 失败：模型调用失败、解析失败、运行时异常，计入 failed。

批量场景下：

- 单个失败不会中断其余项。
- 所有项结束后统一汇总 `成功 / 失败 / 跳过`。
- 失败详情最多展示前 3 项，避免通知过长。

## 智能抓取并发设计

### 现状

当前抓取运行时已经有清晰的任务边界：

- `run_queued_crawl_jobs_once` 一次只领取 1 个 queued job。
- `_run_profile_crawl_job` 处理单个详情页入口。
- `_enrich_saved_candidates` 对待补全候选逐个串行抓详情页、抽字段、写回数据库。

其中最适合做受控并发的是 `_enrich_saved_candidates`，因为它天然是「同一批待补全候选上的重复步骤」。

### 推荐方案

#### 任务级并发

将 crawler worker 从固定 1 个循环改为可配置数量。

新增配置：

```text
crawler_worker_count = 2
```

默认值为 `2`，不直接上 `3+`，原因是：

- 现有抓取站点类型复杂，先用 2 个 worker 验证限流与 CPU 开销更稳妥。
- 领取任务逻辑已经基于数据库状态做 claim，具备多 worker 并发领取基础。

#### 任务内阶段划分

单个抓取任务拆成 3 个阶段：

1. 发现阶段：抓列表页、收集候选或 profile URL。
2. 富化阶段：抓详情页并抽取研究方向、论文、邮箱等缺失字段。
3. 保存阶段：把候选或补全结果写回数据库并记录 trace。

并发策略：

- 发现阶段：低并发，保持串行或最多 `1` 到 `2`。
- 富化阶段：中并发，默认 `3`。
- 保存阶段：单写入通道。

#### 富化阶段并发模型

把 `_enrich_saved_candidates` 改造成固定并发池，而不是简单 `gather` 全开。

新增配置：

```text
crawler_profile_enrichment_concurrency = 3
crawler_host_concurrency = 1
crawler_profile_fetch_max_retries = 2
crawler_profile_queue_max_size = 50
```

执行模型：

1. 先查询当前 job 下待补全候选列表。
2. 创建有界队列，队列元素为待补全候选。
3. 启动固定数量的 enrichment worker。
4. 每个 worker 取一个候选后执行：
   - 检查 job 是否仍可继续；
   - 获取对应 host 的 semaphore；
   - 抓取详情页；
   - 调用 LLM 补全或 fallback 规则补全；
   - 生成「补全结果事件」发送到保存队列。
5. 保存协程从保存队列顺序消费，统一执行 `_apply_candidate_enrichment` 和 trace 写入。

这样做的原因：

- 抓取和解析是 IO/LLM 密集型，适合并发。
- 写数据库和写 trace 更适合单通道顺序收口，避免更新覆盖和计数混乱。

#### Host 级限流

对每个 hostname 维护独立 semaphore，默认同 host 并发为 `1`。

规则：

- 同一学校站点的多个详情页默认串行抓，降低被封风险。
- 不同 host 可以并发执行。
- 若后续证实某些站点更稳定，可按 host 白名单放宽到 `2`。

#### 保存阶段单写入通道

保存协程负责以下动作：

- 执行 `_apply_candidate_enrichment`。
- 统一计算 enriched / unchanged / failed 计数。
- 统一写入 trace。
- 汇总失败原因。

不允许多个 enrichment worker 直接并发写同一批候选，原因是：

- 现在的运行指标和 trace 逻辑依赖顺序性。
- 直接并发写入更容易出现最后写入覆盖前一轮字段的情况。

#### 多起始 URL 策略

当前同一 job 的多个 `start_url` 是按 `for start_url in start_urls` 串行执行。

本次保持这一行为不变，不对 `start_url` 级别并发化，原因是：

- 当前任务级并发已经会同时运行多个 job。
- 单 job 内同时并发多个入口，容易放大重复候选和站点压力。
- 本次优先把并发预算用在详情页富化阶段。

## 抓取错误与重试策略

### 错误分类

抓取并发下统一分为三类：

#### 可重试错误

- HTTP 超时
- 临时网络错误
- 429
- 临时 5xx
- 浏览器抓取超时

处理规则：

- 单个详情页最多重试 `2` 次。
- 使用指数退避，建议 `1 s -> 2 s`。
- 只重试详情页抓取，不重试整个 job。

#### 不可重试错误

- 页面正文为空且 fallback 后仍无法提取
- 结构化输出不合法
- 页面不包含有效导师信息
- 候选没有可用 profile URL

处理规则：

- 记单项失败并继续处理其他候选。
- 失败项进入 job summary。

#### 终止级错误

- `CrawlJobCanceled`
- `CrawlJobPaused`
- `CrawlJobSaveBudgetExceeded`
- 数据库不可写
- 保存协程崩溃

处理规则：

- 结束当前 job。
- 停止继续消费待抓取候选。
- 保留已成功保存的数据和已写 trace。

### 失败预算与汇总

在现有保存失败熔断机制之外，为富化阶段补充运行摘要：

```text
candidate_count
success_count
unchanged_count
failed_count
retry_count
host_limited_count
top_failure_reasons
```

任务完成后前端展示：

- 成功多少位
- 未变化多少位
- 失败多少位
- 是否命中过限流或重试

## 并发参数表

| 参数名 | 默认值 | 作用层级 | 说明 |
| --- | --- | --- | --- |
| `match_analysis_concurrency` | `3` | 前端匹配批量 | 匹配分析批量并发上限 |
| `crawler_worker_count` | `2` | 后台任务级 | 同时运行的 crawl job 数量 |
| `crawler_profile_enrichment_concurrency` | `3` | 单 job 富化级 | 同一抓取任务内并发补全详情页数量 |
| `crawler_host_concurrency` | `1` | 单 host | 同一站点详情页抓取并发上限 |
| `crawler_profile_fetch_max_retries` | `2` | 单详情页 | 可重试错误的最大重试次数 |
| `crawler_profile_queue_max_size` | `50` | 单 job 队列 | 待补全候选队列最大长度 |

## API 与数据模型变更

### 匹配分析

- 扩展 `match_analysis_runs`：
  - `status`
  - `started_at`
  - `finished_at`
  - `error_kind`
- `POST /api/email-tasks/{task_id}/calculate-match` 增加 `409 Conflict` 语义：
  - 当前 task 已有运行中的匹配分析时返回

### 智能抓取

不新增外部 API，但补充内部运行指标字段。推荐在 `crawl_job_runs` 增加：

```text
retry_count
host_limited_count
failed_candidate_count
unchanged_candidate_count
```

这些字段用于任务列表、详情面板和后续并发调优，不影响现有抓取审核接口。

## 可观测性

### 日志

新增或补充以下 trace event：

- `match_analysis_conflict`
- `enrichment_worker_started`
- `enrichment_retry_scheduled`
- `enrichment_host_limited`
- `enrichment_result_saved`
- `enrichment_summary`

### 指标

至少记录：

- 匹配分析最大活跃并发数
- 抓取 worker 活跃数
- 单 job 富化最大活跃数
- 每 host 实际并发命中次数
- 可重试失败次数
- 最终失败次数

## 测试策略

### 匹配分析

需要覆盖：

1. 同一 `email_task` 并发触发 2 次时，只允许 1 次进入运行态，另一次返回 `409`。
2. 批量匹配 5 位导师时，最大活跃数不超过配置值。
3. warm-up 仍然先跑第 1 个，再并发跑剩余项。
4. 单项失败不会中断整批，最终汇总包含 success / failed / skipped。

### 智能抓取

需要覆盖：

1. `crawler_worker_count=2` 时，两个 queued job 可以被不同 worker 同时领取。
2. 单个 job 富化 10 位候选时，最大活跃抓取数不超过 `crawler_profile_enrichment_concurrency`。
3. 同一 host 的多个详情页不会超过 `crawler_host_concurrency`。
4. 可重试错误最多重试 2 次，最终失败后继续其他候选。
5. 保存协程串行消费时，不会产生重复写入或计数错乱。
6. job 被取消后，未消费的候选不会继续写库。
7. 保存失败熔断触发后，任务按终止级错误结束。

## 风险与缓解

### 风险 1：前端并发和后端防重入同时存在，可能出现“用户看到失败但后端其实在跑”

缓解：

- 对 `409` 返回统一错误码和消息。
- 前端收到 `409` 时展示「该任务正在分析中」，不显示普通失败。

### 风险 2：抓取并发后，站点更容易触发限流或封禁

缓解：

- 默认 `crawler_host_concurrency=1`。
- 默认 worker 数仅 `2`。
- 对 429 和超时使用保守重试。

### 风险 3：抓取并发后，数据库更新顺序变复杂

缓解：

- 把数据库写入收敛到保存协程。
- 运行状态和统计字段只在保存协程或任务收尾阶段更新。

### 风险 4：并发参数过高导致本地资源占满

缓解：

- 所有参数都配置化。
- 默认值保守。
- 后续调优以真实日志和指标为依据，不靠一次性放大。

## 迁移与发布顺序

建议分 3 步上线：

1. 先给匹配分析补后端防重入和 `409` 语义。
2. 再把 crawler worker 数从 `1` 提升到 `2`，验证任务级并发。
3. 最后引入 `_enrich_saved_candidates` 的受控并发和单写入通道。

这样做可以在每一步都单独验证问题来源，避免一次性改动过多导致回归难以定位。

## 验收标准

- 批量匹配 4 位及以上导师时，保持 warm-up 语义且最大并发不超过配置值。
- 同一导师任务被重复触发匹配分析时，不会产生并行重复运行。
- 同时创建 2 个抓取任务时，后台可以并行推进。
- 单个抓取任务存在多个待补全候选时，详情页富化明显快于串行版本。
- 任意单项失败不会导致整批立即终止，最终会返回完整汇总。
- 日志和指标能明确说明并发是否命中上限、哪些子项失败、是否发生重试。
