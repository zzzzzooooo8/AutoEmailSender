# 智能抓取诊断日志导出与取消任务候选审核设计

## 背景

当前智能抓取已经具备原始流式事件的 JSONL 记录能力，但默认通过 `CRAWLER_DEBUG=false` 关闭。用户在排查抓取问题时，往往需要查看一次任务的完整抓取过程，包括页面抓取、候选保存、失败原因和中断位置。

与此同时，智能抓取任务在被用户手动取消或因系统进程中断而意外结束时，如果已经保存了一部分候选导师，当前流程没有把这批已抓到但未完成的候选明确暴露出来，用户无法继续人工审核并选择部分导师入库。

## 目标

- 默认开启智能抓取 JSONL 调试记录，仍允许通过环境变量关闭。
- 在个人中心的“开发诊断日志”里增加最近 50 次抓取任务的下拉选择。
- 用户选中某次抓取任务后，可以直接导出该任务对应的完整 JSONL 调试数据。
- 抓取任务被手动取消或意外中断后，只要已有候选导师落库，就允许继续在任务详情里审核并选择部分候选添加到系统。
- `paused` 任务不进入这条补救审核流程。

## 非目标

- 不改造智能抓取的核心发现逻辑。
- 不把 JSONL 调试数据并入现有操作日志表。
- 不新增候选审核之外的自动修复策略。
- 不把 `paused` 任务视为已中断待审核。
- 不新增面向普通用户的抓取日志总览页。

## 方案选择

采用“任务级 JSONL 文件 + 个人中心下拉导出 + 取消后继续审核”的组合方案。

后端继续以文件形式保存抓取调试事件，默认开启后，抓取任务会自然留下可导出的完整上下文。个人中心只负责发现最近 50 次抓取任务并触发导出，不负责解析 JSONL 内容。对于中断后的候选审核，后端放宽审核入口对 `canceled` 任务的限制，只要任务下仍有候选记录，就允许继续执行审核并入库。

没有选择把 JSONL 写入数据库，因为这会显著放大存储和迁移成本，也会把调试数据和业务数据耦合得过紧。没有选择把取消任务自动切成 `needs_review`，因为这会抹掉任务已结束的真实状态。

## 抓取调试日志

### 默认行为

`CRAWLER_DEBUG` 默认值改为 `true`。如果用户显式设置为 `false`，仍然关闭写入。

默认输出目录继续使用现有约定：`data/logs/crawler`，单任务文件名保持为：

```text
crawl-job-<job_id>.jsonl
```

### 文件内容

JSONL 每行仍然是一条抓取事件记录，沿用现有脱敏规则，不新增明文敏感字段输出。

### 导出入口

新增按任务导出的接口，按 `crawl_job_id` 返回对应 JSONL 文件下载。接口只负责文件读取和流式输出，不解析内容、不重写格式。

## 个人中心交互

在“开发诊断日志”卡片中新增一个抓取任务下拉框：

- 默认展示最近 50 次抓取任务。
- 下拉项优先展示任务状态、学校、学院、创建时间，必要时补充入口 URL。
- 用户选择任务后，点击“导出抓取日志”下载该任务 JSONL。
- 现有的后端操作日志导出继续保留，两个导出入口分开。

### 空状态

- 没有抓取任务时，下拉框显示为空。
- 导出按钮在未选择任务时禁用。

## 取消后继续审核

### 适用范围

- 适用：`canceled` 任务，且任务下存在至少 1 条候选导师。
- 不适用：`paused` 任务。

### 交互规则

任务详情页继续展示候选列表和审核操作。对于 `canceled` 任务：

- 仍显示任务已取消的状态。
- 允许进入候选审核。
- 允许勾选部分候选并执行入库。
- 审核完成后，任务状态可以保持 `canceled`，不强制改写为 `needs_review` 或 `completed`。

这样既保留任务真实结束状态，也允许用户对已抓到的候选做人工补救。

### 后端规则

现有 `approve` 接口当前只接受 `needs_review`。这里改为：

- `needs_review` 允许审核。
- `canceled` 只要有候选也允许审核。
- `paused` 继续拒绝。
- `completed`、`failed` 的审核行为保持现状，若现有逻辑已允许则不额外改动。

如果任务被意外中断但已有候选，后端仍按 `canceled` 处理，不额外引入新状态。

## API 与数据边界

### 新增或修改的后端能力

- 读取最近 50 条抓取任务摘要，用于下拉框。
- 按抓取任务 ID 导出 JSONL 文件。
- 放宽抓取候选审核对 `canceled` 任务的限制。

### 需要保留的边界

- JSONL 导出不走 `operation_logs`。
- 抓取调试文件读取失败时只影响导出，不影响任务数据。
- 审核流程只处理已存在的候选，不回头重跑抓取。

## 测试计划

后端测试：

- `CRAWLER_DEBUG` 默认开启，环境变量可关闭。
- 抓取任务导出接口可返回指定任务的 JSONL。
- 最近 50 次抓取任务可供前端下拉选择。
- `canceled` 任务在有候选时可继续审核。
- `paused` 任务继续拒绝审核。
- 意外中断场景下，只要候选已保存，审核入口仍可用。

前端测试：

- 个人中心“开发诊断日志”可以拉取并展示最近 50 次抓取任务。
- 选中任务后，导出按钮会携带对应任务 ID。
- 未选择任务时导出按钮保持禁用。

## 成功标准

- 默认情况下，智能抓取会留下可导出的完整 JSONL。
- 用户可以在个人中心直接选最近 50 次抓取任务并导出对应日志。
- `canceled` 任务如果已有候选，仍可继续审核和入库。
- `paused` 任务不被误当成中断善后场景。
- 现有抓取与审核主流程不回归。
