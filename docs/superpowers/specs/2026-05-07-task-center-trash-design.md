# 任务中心回收站设计

## 1. 背景

任务中心现在展示批量邮件、教师抓取、匹配分析 3 类任务。任务数量持续增长后，已完成、已取消或失败后不再处理的历史任务会长期堆在列表里，影响用户查找仍需处理的任务。

本次设计把用户语义定为「删除 / 回收站 / 恢复」。删除不是物理清除数据，而是把任务从当前列表隐藏，并允许用户在回收站恢复。

## 2. 设计目标

- 所有任务类型都支持删除到回收站和从回收站恢复。
- 当前任务列表默认不显示已删除任务，避免任务堆积。
- 删除行为不改变正在运行的任务状态；运行中任务必须先中止、取消或处理结束后才能删除。
- 用户无需理解「归档」概念，界面统一使用「删除」「回收站」「恢复」。
- 已删除任务保留详情和历史记录，方便用户回看和恢复。

## 3. 非目标

- 本次不做物理删除，不清除任务、子任务、日志、候选人或匹配结果。
- 本次不新增跨类型统一任务时间线。
- 本次不重构任务中心整体页面布局。
- 本次不改变批量邮件、教师抓取、匹配分析已有的运行、中止、取消和重试语义。

## 4. 方案选择

### 4.1 已比较方案

1. 每个任务分类内加「当前任务 / 回收站」切换。
2. 任务中心顶部新增统一「回收站」页签，混合展示所有任务类型。
3. 只在列表加「显示已删除」开关，不提供明确回收站入口。

最终选择第 1 种方案。

### 4.2 选择理由

- 删除后的任务仍属于原任务类型，放在同类回收站中最容易理解。
- 三类任务卡片字段差异较大，混排会增加 UI 和数据聚合复杂度。
- 「当前任务 / 回收站」切换保留现有分类、分页、详情抽屉和操作边界，改动可控。

## 5. 交互设计

### 5.1 页面结构

任务中心保留现有 3 个分类页签：

- 批量邮件
- 教师抓取
- 匹配分析

每个分类内部新增一个轻量切换：

- 当前任务
- 回收站

切换只影响当前分类。例如在「教师抓取」页签进入回收站，只展示已删除的教师抓取任务。

### 5.2 当前任务列表

当前任务列表默认只展示未删除任务。

每张任务卡增加「删除」操作。删除前弹出确认：

> 删除后会移入回收站，不会清除任务记录，可在回收站恢复。

确认后调用对应类型的删除接口。删除成功后：

- 从当前列表移除该任务。
- 刷新当前分类列表和计数。
- 如果详情抽屉正在展示该任务，关闭详情抽屉。

### 5.3 回收站列表

回收站列表只展示同类型已删除任务。

回收站内任务卡保留：

- 任务名称、状态、进度或结果摘要
- 创建时间、更新时间
- 查看详情
- 恢复

回收站内不提供运行控制动作，例如暂停、继续、中止、取消、重试。

恢复成功后：

- 任务从回收站列表移除。
- 任务回到同类型当前任务列表。
- 当前仍停留在回收站视图，用户可继续处理其他已删除任务。

### 5.4 可删除状态

删除只允许用于已结束任务。

批量邮件任务可删除状态：

- `stopped`
- `completed`

批量邮件任务不可删除状态：

- `running`
- `paused`

教师抓取任务可删除状态：

- `completed`
- `failed`
- `canceled`

教师抓取任务不可删除状态：

- `queued`
- `running`
- `paused`
- `needs_review`

匹配分析任务可删除状态：

- `completed`
- `partial_failed`
- `failed`
- `canceled`

匹配分析任务不可删除状态：

- `queued`
- `running`

不可删除时，前端可以禁用删除按钮或不展示删除按钮；后端仍必须做状态校验。后端拒绝时返回 `400`，提示：

> 请先中止/取消任务后再删除。

## 6. 数据模型

### 6.1 新增字段

3 张任务主表新增统一字段：

- `batch_tasks.deleted_at: DATETIME NULL`
- `crawl_jobs.deleted_at: DATETIME NULL`
- `match_analysis_jobs.deleted_at: DATETIME NULL`

字段语义：

- `NULL` 表示当前任务。
- 非 `NULL` 表示任务已进入回收站。
- 时间值记录删除发生时间，用于后续排序和审计。

### 6.2 字段命名

代码使用 `deleted_at`，而不是 `archived_at`。原因是前端用户文案使用「删除」和「回收站」，字段名保持一致能减少理解成本。

导师管理已有 `archived_at` 字段，本次不迁移或重命名导师相关数据结构。

## 7. 后端接口

### 7.1 列表过滤

三类列表接口增加回收站视图参数，参数名使用 `view`：

- `view=current`：默认值，只返回 `deleted_at is null`。
- `view=trash`：只返回 `deleted_at is not null`。

涉及接口：

- `GET /api/batch-tasks?view=current|trash`
- `GET /api/crawl-jobs?view=current|trash`
- `GET /api/match-analysis-jobs?view=current|trash`

现有身份、模型、数量限制等查询参数保持不变。

### 7.2 删除接口

新增接口：

- `POST /api/batch-tasks/{task_id}/delete`
- `POST /api/crawl-jobs/{job_id}/delete`
- `POST /api/match-analysis-jobs/{job_id}/delete`

行为：

- 校验任务存在。
- 校验任务处于可删除状态。
- 如果 `deleted_at` 已有值，直接返回当前任务，保持幂等。
- 否则写入当前时间到 `deleted_at`，更新 `updated_at`。
- 记录操作日志。

### 7.3 恢复接口

新增接口：

- `POST /api/batch-tasks/{task_id}/restore`
- `POST /api/crawl-jobs/{job_id}/restore`
- `POST /api/match-analysis-jobs/{job_id}/restore`

行为：

- 校验任务存在。
- 如果 `deleted_at` 已为 `NULL`，直接返回当前任务，保持幂等。
- 否则清空 `deleted_at`，更新 `updated_at`。
- 记录操作日志。

### 7.4 详情接口

详情接口仍允许读取回收站任务。这样用户在回收站里可以查看历史详情，再决定是否恢复。

子资源接口也应保持可读，例如：

- 批量任务子任务列表
- 抓取任务页面、候选人、事件
- 匹配分析任务明细

## 8. 后台调度约束

虽然前端和删除接口会禁止删除未结束任务，后端调度查询仍必须排除已删除任务，避免异常状态漏网。

需要排除 `deleted_at is not null` 的场景包括：

- 批量邮件调度器扫描可继续处理的批量任务。
- 抓取任务调度器扫描排队或暂停恢复后的任务。
- 匹配分析任务调度器扫描排队或运行中的任务。

如果某个已删除任务因为旧数据或并发问题仍处于运行态，调度器不应继续推进它。

## 9. 前端设计

### 9.1 类型和 API

前端 DTO 增加 `deleted_at: string | null`：

- `BatchTaskCardDTO`
- `CrawlJobSummaryDTO`
- `MatchAnalysisJobDTO`

API 客户端新增：

- `deleteBatchTask`
- `restoreBatchTask`
- `deleteCrawlJob`
- `restoreCrawlJob`
- `deleteMatchAnalysisJob`
- `restoreMatchAnalysisJob`

列表 API 增加 `view` 参数。

### 9.2 状态管理

`TasksPage` 为每个分类维护独立回收站视图状态，定义类型：

```ts
type TaskListView = "current" | "trash";
```

为了让用户切换分类时保留各自视图，使用分类型状态：

```ts
const [taskListViews, setTaskListViews] = useState<Record<TasksTab, TaskListView>>({
  batch: "current",
  crawl: "current",
  match: "current",
});
```

列表加载时按当前分类传入 `view`。

### 9.3 计数规则

任务中心顶部统计和页签数量默认只统计当前任务，不统计回收站任务。

包括：

- 批量邮件数量
- 教师抓取数量
- 匹配分析数量
- 运行中数量
- 待处理数量

本次「当前任务 / 回收站」切换按钮不显示额外数量。原因是现有列表接口返回数组而非分页总数，显示已删除数量需要额外查询或扩展聚合接口，不属于本次清理任务堆积的必要范围。

### 9.4 文案

界面统一使用：

- 删除
- 回收站
- 恢复
- 当前任务

不在界面使用：

- 归档
- 隐藏

错误提示：

- 运行中或未结束任务：`请先中止/取消任务后再删除。`
- 删除失败：`删除任务失败`
- 恢复失败：`恢复任务失败`

成功提示：

- `已移入回收站`
- `已恢复任务`

## 10. 操作日志

新增操作日志事件：

- `batch_task.deleted`
- `batch_task.restored`
- `crawl_job.deleted`
- `crawl_job.restored`
- `match_analysis_job.deleted`
- `match_analysis_job.restored`

记录元数据：

- 任务状态
- 删除或恢复前的 `deleted_at`
- 身份 ID 和模型 ID（如果任务类型有这些字段）

## 11. 幂等与并发

删除和恢复都要保持幂等：

- 已在回收站的任务再次删除，返回成功。
- 未在回收站的任务再次恢复，返回成功。

并发场景下以后端最终写入为准。前端收到成功响应后刷新当前分类列表，不依赖本地乐观更新作为最终状态。

## 12. 测试计划

### 12.1 后端测试

覆盖三类任务：

- 当前列表不返回 `deleted_at is not null` 的任务。
- 回收站列表只返回 `deleted_at is not null` 的任务。
- 已结束任务可以删除。
- 未结束任务删除返回 `400`。
- 删除后可以恢复。
- 重复删除和重复恢复保持成功。
- 详情和明细接口可读取回收站任务。

### 12.2 前端测试

覆盖任务中心：

- 每个分类都能在「当前任务 / 回收站」间切换。
- 当前列表展示删除入口，回收站展示恢复入口。
- 不可删除状态不展示或禁用删除入口。
- 删除确认后刷新当前列表。
- 恢复成功后刷新回收站列表。
- 回收站任务不参与顶部统计和页签数量。

### 12.3 手动验证

至少验证 3 条完整路径：

- 批量邮件任务结束后删除，进入回收站，查看详情，恢复。
- 教师抓取任务取消后删除，进入回收站，查看候选人和事件，恢复。
- 匹配分析任务失败后删除，进入回收站，查看明细，恢复。

## 13. 验收标准

- 三类任务都可以在已结束状态下删除到回收站。
- 三类任务都可以从各自回收站恢复。
- 未结束任务不能删除，且提示清楚。
- 当前任务列表和统计默认不包含回收站任务。
- 回收站任务详情可读，但不可继续执行运行类动作。
- 后端调度不会继续推进已删除任务。
- 删除和恢复接口具备幂等性。
