# 锚点化模板改写设计

## 背景

当前工作区草稿生成使用模板 run 级改写。后端把富文本 HTML 中的文本节点拆成
`segment_id` 和 `run_id`，模型只能返回对应 run 的替换文本，再由后端写回原始
HTML。

这个机制能保留格式，但原始 run 是由 HTML 文本节点和局部样式切出来的，不是自然
语言边界。实际生成中已经出现以下问题：

- `科研` 和 `项目）：...` 被拆到相邻 run，模型把后者改成 `科研项目）：...`，
  最终拼成 `科研科研项目`。
- `未来规划` 和独立逗号被拆开，模型在前一 run 末尾补标点后，原逗号仍保留。
- `推免资格` 后的句号位于下一个普通 run 开头，模型改写该 run 时删除句号，
  导致 `推免资格冒昧联系您`。

已补充的 `segment_text` 能让模型看到连续上下文，但不能约束模型稳定保留跨 run
边界标点，也不能避免同一 run 内的冗余表达。

## 外部参考

- Tiptap Content AI 的 Server AI Toolkit 采用专用编辑工作流：模型输出结构化编辑
  结果，再由工具层执行，不让模型直接操作完整 HTML。
- Tiptap demo 中的 `tiptap-edit-workflow` 使用 workflow 提供的 system prompt 和
  schema，让模型输出受控对象。
- `python-docx` 社区曾讨论同类问题：按 paragraph 改写会丢样式，按 run 改写会
  破坏句子语义。
- DocuTranslate 的 DOCX 方案会把相邻 run 合并成可翻译文本段，只按显著格式、
  图片、字段等结构边界切分，再把译文写回文档结构。

这些实现共同指向一个原则：模型不应直接处理格式碎片；格式应由后端结构化保留。

## 目标

1. 让模型基于连续自然语言改写模板段落，减少 run 边界导致的标点和词语拼接问题。
2. 保留原始富文本结构，包括段落、表格、列表、链接、占位符和重点格式。
3. 不把完整 HTML 交给模型生成，避免样式污染和安全风险。
4. 不做生成后「去重」式修补，错误应通过结构化输入和校验减少。

## 非目标

- 不重写整个富文本编辑器。
- 不要求模型返回 HTML。
- 不实现任意复杂的文本 diff 和格式自动迁移。
- 不保证模型一定生成最佳中文表达；这里只解决格式碎片干扰改写的问题。

## 核心设计

新增「锚点化 segment 改写」路径。每个段落或表格单元仍是一个 segment，但 prompt
不再把所有原始 run 作为主要改写单位，而是提供一段带锚点的连续文本。

### 锚点类型

后端在构造 prompt 时，将以下内容替换为不可变锚点：

- 占位符，例如 `{{name}}`、`{{sender_name}}`。
- 链接文本及链接地址。
- 有显著格式的文本，例如加粗、斜体、下划线。
- 表格、列表、段落等块级结构边界。
- 以后可扩展的不可编辑内联结构。

普通样式差异（例如字体、字号、普通 span）不作为语义锚点。它们只用于最终写回时
选择保留位置，不暴露给模型作为独立改写单位。

### Prompt 表示

每个 segment 发送给模型的数据改为：

```json
{
  "segment_id": "seg_3",
  "rewrite_text": "我是江西财经大学...[[A1]]。现在联系您或许有些晚了...，附件中是我的简历，供您参考。",
  "anchors": [
    {
      "anchor_id": "A1",
      "text": "以专业第一的成绩获得了推免资格",
      "source_runs": ["run_4", "run_5", "run_6"],
      "marks": ["strong"]
    }
  ]
}
```

模型只返回：

```json
{
  "segment_id": "seg_3",
  "text": "我是江西财经大学...[[A1]]。冒昧来信咨询，不知老师今年是否还有硕士招生名额？..."
}
```

约束：

- 锚点必须原样保留。
- 锚点顺序必须不变。
- 不能新增未知锚点。
- 不能删除、改写、移动占位符锚点。

## 写回策略

写回分为两层。

### 结构层

段落、表格、列表、链接和占位符结构保持原位置。模型只影响当前 segment 内的
可编辑文本内容。

### 文本层

对模型返回的锚点化文本进行解析：

1. 将普通文本片段写入当前 segment 的普通文本承载 run。
2. 遇到锚点时，恢复对应原始 run 及其格式。
3. 对被普通文本覆盖的多余普通 run 清空或移除。
4. 不跨 segment 写入。

第一版可采用保守策略：每个锚点之间的普通文本写入该区间第一个普通 run，其余普通
run 清空。这样实现简单，能避免标点被卡在碎 run 开头，同时保留锚点格式。

## 校验与失败处理

后端在写回前必须校验：

- 返回 JSON 结构合法。
- 每个返回项只引用存在的 `segment_id`。
- 每个 segment 的锚点集合与输入一致。
- 锚点顺序与输入一致。
- 占位符锚点未丢失。
- 返回文本非空，且包含可见正文。

校验失败时，不应静默生成有瑕疵的草稿。第一版处理方式：

- 直接抛出可见错误，说明是「模型未按模板锚点约束返回」。
- 在错误信息中带上失败的 `segment_id` 和原因。
- 不降级到旧富文本草稿路径。

后续可以增加一次自动重试，但重试不是第一版必须项。

## 与现有代码的关系

现有 `build_template_run_document()` 仍负责解析 HTML 和记录原始 run。

建议将锚点化逻辑合并进草稿改写相关服务模块，避免再拆出旧路径。

职责：

- 从 `TemplateRunDocument` 构造锚点化 segments。
- 验证模型返回。
- 将锚点化文本写回原始 `TemplateRunDocument`。

`llm_runtime.generate_draft_content()` 保持现有入口，只把模板改写 prompt 和结果解析切换到锚点化路径。

## 测试计划

至少补充以下单元测试：

1. 加粗文本被锚点化，并在写回后保留 `<strong>`。
2. `推免资格` 与后续句子不再因 run 边界丢失句号。
3. `未来规划，附件中...` 不再产生 `。，` 或 `， ，`。
4. `科研项目` 被拆分为多个原始 run 时，锚点化写回不产生重复词。
5. 占位符锚点丢失时抛出明确错误。
6. 锚点顺序被模型打乱时抛出明确错误。

## 风险与取舍

- 普通 span 级字体差异可能被合并到一个普通文本 run 中，局部字体细节可能减少。
- 加粗、链接、占位符等重要格式通过锚点保留。
- 方案复杂度高于单纯 prompt，但能从结构上消除格式碎片对自然语言改写的干扰。

## 推荐实施顺序

1. 新增锚点化构造和校验测试。
2. 实现锚点化 prompt 构造。
3. 实现锚点化写回。
4. 切换 `generate_draft_content()` 的模板改写路径。
5. 保留旧 run 改写代码一段时间，作为内部对照，不作为自动降级路径。
