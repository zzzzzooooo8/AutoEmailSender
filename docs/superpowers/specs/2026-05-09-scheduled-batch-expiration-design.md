# 定时批量任务过期处理规格

## 背景

Auto Email Sender 支持定时批量发送。当前实现只在后台调度时判断当前时间是否落在发送窗口内：如果不在窗口内，就跳过发送。

这个行为在简单场景下可用，但在 AI 生成模式中存在明显问题：用户可能长时间没有审核草稿，等到审核时，定时发送窗口已经全部过去。此时任务仍可能停留在 `running`，未发送子任务停留在 `review_required`、`approved` 或 `scheduled` 等状态，用户无法直接判断任务为什么不再发送。

本规格定义定时批量任务的过期语义，让系统在发送窗口全部错过后给出明确状态，并取消剩余未完成子任务。

## 目标

- 避免定时批量任务在所有发送窗口过期后静默停留在 `running`。
- 明确区分「发送窗口过期」与「用户主动停止」「发送失败」「正常完成」。
- 让任务中心可以展示未发送邮件的取消原因。
- 覆盖 AI 生成后长期未审核、暂停后恢复、创建时窗口已过期等场景。

## 非目标

- 不实现「重新安排发送时间」编辑器。
- 不自动复制过期任务为新任务。
- 不实现提醒、通知或定时弹窗。
- 不改变现有真实发送间隔、每日发送数量上限和 SMTP 发送逻辑。

## 术语

- **发送窗口：** 一个 `scheduled_date` 与 `window_start_time` / `window_end_time` 组成的本地时间区间。
- **未来发送窗口：** 当前本地时间早于某个发送窗口的 `window_end_time`。如果当前时间已经晚于开始时间但仍早于结束时间，也视为仍可发送。
- **最终状态子任务：** 已经不会再参与自动发送流转的邮件子任务。
- **未完成子任务：** 仍可能等待草稿生成、审核或发送的邮件子任务。

## 状态模型

### 批量任务状态

新增批量任务状态：

```text
expired
```

含义：系统判定该定时批量任务的所有发送窗口都已错过，剩余邮件不会再自动发送。

不复用其他状态：

- `completed` 表示任务正常完成，不适合还有邮件未发送的场景。
- `send_failed` 是邮件子任务发送失败语义，不适合作为批量任务过期语义。
- `stopped` 更接近用户主动中止，不适合系统自动过期。

### 邮件子任务取消原因

新增邮件子任务取消原因：

```text
schedule_expired
```

含义：该邮件因为所属定时批量任务发送窗口已全部过期，被系统取消。

## 过期判定

定时批量任务满足以下条件时，判定为过期：

- `schedule_type` 为 `scheduled`。
- 当前本地时间已经晚于所有 `scheduled_dates` 的 `window_end_time`。
- 任务状态仍允许调度处理，例如 `running`。
- 任务下存在未完成子任务。

最终状态子任务包括：

- `sent`
- `reply_detected`
- `send_failed`
- `canceled`

未完成子任务包括：

- `discovered`
- `matched`
- `generating_draft`
- `draft_failed`
- `review_required`
- `approved`
- `scheduled`

## 后台调度行为

后台调度扫描定时批量任务时按以下顺序处理：

1. 如果当前本地时间在发送窗口内，按现有逻辑选择并发送可发送子任务。
2. 如果当前本地时间不在窗口内，但仍存在未来发送窗口，继续等待，不改变任务状态。
3. 如果所有发送窗口均已过期：
   - 将批量任务状态更新为 `expired`。
   - 将该批量任务下所有未完成子任务更新为 `canceled`。
   - 写入 `cancellation_reason = "schedule_expired"`。
   - 不调用真实发送逻辑。
   - 记录操作日志，例如 `batch_task.expired`。

已经处于最终状态的子任务不得被改写。

## 审核与批准行为

当用户审核通过一封属于定时批量任务的邮件时，后端需要检查所属批量任务是否仍有未来发送窗口。

如果所属批量任务已经过期，或者根据当前本地时间判断已经没有未来发送窗口：

- 拒绝批准或拒绝排入待发送队列。
- 返回明确错误信息：

```text
当前批量任务的发送窗口已全部过期，请重新安排发送时间后再审核发送。
```

第一版采用「拒绝批准」策略，避免邮件进入 `approved` 或 `scheduled` 后仍无法自动发送。

## 创建行为

创建定时批量任务时增加基础防呆校验：

- 如果所有选择日期的 `window_end_time` 都早于当前本地时间，拒绝创建。
- 如果今天的 `window_start_time` 已过但 `window_end_time` 未过，允许创建。
- 如果今天窗口已过但未来还有其他选择日期，允许创建。

错误信息：

```text
当前定时发送窗口已全部过期，请重新选择发送日期或结束时间。
```

该校验只是前置防呆，不能替代运行时过期处理。

## 暂停与恢复行为

暂停状态代表用户主动控制，后台调度不主动将 `paused` 任务标记为 `expired`。

当用户恢复定时批量任务时，后端需要检查是否仍存在未来发送窗口：

- 如果仍存在未来发送窗口，恢复为 `running`。
- 如果所有窗口都已过期，直接转为 `expired`，并将剩余未完成子任务取消为 `schedule_expired`。

## 前端展示

任务中心需要识别并展示 `expired` 状态：

```text
已过期
```

邮件子任务取消原因 `schedule_expired` 展示为：

```text
发送窗口已过期
```

批量任务卡片可展示辅助文案：

```text
发送窗口已过期，剩余邮件已取消。可重新创建任务。
```

第一版不展示「重新安排发送时间」入口，避免出现不可用操作。

## 操作日志

系统自动过期批量任务时，记录一条操作日志：

- `category`: `email`
- `event_name`: `batch_task.expired`
- `entity_type`: `batch_task`
- `entity_id`: 批量任务 ID
- `metadata`:
  - `canceled_count`: 本次取消的子任务数量
  - `scheduled_dates`: 原定发送日期
  - `window_end_time`: 原定结束时间

## 测试范围

后端测试覆盖：

- 当前时间在发送窗口内时，任务正常发送。
- 当前日期未选中但未来还有发送日期时，不发送、不标记过期。
- 当前时间晚于最后一个发送窗口时，批量任务变为 `expired`，未完成子任务变为 `canceled`，取消原因是 `schedule_expired`。
- 已发送子任务保持 `sent`。
- 已回复子任务保持 `reply_detected`。
- 发送失败子任务保持 `send_failed`。
- 已取消子任务保持原状态，不覆盖取消原因。
- 过期后审核批准被拒绝。
- 创建时所有发送窗口已过期被拒绝。
- 暂停任务恢复时，如果窗口已过期，任务转为 `expired` 并取消剩余未完成子任务。

前端测试覆盖：

- 任务中心能展示 `expired` 为「已过期」。
- 子任务取消原因 `schedule_expired` 展示为「发送窗口已过期」。
- 创建定时任务时，前端能对明显已过期窗口给出即时提示。

## 实现建议

后端建议将时间窗口判断集中到 `app.services.batch_schedule`，避免创建接口、调度逻辑、审核逻辑各自实现一套日期比较规则。

建议新增两个纯函数：

- 判断当前时间是否仍有未来发送窗口。
- 判断当前时间是否已经晚于所有发送窗口。

批量任务过期和子任务取消建议封装为一个服务函数，供后台调度和恢复任务复用。

## 验收标准

- 过期定时批量任务不会长期停留在 `running`。
- 窗口过期后不会调用真实发送逻辑。
- 未发送邮件有明确取消原因。
- 用户在任务中心可以看出任务是「已过期」，而不是误以为发送失败或已完成。
- AI 生成模式下，即使用户长期未审核，任务也会在窗口全部错过后进入明确终态。
