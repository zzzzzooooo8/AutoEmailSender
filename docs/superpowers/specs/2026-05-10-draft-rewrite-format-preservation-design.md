# AI 改写草稿格式保真重做设计

## 背景

当前草稿改写链路已经支持模板级改写、富文本回填和局部样式保留，但在实际使用中仍有两个问题：

1. 模板内容会被拆成较细的 run / 锚点结构，LLM 容易把“该保留的局部样式”理解错。
2. 表格、字体字号、占位符的处理方式还没有统一成一套明确规则，导致改写结果和原稿视觉基线不稳定。

本次重做的目标是：以“段落级改写”为主，保留局部格式证据，表格原封不动，输出必须是真实内容，不再携带占位符。

## 目标

- 按段落或同类文本块进行改写，不按字符区间切片。
- 表格内容原样保留，不进入 LLM 改写流程。
- 原文中的局部格式信息要以“原文片段”的形式喂给 LLM，而不是笼统地说“整段加粗”。
- 占位符先替换成真实内容后再发给 LLM，模型输出也必须直接是真实内容。
- 全文统一使用原稿中可见文字占比最大的字体和字号作为基准样式。

## 非目标

- 不让 LLM 直接生成 HTML。
- 不支持让 LLM 重新设计表格结构。
- 不追求 Word 级别的完整排版能力。
- 不把字体字号交给模型自由决定。

## 输入归一化

系统在调用 LLM 前，把模板 HTML 归一化为一组 `source_blocks`：

- `paragraph`
- `heading`
- `list_item`
- `table`

其中：

- `paragraph`、`heading`、`list_item` 会进入改写流程。
- `table` 不进入改写流程，后端会原样回填原始 HTML fragment，不经过 LLM 改写。

每个可改写块包含：

- `segment_id`
- `type`
- `text`：占位符已替换后的真实文本
- `style_spans`：原文中具有格式的连续片段

`style_spans` 不是“整段是什么格式”，而是按原始 DOM 顺序提取的局部连续片段，例如：

- `加粗文本：xxxxx`
- `下划线文本：yyyyy`

如果同一段里有多处局部格式，它们要分别列出。

## 占位符规则

- 调用 LLM 前，所有可解析占位符都替换为真实值。
- 如果某个占位符没有对应真实值，保留原占位符文本，不猜测补全。
- 模型输出不得再出现占位符。
- 后端落库前只接受真实内容，不做二次占位符推断。

## LLM 提示词

### 系统提示词

系统提示词只负责约束行为，不承载具体业务内容。核心要求：

- 只输出 JSON。
- 只改写输入中的可改写块。
- 表格不改写。
- 只能返回输入中已有的 `segment_id`。
- 不要新增、删除、合并、拆分或重排段落。
- 不要输出占位符。
- 不要输出 HTML。
- 不要编造导师、学生或材料里不存在的信息。
- 字体和字号不由模型决定，由系统统一处理。

### 用户输入 payload

建议 payload 包含：

- `context`
  - `identity`
  - `professor`
  - `primary_material`
  - `rewrite_preferences`
- `source_blocks`
  - 每个块的 `segment_id`
  - `type`
  - `text`
  - `style_spans`
  - `table_html` 仅对表格块保留，用于原样透传
- `response_schema`

### 风格证据表达

提示词中不要写成“这段原本是加粗”。

应直接给原文片段，例如：

- `加粗文本：xxxxx`
- `下划线文本：yyyyy`
- `斜体文本：zzzzz`

模型可以基于这些证据决定改写后的内容哪些部分继续加粗或下划线。

## LLM 输出 schema

建议使用扁平 `runs` 结构，避免深层 `children` 树：

```json
{
  "subject": "邮件主题",
  "replacements": [
    {
      "segment_id": "seg_1",
      "runs": [
        { "text": "李老师，您好：" },
        { "text": "信息抽取", "marks": ["strong"] },
        { "text": "方向的研究", "marks": ["underline"] }
      ]
    }
  ]
}
```

约束：

- `segment_id` 必须来自输入。
- `text` 必须是真实最终内容。
- `marks` 只允许白名单样式。
- 第一版优先支持 `strong`、`underline`，可保守兼容 `emphasis`。
- 不要求模型输出表格相关内容。

为什么用扁平 `runs`：

- 比字符 offset 更不容易算错。
- 比深层嵌套 `children` 更容易稳定输出。
- 更接近当前后端的 run 处理方式。

## 回填规则

后端根据 `segment_id` 把 LLM 返回的 `runs` 渲染成 HTML。

规则如下：

- 段落级容器保留原结构。
- 段内按 `runs` 重新拼接文本和样式。
- 相邻同样式 `runs` 可以合并。
- 未改写块保持原样。
- 表格块按原 HTML 原样回填，不经过 LLM。

如果 LLM 返回无效 `segment_id`、空 `runs`、非法样式或空文本，视为改写失败。

## 字体字号统一规则

全文字体和字号由系统决定，不由 LLM 决定，且基准样式只从原稿中统计，不从改写结果重新计算。

计算方式：

- 统计原稿中所有可见文字的字符数。
- 按“字体 + 字号”的组合分组统计。
- 字符数占比最大的组合，作为全文基准字体和字号。
- 如果出现并列，优先选择文档中最早出现的组合。

应用范围：

- 适用于最终生成的非表格正文。
- 表格保持原样，不强行重写字体字号。

## 错误处理

- LLM 输出不是 JSON：失败。
- `segment_id` 不存在：失败。
- `runs` 为空或全是空文本：失败。
- 输出包含占位符：失败。
- 输出试图改写表格：失败。
- 输出样式超出白名单：忽略或失败，取决于实现层级；建议首版直接失败，便于暴露 prompt 问题。

失败时应保留原草稿，并把错误原因写入任务错误信息，避免静默损坏邮件正文。

## 测试

后端测试至少覆盖：

- 表格块不会进入 LLM prompt。
- `style_spans` 里传的是原文片段，而不是整段样式描述。
- 占位符会先替换为真实内容。
- 输出 `runs` 可正确渲染回 HTML。
- 未替换占位符时保留原文，不会被猜测补全。
- 字体字号基准会按可见文字字符数选出最大占比组合。
