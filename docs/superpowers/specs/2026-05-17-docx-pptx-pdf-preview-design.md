# 材料系统打开方式与只读副本设计

## 背景

材料库当前有“打开”和“下载”两个动作：

- “打开”预期用于快速查看材料。
- “下载”用于导出原始上传文件。

现有 `/api/materials/{material_id}/open` 会把原文件以 `Content-Disposition: inline` 返回给浏览器。这个策略只对浏览器内置支持的格式比较有效，例如 PDF；对于 `.docx`、`.pptx`、`.xlsx` 等 Office 文件，Electron/Chromium 通常无法直接预览，结果会退化成保存或下载行为。

此前方案考虑将 `.docx` 和 `.pptx` 按需转换为 PDF 后再预览。但该方案依赖 LibreOffice 或其他转换工具，会引入安装包体积、转换耗时、转换失败、格式失真、目标机器依赖缺失等问题。

本设计改为：不管上传的材料是什么类型，点击“打开”时优先使用本地系统能力，让用户通过系统应用或系统“打开方式”查看文件；实际打开的不是原始上传文件，而是一个只读副本。

## 已确认的产品决策

- “打开”不再追求应用内统一预览。
- “打开”改为调用本地系统能力查看材料。
- 所有材料类型尽量走同一套打开逻辑，而不是只针对 `.docx/.pptx` 做特殊转换。
- 打开时使用原文件的只读副本，不直接打开材料库中的原始文件。
- 用户可以通过系统默认应用或系统“打开方式”选择合适的软件查看。
- “下载”动作继续下载原始上传文件，不下载临时副本。
- 不引入 LibreOffice、在线预览服务或应用内 Office 渲染器。

## 目标与非目标

### 目标

- 让任意类型材料点击“打开”后尽量交给本地系统应用处理。
- 避免浏览器对不支持格式退化成保存窗口。
- 避免外部应用直接修改材料库中的原始上传文件。
- 保持“下载”接口和按钮下载原始上传文件。
- 对临时只读副本做统一管理和清理。
- 当系统无法打开文件时，给出明确错误提示。

### 非目标

- 不实现 DOCX/PPTX 转 PDF。
- 不实现 Office、PDF、图片、文本等格式的应用内统一渲染器。
- 不接入在线 Office 预览服务。
- 不把 LibreOffice 或其他转换工具打包进安装包。
- 不改变材料上传、删除、设为默认材料、草稿生成等业务逻辑。
- 不保证所有目标机器都安装了能打开对应材料格式的软件。

## 当前数据流

现有打开链路：

1. 前端材料库按钮调用 `openFileInNewTab(getMaterialOpenUrl(material.id))`。
2. `getMaterialOpenUrl` 生成 `/api/materials/{material_id}/open`。
3. 后端读取 `IdentityMaterial.file_path`。
4. 后端 `FileResponse(file_path, media_type=material.mime_type)` 返回原文件。
5. 后端设置 `Content-Disposition: inline`。
6. Electron/Chromium 决定是否能预览。

问题发生在第 6 步：浏览器只能预览部分格式；对于 Office 等格式，会出现保存/下载行为或空白窗口，用户体验不稳定。

## 新数据流

新的打开链路：

1. 前端材料库按钮触发桌面端打开动作。
2. 前端把 `material.id` 发送给 Electron preload 暴露的安全 IPC 方法。
3. Electron 主进程调用后端或本地服务获取材料元信息和原始文件路径。
4. 主进程创建一个临时只读副本。
5. 主进程调用系统能力打开该副本。
6. 用户在系统默认应用或系统“打开方式”中查看材料。

建议链路：

```text
Frontend
  -> preload API
  -> Electron main IPC
  -> backend/material metadata
  -> create read-only copy
  -> Windows open/open-with
  -> external application
```

## 方案比较

### 方案 A：后端按需转换为 PDF 后返回

`.docx/.pptx` 打开时转换为 PDF，再交给 Electron/Chromium 预览。

优点：

- 可以保留应用内 PDF 预览体验。
- 前端入口变化较少。
- 对 DOCX/PPTX 能形成统一预览形态。

缺点：

- 依赖 LibreOffice 或其他转换工具。
- 首次打开需要等待转换。
- 转换可能失败或格式失真。
- 安装包和运行环境复杂度增加。
- 只能优先覆盖 DOCX/PPTX，其他格式仍需额外方案。

### 方案 B：上传时立即转换 PDF

上传 Office 文件时同步或异步生成 PDF 预览，打开时直接返回 PDF。

优点：

- 打开速度快。
- 打开链路简单。

缺点：

- 上传流程变慢或变复杂。
- 上传失败和转换失败语义纠缠。
- 用户可能从不打开该材料，提前转换浪费资源。
- 同样依赖转换工具。

### 方案 C：系统打开方式 + 只读副本

所有材料打开时都创建只读副本，并交给本地系统应用处理。

优点：

- 不依赖 LibreOffice。
- 不需要维护格式转换链路。
- 支持范围天然更广，不局限于 DOCX/PPTX。
- 文件保真度取决于用户本机专业软件，通常优于转换预览。
- 只读副本可以降低误修改原始材料的风险。
- 产品语义清晰：“打开”是查看，“下载”是导出原始文件。

缺点：

- 不再是应用内预览。
- 依赖用户本机安装可打开对应格式的软件。
- 不同系统和不同文件关联设置下体验会有差异。
- Windows 上“每次弹出打开方式选择器”需要额外系统调用，不是普通 `shell.openPath` 默认行为。

## 推荐方案

采用方案 C：系统打开方式 + 只读副本。

原因：

- 它直接解决用户无法可靠打开 Office 等材料的问题。
- 它不引入沉重的转换依赖。
- 它对所有材料格式更统一，不需要逐个格式扩展预览能力。
- 它避免外部程序直接修改材料库原始文件。
- 它更符合桌面应用能力边界：桌面壳负责调用本地系统能力，后端负责材料存储和权限校验。

## 桌面端打开服务设计

建议在 Electron 主进程中新增材料打开服务，例如：

```text
desktop/src/main/materialOpenService.ts
```

职责：

- 接收材料 ID。
- 查询材料元信息和源文件位置。
- 校验源文件存在且仍在允许的数据目录内。
- 生成临时副本路径。
- 复制源文件到临时目录。
- 设置临时副本为只读。
- 调用系统默认应用或系统“打开方式”。
- 返回打开成功或失败结果。

### 临时副本目录

建议路径：

```text
data/temp/open-materials/<material_id>/<timestamp>-<safe_filename>
```

其中：

- `material_id` 用于隔离不同材料。
- `timestamp` 避免同一材料多次打开时文件被占用导致覆盖失败。
- `safe_filename` 使用材料原文件名或下载文件名，并做路径字符清理。
- 文件扩展名必须保留，方便系统识别应用关联。

### 只读策略

创建副本后设置只读属性：

- Windows：设置文件只读属性。
- 后续如果支持 macOS/Linux，可使用文件权限控制写入位。

注意：

- 只读不是安全沙箱，只是防止普通外部应用直接覆盖该副本。
- 某些应用可能仍会创建锁文件、临时文件，或提示用户“另存为”。
- 即使用户修改了副本，也不应影响材料库原始文件。

### 打开方式策略

提供两种策略：

1. 默认打开：使用系统默认应用打开文件。
2. 选择打开方式：弹出系统“打开方式”选择器，让用户选择应用。

Windows 下可选实现：

- 默认打开：Electron `shell.openPath(copyPath)`。
- 选择打开方式：调用 `rundll32.exe shell32.dll,OpenAs_RunDLL <copyPath>`。

产品建议：

- 如果按钮文案是“打开”，优先使用默认应用打开。
- 如果明确希望用户选择应用，按钮文案可改为“打开方式”或在“打开”旁增加下拉选项。
- 也可以做成：主按钮“打开”，更多菜单中提供“选择打开方式”。

## 后端接口设计

后端不再需要把 `/api/materials/{material_id}/open` 作为主要预览入口。

建议保留或调整为兼容接口：

### `GET /api/materials/{material_id}/download`

保持不变：

- 返回原始上传文件。
- 文件名继续使用现有下载命名逻辑。
- 不返回临时副本。

### 材料元信息接口

Electron 主进程需要安全获取材料信息。可复用现有材料列表数据；如果当前数据不包含足够信息，可新增内部接口或后端服务方法。

需要的信息：

- `material_id`
- 原始文件路径或可由后端解析出的安全文件路径
- 原始文件名
- MIME 类型
- 文件大小

安全要求：

- 不允许前端直接传入任意本地路径让主进程打开。
- 主进程只能根据后端已登记的材料 ID 查找文件。
- 文件路径必须来自数据库或后端受控存储，而不是用户输入。

### `GET /api/materials/{material_id}/open`

可以有两种处理方式：

1. 保留现有行为作为 Web 兼容入口。
2. 返回明确提示，说明桌面端应使用系统打开能力。

为了减少破坏性，建议短期保留现有行为；桌面端材料库按钮改走新的 IPC 打开链路。

## 前端行为

材料库按钮逻辑需要调整：

- “打开”不再调用 `openFileInNewTab(getMaterialOpenUrl(material.id))`。
- “打开”改为调用 preload 暴露的桌面 API，例如 `window.desktopApi.openMaterial(material.id)`。
- 打开成功时可不显示额外提示。
- 打开失败时在当前页面展示 toast 或错误提示。
- “下载”继续调用 `getMaterialDownloadUrl(material.id)`。

建议错误提示：

- 找不到原文件：`材料文件不存在，可能已被移动或删除。`
- 系统无法打开：`系统没有找到可打开该文件的应用，请下载后手动选择应用。`
- 创建副本失败：`无法创建只读副本，请稍后重试。`
- 权限失败：`没有权限打开该材料。`

## IPC 设计

preload 暴露最小能力：

```ts
window.desktopApi.openMaterial(materialId: string): Promise<OpenMaterialResult>
window.desktopApi.openMaterialWithDialog?(materialId: string): Promise<OpenMaterialResult>
```

返回结构建议：

```ts
type OpenMaterialResult =
  | { ok: true }
  | { ok: false; code: string; message: string }
```

主进程 IPC：

```text
materials:open
materials:open-with
```

安全要求：

- 校验 `materialId` 类型和格式。
- 不接受前端传入的本地文件路径。
- 不把真实原始文件路径返回给前端。
- 主进程内部完成复制、设只读、打开。

## 错误处理

建议错误类型：

- `MaterialNotFound`：材料记录不存在。
- `MaterialFileMissing`：数据库记录存在，但源文件不存在。
- `MaterialFileOutOfBounds`：源文件路径不在允许的数据目录内。
- `MaterialOpenCopyFailed`：创建临时副本失败。
- `MaterialOpenReadonlyFailed`：设置只读属性失败。
- `MaterialOpenSystemFailed`：系统打开失败或没有关联应用。

处理原则：

- 不暴露本地绝对路径给前端错误消息。
- 日志中可以记录详细路径，便于排查。
- 前端展示用户可理解的中文提示。
- 打开失败不自动改为下载，避免混淆“打开”和“下载”的产品语义。

## 安全与文件边界

- 前端只能传 `materialId`，不能传本地路径。
- 主进程必须根据受控数据源解析材料文件路径。
- 源文件路径必须校验在应用数据目录或允许的材料存储目录内。
- 临时副本目录固定在应用数据目录下。
- 临时副本文件名必须清理非法路径字符。
- 打开的永远是副本，不是材料库原文件。
- 不向渲染进程暴露真实原始文件路径。

## 临时文件清理

建议清理策略：

- 应用启动时清理超过一定时间的旧打开副本。
- 每次打开材料前顺手清理过期副本。
- 默认保留时间可设为 24 小时。
- 清理失败不影响打开流程，只记录日志。

需要注意：

- 外部应用可能仍占用文件，删除失败是正常情况。
- 删除失败时跳过即可，下次再尝试。
- 不应强制关闭用户正在查看的外部应用。

## 测试方案

### 桌面端单元测试

新增或扩展 `desktop/test`：

- 根据材料 ID 创建临时副本。
- 临时副本保留原扩展名。
- 临时副本设置为只读。
- 源文件不存在时返回明确错误。
- 非法材料 ID 不会触发任意路径打开。
- 源文件路径越界时拒绝打开。
- 系统打开失败时返回 `MaterialOpenSystemFailed`。
- 清理逻辑会跳过仍被占用或删除失败的文件。

### preload / IPC 测试

- `openMaterial(materialId)` 正确调用 `materials:open`。
- 不暴露任意路径打开 API。
- 打开失败时返回结构化错误。

### 前端测试

扩展材料库相关测试：

- “打开”调用桌面 API，而不是 `getMaterialOpenUrl`。
- “下载”仍使用 `getMaterialDownloadUrl(material.id)`。
- 打开失败时显示错误提示。
- 防止后续误把“打开”和“下载”接到同一个端点。

### 后端测试

如果新增材料元信息内部接口或服务方法，需要覆盖：

- 材料不存在。
- 材料存在但文件缺失。
- 返回的文件信息不包含不必要的敏感字段。
- 权限校验与现有材料访问逻辑一致。

### 手动验证

在 Windows 环境中：

1. 上传 `.pdf` 材料，点击“打开”，确认系统应用打开只读副本。
2. 上传 `.docx` 材料，点击“打开”，确认 Word/WPS 或系统打开方式可查看。
3. 上传 `.pptx` 材料，点击“打开”，确认 PowerPoint/WPS 或系统打开方式可查看。
4. 上传 `.xlsx`、图片、文本等材料，确认能交给系统默认应用处理。
5. 修改外部应用中的副本后，确认材料库原文件不受影响。
6. 点击“下载”，确认下载的是原始上传文件。
7. 删除或移动源文件后点击“打开”，确认出现明确错误提示。
8. 检查过期临时副本能被清理。

## 发布与依赖风险

该方案不新增 LibreOffice 等重量级依赖，因此发布风险较低。

主要风险来自系统差异：

- 用户机器可能没有安装可打开某格式的软件。
- 文件关联可能被错误配置。
- “默认打开”和“选择打开方式”的系统行为不同。
- 杀毒软件或权限策略可能影响临时副本创建或外部应用打开。

应对策略：

- 打开失败时给出清晰提示。
- 保留“下载”作为兜底路径。
- 后续可增加“打开所在临时目录”或“重新选择打开方式”等辅助能力，但不作为本次必要范围。

## 验收标准

- 任意材料点击“打开”时不再通过浏览器 `window.open` 预览入口。
- “打开”会创建只读临时副本。
- 外部应用打开的是副本，不是材料库原始文件。
- 副本保留原文件扩展名。
- “下载”仍返回原始上传文件。
- 系统无法打开文件时，前端显示明确中文错误。
- 前端不能传入任意本地路径触发打开。
- 临时副本有过期清理机制。
- 桌面端、preload/IPC、前端相关测试通过。
