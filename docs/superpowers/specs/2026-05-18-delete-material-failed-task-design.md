# 失败任务引用材料时允许删除材料 - 设计规格

日期：2026-05-18

## 背景

用户反馈：界面上没有未完成任务时，删除材料仍失败，提示“当前材料仍被未完成任务作为默认材料使用”。排查发现，后端删除材料逻辑只把 `sent`、`reply_detected`、`canceled` 视为不会阻塞材料删除的终态。`draft_failed` 和 `send_failed` 虽然在用户心智上属于失败/不可自动继续状态，但仍被当前删除校验视为未完成任务。

## 目标

当材料仅被失败任务引用时，允许删除该材料。材料删除后，对应失败任务不再支持基于该材料的原地重试，后续如果要继续联系，应重新选择或上传材料并重新生成内容。

## 非目标

- 不改变正在进行、待审核、待发送、已排程、发送中的任务对材料删除的阻止行为。
- 不新增失败任务的重试流程。
- 不修改前端任务列表、导师看板或任务详情的展示文案。
- 不迁移历史数据；行为在删除材料时即时生效。

## 业务规则

材料删除时，邮件任务状态按以下规则处理：

- 不阻止删除：`sent`、`reply_detected`、`canceled`、`draft_failed`、`send_failed`。
- 继续阻止删除：`discovered`、`matched`、`generating_draft`、`review_required`、`approved`、`scheduled`、`sending`。

如果失败任务引用了被删除材料：

- 该任务的 `primary_material_id` 应清空，避免留下悬挂引用。
- 如果该材料出现在 `selected_material_ids` 中，也应从列表中移除。
- 任务状态仍保持原失败状态，不自动变为 `canceled`。
- 任务已有错误信息、生成内容、日志不主动清理。

## 数据流

1. 用户调用 `DELETE /api/materials/{material_id}`。
2. 后端查询同身份下仍会阻止材料删除的邮件任务。
3. 如果阻止任务仍引用该材料，返回当前 400 错误。
4. 如果只有失败/终态任务引用该材料，则清理这些任务中的材料引用。
5. 如果材料是身份当前默认材料，清空 `identity.current_primary_material_id`。
6. 记录材料删除日志，删除文件和材料记录，提交事务。

## 错误处理

- 进行中任务引用默认材料：继续返回“当前材料仍被未完成任务作为默认材料使用”。
- 进行中任务引用随信材料：继续返回“当前材料仍被未完成任务选为随信材料”。
- 失败任务引用材料：不返回错误，删除成功。
- 如果文件删除失败，沿用现有 `delete_file` 行为，不在本次范围内改变。

## 测试计划

新增后端接口测试覆盖：

1. `draft_failed` 任务引用材料作为默认材料时，删除材料返回 204，并清空该任务的 `primary_material_id`。
2. `send_failed` 任务引用材料作为默认材料时，删除材料返回 204，并清空该任务的 `primary_material_id`。
3. 失败任务的 `selected_material_ids` 包含该材料时，删除材料返回 204，并从列表中移除该材料。
4. 现有进行中状态任务引用材料时，删除仍返回 400，确保保护逻辑不被放宽。

## 实现边界

建议集中修改：

- `backend/app/services/materials.py`：调整“不会阻止材料删除”的状态集合，纳入失败状态。
- `backend/app/api/materials.py`：删除前清理失败/终态任务中的材料引用，避免悬挂外键或残留 JSON 引用。
- `backend/test/test_api_endpoints.py`：新增接口级回归测试。

不建议修改数据库迁移、前端 API 类型或桌面端代码。

## 验收标准

- 用户在只有失败任务引用材料时可以成功删除材料。
- 删除后失败任务不会继续引用已删除材料。
- 正在进行或待处理任务仍会阻止材料删除。
- 后端相关测试通过。
