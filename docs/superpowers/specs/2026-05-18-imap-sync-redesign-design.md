# IMAP 回复同步重构设计

## 背景

当前 IMAP 回复检测使用固定窗口轮询：后台每 60 秒遍历所有已配置 IMAP 的发件身份，对每个身份搜索近 72 小时收件箱，并用 `RFC822` 拉取整封邮件。这个模型存在三个问题：

1. 流量不可控：每轮重复下载近 72 小时内的所有命中邮件，附件也会随整封邮件一起拉取。
2. 长期离线会漏同步：用户超过 72 小时未打开软件时，离线期间较早收到的回复不会被发现。
3. 并发可能叠加：后台轮询和手动刷新都可能对同一个邮箱发起 IMAP 拉取。

本次重构目标是把 IMAP 检测从“固定窗口重复扫描”改为“有状态、低流量、可恢复的回复同步”。

## 目标

- 日常后台同步只处理新增邮件，不重复扫描最近 72 小时历史邮件。
- 历史扫描只针对已经进入沟通链路的导师，不扫描整个邮箱。
- 老版本用户升级后自动补齐历史回复，并把相关任务推进到已回复。
- 默认不下载附件二进制内容，显著降低 IMAP 流量。
- 同一个发件身份的 IMAP 同步具备并发保护，避免后台与手动刷新叠加请求。
- 已有通信记录不被覆盖；仅在旧字段为空时补齐缺失字段。

## 非目标

- 不在本次实现附件下载功能。
- 不实现用户可浏览邮箱全部邮件的通用邮箱客户端能力。
- 不默认扫描未进入沟通链路的全部导师。
- 不在数据库迁移过程中执行任何 IMAP 网络请求。

## 核心概念

## 邮箱服务商兼容要求

本功能必须兼容 163 邮箱和 QQ 邮箱。这两个服务商都是目标用户的高频邮箱提供方，设计不能依赖少数海外邮箱才稳定支持的高级 IMAP 行为。

### 163 邮箱

根据网易邮箱官方帮助中心：

- 163/126/yeah.net 免费邮箱支持第三方客户端协议，需要先开启客户端授权码。
- 网易官方帮助列出“网易邮箱服务器参数如何设置？”和“客户端设置教程”，并提示创建第三方客户端前应确保客户端授权码已开启。
- 网易官方帮助明确提示：如果使用第三方客户端导致流量超限，开通会员无法提升第三方客户端流量上限。
- 网易官方帮助说明 IMAP 协议支持海外使用。

对本设计的约束：

- 不能高频重复下载历史邮件或附件，否则会直接触发网易第三方客户端流量限制。
- 登录失败文案需要提示用户检查客户端授权码，而不是只提示邮箱密码错误。
- 默认配置应支持 `imap.163.com:993`、`imap.126.com:993`、`imap.yeah.net:993` 的 SSL IMAP 连接。
- 实现必须能在标准 IMAP 能力下工作，不能要求服务商支持 Gmail 专有扩展。

### QQ 邮箱

根据 QQ 邮箱官方帮助中心，QQ 邮箱提供“POP/IMAP 和 Exchange 服务的设置方法”文档，并在官方帮助中列出邮件服务频率限制相关错误，例如 `Ip frequency limited`、`Connection frequency limited` 等。

对本设计的约束：

- 默认配置应支持 `imap.qq.com:993` 的 SSL IMAP 连接。
- 登录失败文案需要提示用户确认 QQ 邮箱已开启 IMAP/SMTP 服务并使用授权码。
- 同一身份必须有连接级节流与并发保护，避免触发 QQ 邮箱连接频率限制。
- 手动刷新不得绕过身份级锁，也不得触发整个邮箱全量扫描。

### 协议能力基线

为保证 163 和 QQ 邮箱兼容，第一版实现只依赖标准 IMAP 能力：

- `LOGIN`
- `SELECT INBOX`
- `UID SEARCH`
- `UID FETCH` 头部字段
- `BODYSTRUCTURE`
- 指定 MIME part 的 `BODY.PEEK[...]`

第一版不依赖以下能力：

- IMAP `IDLE`
- Gmail `X-GM-*` 扩展
- 服务商专有搜索扩展
- 服务端全文搜索能力

如果某个服务商对 `BODYSTRUCTURE` 或 part 拉取表现异常，需要降级为只拉取头部并记录“正文待补齐”，但不能退回默认整封 `RFC822` 拉取附件。

### 已进入沟通链路的导师

历史扫描仅覆盖满足任一条件的导师：

- 存在 `email_tasks` 任务记录。
- 存在与该导师关联的 `email_logs` 通信记录。
- 用户打开过工作区并由系统创建过工作区任务。

导师必须有可用邮箱地址。没有邮箱地址的导师不进入 IMAP 扫描队列。

### 身份级邮箱同步状态

每个发件身份和邮箱文件夹维护一条同步状态，用于日常增量监听。

建议新增表 `imap_mailbox_sync_states`：

- `id`
- `identity_id`
- `folder`，默认 `INBOX`
- `uidvalidity`
- `last_seen_uid`
- `last_sync_at`
- `last_error`
- `created_at`
- `updated_at`

唯一约束：`identity_id + folder`。

### 导师级历史扫描状态

每个发件身份和导师维护一条历史扫描状态，用于升级补扫、新导师补扫和手动修复。

建议新增表 `imap_professor_sync_states`：

- `id`
- `identity_id`
- `professor_id`
- `professor_email`
- `folder`，默认 `INBOX`
- `historical_scan_status`：`pending`、`running`、`completed`、`failed`
- `last_scanned_uid`
- `historical_scan_started_at`
- `historical_scan_completed_at`
- `last_error`
- `created_at`
- `updated_at`

唯一约束：`identity_id + professor_id + professor_email + folder`。

### 邮件去重键

优先使用邮件 `Message-ID` 去重。若邮件没有 `Message-ID`，使用稳定指纹兜底：

- `identity_id`
- `professor_id`
- 规范化后的 `from_email`
- 规范化后的 `subject`
- 邮件真实时间，优先 `INTERNALDATE`，其次 `Date`
- 正文摘要哈希

指纹只作为兜底，避免没有 `Message-ID` 的邮件重复入库。

## IMAP 拉取策略

### 不再使用整封邮件拉取作为默认路径

当前 `FETCH (RFC822 INTERNALDATE)` 会下载整封邮件，包括附件。本次重构后默认不使用它处理日常同步和历史补扫。

新策略分阶段拉取：

1. 搜索候选 UID。
2. 拉取轻量头部，例如 `Message-ID`、`From`、`To`、`Cc`、`Subject`、`Date`、`In-Reply-To`、`References`。
3. 拉取 `BODYSTRUCTURE`，识别文本正文 part 和附件 part。
4. 只拉取 `text/plain` 和 `text/html` 正文 part。
5. 跳过 `attachment` 和大体积非文本 part。

第一版可以不保存附件元信息；如果后续需要，可以只保存“存在附件”和文件名，不保存附件内容。

### 历史扫描

历史扫描按导师执行，不扫描整个邮箱。

搜索条件：

- 主条件：`FROM "professor@example.edu"`
- 若服务商搜索能力受限，可退化为按邮箱服务商支持的等价搜索。

扫描范围：

- 全部历史，但仅限该导师发来的邮件。
- 每次处理一个导师或少量导师，后台渐进执行。
- 扫描进度通过 `last_scanned_uid` 持久化，应用关闭后可恢复。

触发时机：

- 老版本升级后，后台根据既有任务和通信记录生成待扫描导师状态。
- 新导师进入沟通链路时，为该导师生成待扫描状态。
- 用户在工作区手动刷新当前导师时，可优先处理当前导师的历史扫描。

### 日常增量监听

日常监听按发件身份执行，使用身份级 `last_seen_uid` 拉取新增邮件。

流程：

1. 打开 `INBOX` 并读取 `UIDVALIDITY`。
2. 若 `UIDVALIDITY` 与已记录值一致，从 `last_seen_uid + 1` 开始搜索新增 UID。
3. 若 `UIDVALIDITY` 变化，当前 UID 游标失效，清空身份级游标并依赖导师级历史扫描状态补扫相关导师。
4. 对新增 UID 先取头部，判断是否与系统导师或已发送邮件相关。
5. 仅对候选邮件取文本正文并写入通信记录。
6. 成功处理后推进 `last_seen_uid` 和 `last_sync_at`。

候选判断包括：

- `From` 命中已进入沟通链路的导师邮箱。
- `In-Reply-To` 或 `References` 命中本地已发送邮件的 `Message-ID`。
- 主题与已发送主题存在回复关系，作为弱兜底，不单独作为唯一依据。

## 数据合并规则

### 已有通信记录

如果扫描到的邮件与已有 `email_logs` 命中同一个 `Message-ID` 或兜底指纹：

- 不覆盖已有 `subject`、`content`、`content_html`、`created_at` 等非空字段。
- 仅当旧字段为空且新扫描有值时补齐。
- 可以补齐缺失的 `reply_headers`、`received_at`、`content_html` 等字段。

### 新回复

如果扫描到数据库中不存在的新回复：

- 新增一条 `direction = received` 的 `email_logs`。
- 邮件时间使用真实邮件时间，优先 `INTERNALDATE`，其次 `Date`，最后才使用当前时间。
- 将对应任务推进到 `reply_detected`。

### 任务状态推进

发现真实回复后，对应任务一律推进到 `reply_detected`。

这包括：

- 已发送任务。
- 已取消任务。
- 回收站任务。
- 其他仍能明确关联到导师、身份和发信记录的任务。

扫描不会因为未发现回复而把任务状态回退。

## 老版本升级兼容

### 迁移原则

- 迁移只创建必要表和字段，不执行 IMAP 网络操作。
- 老版本已有 `email_logs` 和 `email_tasks` 保留为事实来源。
- 迁移后首次后台启动时生成历史扫描状态。
- 历史扫描可中断、可恢复、可重试。

### 升级后的后台启动流程

1. 找出所有完整配置 IMAP 的发件身份。
2. 为每个身份创建 `imap_mailbox_sync_states`，但不直接跳过历史补扫。
3. 从 `email_tasks` 和 `email_logs` 找出已进入沟通链路、且有邮箱地址的导师。
4. 为 `identity_id + professor_id + professor_email` 创建 `pending` 历史扫描状态。
5. 后台 worker 渐进执行这些历史扫描。
6. 扫描到旧版本漏掉的回复后新增通信记录，并把任务推进到 `reply_detected`。

### 与旧数据不一致时的处理

新扫描结果不覆盖旧数据。只有旧字段为空时才用新扫描结果补齐。

示例：

- 旧记录已有正文，新扫描解析出不同正文：保留旧正文。
- 旧记录正文为空，新扫描有正文：补齐正文。
- 旧记录缺少 HTML，新扫描有 HTML：补齐 HTML。
- 旧记录缺少真实接收时间，新扫描有 `INTERNALDATE`：补齐接收时间。

## 并发与节流

### 身份级并发保护

同一个 `identity_id` 同一时间只允许一个 IMAP 同步任务运行。

后台 poller、历史扫描和手动刷新都必须经过同一个身份级锁或数据库级 claim 机制。

### 渐进扫描

历史扫描不在一次 worker tick 内处理所有导师。

建议第一版限制：

- 每轮最多处理 1 个或少量 `pending` 导师。
- 单个导师扫描按 UID 批次推进。
- 失败后记录 `last_error`，稍后重试，不阻塞其他导师。

### 手动刷新

工作区“刷新回复”只优先处理当前导师。

如果当前身份正在后台同步：

- 可以等待已有同步短时间完成后返回。
- 或直接返回当前本地通信记录，并提示正在后台同步。

不得因为手动刷新而触发整个邮箱全量扫描。

## 错误处理

- IMAP 登录失败：记录到同步状态 `last_error`，不清空历史扫描状态。
- `UIDVALIDITY` 变化：身份级 UID 游标失效，重新建立游标，并保留导师级历史扫描作为补偿。
- 单封邮件解析失败：记录错误并跳过该邮件，继续处理后续邮件。
- 服务商不支持某些 IMAP 扩展：退化到标准 `SEARCH`、`FETCH BODYSTRUCTURE` 和正文 part 拉取。
- 搜索语法失败：当前导师状态标记为 `failed` 并记录错误，避免无限重试打爆 IMAP。

## 测试计划

### 单元测试

- 轻量 MIME 拉取只请求文本 part，不请求附件 part。
- 同 `Message-ID` 已存在时不覆盖非空字段。
- 旧字段为空时补齐新扫描结果。
- 扫描到回复后，已取消和回收站任务也推进到 `reply_detected`。
- 没有 `Message-ID` 的邮件使用兜底指纹去重。

### 集成测试

- 老版本数据升级后生成导师级历史扫描状态。
- 历史扫描只针对有任务或通信记录的导师，不扫描无关导师。
- 日常增量同步从 `last_seen_uid + 1` 开始，不重复处理旧 UID。
- 后台轮询和手动刷新同一身份时不会并发执行两次 IMAP 拉取。

### 回归测试

- 已有工作区通信记录展示不变。
- 已发送邮件、回复检测、任务状态推进仍正常。
- IMAP 配置测试仍只验证连接，不创建同步状态或执行历史扫描。

## 验收标准

- 默认后台同步不再使用近 72 小时固定窗口重复拉取。
- 默认 IMAP 拉取不下载附件二进制内容。
- 用户离线超过 72 小时后再次打开软件，仍能通过游标和导师级历史扫描补齐期间回复。
- 老版本已有记录不被覆盖；空字段可被补齐。
- 发现回复后，对应任务统一进入 `reply_detected`，包括取消和回收站任务。
- 同一个发件身份不会因为后台轮询和手动刷新产生并发 IMAP 拉取。
