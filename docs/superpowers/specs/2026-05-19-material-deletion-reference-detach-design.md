# 材料删除引用断开策略设计

## 背景

用户在材料库删除已上传材料时，可能遇到「当前材料仍被未完成任务作为默认材料使用」的错误。实际排查发现，材料并不一定被用户可见的进行中批量任务占用，也可能被工作区或匹配分析自动创建的 `EmailTask` 引用。

现有逻辑使用 `TERMINAL_MATERIAL_REFERENCING_STATUSES` 判断哪些任务可以在删除材料时自动清理引用。这个命名和语义存在偏差：其中包含 `draft_failed`、`send_failed` 等失败状态，但不包含 `discovered`、`matched`。因此，尚未生成草稿的任务也会阻止删除材料，用户难以理解，也缺少显式处理入口。

## 目标

- 让用户可以顺畅删除不再需要的材料，避免被隐藏或草稿前任务阻塞。
- 保证已生成、已审核或待发送内容不会静默失去材料上下文。
- 明确区分「任务终态」和「删除材料时可断开引用的状态」。
- 为后续实现提供清晰的状态处理规则、提示文案和测试口径。

## 非目标

- 不新增「放弃草稿」或「取消邮件任务」入口作为本次必要方案。
- 不改变邮件任务整体状态机的发送、回信、跟进语义。
- 不处理材料文件本身的存储、下载、预览逻辑。
- 不引入硬删除任务或批量清理历史任务功能。

## 术语

- **默认材料：** `EmailTask.primary_material_id` 指向的材料，用于匹配分析或草稿生成。
- **随信材料：** `EmailTask.selected_material_ids` 中的附件材料，发送邮件时作为附件使用。
- **断开引用：** 删除材料时清空 `primary_material_id`，或从 `selected_material_ids` 中移除该材料 ID。
- **废弃草稿：** 清空基于旧材料生成或审核的草稿字段，并将任务回退到需要重新生成或重新审核的状态。
- **可断开状态：** 删除材料时可以自动处理引用的任务状态，不等同于业务终态。

## 当前问题

### `discovered` / `matched` 阻止删除

`discovered` 和 `matched` 表示任务还没有进入草稿审核阶段。它们可能由工作区打开导师、匹配分析任务或批量任务初始化创建。这些任务通常只是预备态，材料引用多数来自当时的默认材料快照。让它们阻止材料删除，会导致用户认为「没有进行中的任务」，但仍无法删除材料。

### `review_required` 没有显式退出口

`review_required` 表示草稿已生成，等待用户审核。它既可能来自手动生成草稿，也可能来自批量任务后台生成草稿，还可能来自取消定时发送后的回退。当前没有「不想发了」的状态转移入口。如果只要求用户手动放弃草稿，用户可能找不到入口，也会增加操作负担。

### `send_failed` 的重试语义不可靠

`send_failed` 表示邮件没有成功发出，但可能保留已审核正文和随信材料。删除材料后，如果保留 `send_failed` 状态和已审核内容，后续重试发送可能使用与当前材料状态不一致的内容或附件。

## 设计原则

1. **删除材料优先清理预备态引用。** 没有进入审核或发送链路的任务不应阻止用户清理材料。
2. **不静默保留基于旧材料的草稿。** 已生成草稿或已审核内容引用被删材料时，必须让内容失效。
3. **发送链路保持保守。** 已批准、已定时或正在发送的任务继续阻止删除，避免用户无意改变待发送内容。
4. **历史事实可以清引用。** 已发送、已回信、已取消等状态可以断开数据库引用，但不改变邮件日志中的事实记录。
5. **可软删除即应可断开。** 任何允许进入回收站的任务状态，都不应继续阻塞材料删除；否则用户会遇到「任务已删除但材料仍删不掉」的矛盾体验。
6. **命名表达真实语义。** 使用「可断开引用」命名，避免把失败态或可继续态误称为终态。

## 状态处理规则

建议用新集合替代 `TERMINAL_MATERIAL_REFERENCING_STATUSES`，例如 `MATERIAL_REFERENCE_DETACHABLE_STATUSES`。该集合表示删除材料时允许自动断开引用的邮件任务状态。

| 状态 | 删除默认材料时 | 删除随信材料时 | 说明 |
| --- | --- | --- | --- |
| `discovered` | 断开引用，保留状态 | 移除附件，保留状态 | 尚未生成草稿，可安全清理。 |
| `matched` | 断开引用，保留状态 | 移除附件，保留状态 | 已有匹配结果但未生成草稿，可安全清理。 |
| `generating_draft` | 阻止删除 | 阻止删除 | 草稿生成中，材料可能正在被读取。 |
| `draft_failed` | 断开引用，保留或回退状态 | 移除附件，保留状态 | 没有可用草稿，可清理失败上下文。 |
| `review_required` | 断开引用并废弃草稿 | 移除附件并废弃已审核内容 | 草稿可能基于材料生成，必须避免旧草稿继续可发送。 |
| `approved` | 阻止删除 | 阻止删除 | 已审核待发送，不能静默改变内容。 |
| `scheduled` | 阻止删除 | 阻止删除 | 已定时发送，不能静默改变内容。 |
| `sending` | 阻止删除 | 阻止删除 | 正在发送，不能改变材料引用。 |
| `sent` | 断开引用，保留状态 | 移除附件引用，保留状态 | 已形成发送事实，邮件日志保留事实记录。 |
| `send_failed` | 断开引用并废弃已审核发送内容 | 移除附件并废弃已审核发送内容 | 邮件未成功发出，删除材料后不应继续重试旧内容。 |
| `reply_detected` | 断开引用，保留状态 | 移除附件引用，保留状态 | 已形成沟通事实，清引用不影响历史。 |
| `canceled` | 断开引用，保留状态 | 移除附件引用，保留状态 | 已取消任务可清引用。 |

## 草稿废弃规则

当删除的材料影响 `review_required` 或 `send_failed` 任务时，系统应自动废弃相关草稿或已审核发送内容。

材料删除必须保持原子性：只要存在 `generating_draft`、`approved`、`scheduled`、`sending` 等必须阻止删除的引用任务，整个删除操作失败，不做任何部分清理。只有所有引用都可断开时，才执行引用清理、草稿废弃和材料删除。

### 默认材料被删除

如果任务的 `primary_material_id` 等于被删除材料 ID：

- 清空 `primary_material_id`。
- 清空 `generated_subject`、`generated_content_text`、`generated_content_html`。
- 清空 `approved_subject`、`approved_body_text`、`approved_body_html`、`approved_at`。
- 清空 `scheduled_at`。
- 清空 `last_error`，避免保留与旧材料相关的失败原因。
- 更新 `updated_at`。
- 状态回退到 `matched` 或 `discovered`：
  - 如果任务保留匹配结果（如 `match_score`、`match_reason`、`fit_points`、`risk_points`、`match_keywords` 任一存在），回退到 `matched`。
  - 否则回退到 `discovered`。

该规则同样适用于 `send_failed`：如果失败发送任务的默认材料被删除，不能保留可重试旧邮件的状态，必须废弃已审核发送内容，并回退到 `matched` 或 `discovered`。

### 随信材料被删除

如果被删除材料只出现在 `selected_material_ids`：

- 从 `selected_material_ids` 中移除该材料 ID。
- 对 `review_required` 和 `send_failed`，清空 `approved_subject`、`approved_body_text`、`approved_body_html`、`approved_at`、`scheduled_at`。
- 本设计建议保留 `generated_*`，因为正文不一定依赖附件；但任务必须重新确认附件变化，不能继续使用旧审核内容直接发送。
- 状态统一落到 `review_required`，让用户重新确认附件变化；如果原状态是 `send_failed`，也应回到 `review_required`，避免继续展示为可重试失败邮件。

### 默认材料和随信材料同时命中

如果同一任务既把材料作为默认材料，又把材料作为随信材料：

- 按「默认材料被删除」处理。
- 同时从 `selected_material_ids` 中移除该材料 ID。
- 状态回退到 `matched` 或 `discovered`。

### `draft_failed` 处理

`draft_failed` 没有可用草稿，但 `last_error` 可能与旧材料相关。处理规则应明确为：

- 如果默认材料命中，清空 `primary_material_id` 和 `last_error`，并回退到 `matched` 或 `discovered`。
- 如果只命中随信材料，从 `selected_material_ids` 移除该材料 ID；`last_error` 可保留，除非错误信息明确来自该材料。
- 不保留指向已删除材料的任何引用。

## 批量任务与软删除

批量任务有 `deleted_at` 软删除字段，但邮件子任务没有。材料删除逻辑应优先根据实际 `EmailTask` 引用处理，不能只依赖任务列表可见性判断。

对 `BatchTask` 本身的材料引用检查建议同步调整：

- `running`、`paused` 等可继续批量任务继续阻止删除材料。
- `stopped`、`completed`、`expired` 可断开引用。
- 批量任务当前只有 `stopped`、`completed`、`expired` 允许软删除，这些状态应与材料删除的可断开状态保持一致。
- 如果未来新增可软删除状态，必须同步加入材料引用可断开规则，避免软删除后的任务继续阻塞材料删除。
- 不建议单纯用 `deleted_at IS NOT NULL` 跳过引用检查；更好的约束是「能被软删除的状态本身就是可断开的」。这样恢复任务后也不会携带已删除材料的悬空引用。

同样的原则适用于抓取任务、匹配分析任务等会间接创建或关联邮件任务的流程：只要某个上层任务允许软删除，它名下仍可影响材料删除的邮件任务引用也应在材料删除时可断开，或在软删除流程中提前转为可断开状态。

## 用户提示

删除材料成功后，如果系统自动清理了任务引用，应在成功提示中补充摘要。例如：

- 「材料已删除，并同步清理 3 个待写信任务的材料引用。」
- 「材料已删除，并重置 1 个引用该材料的待审核草稿。」
- 「材料已删除，并移除 2 个失败发送任务中的旧附件引用。」

如果删除被阻止，应说明具体原因和可操作方向。例如：

- 「该材料仍被已批准或定时发送的任务使用，请先取消定时或调整任务后再删除。」
- 「该材料正在被草稿生成或发送中的任务使用，请稍后再试。」

## 后端实现建议

1. 将 `TERMINAL_MATERIAL_REFERENCING_STATUSES` 重命名为 `MATERIAL_REFERENCE_DETACHABLE_STATUSES`。
2. 在材料删除流程中拆分处理：可直接断开引用的状态、需要废弃草稿的状态、必须阻止删除的状态。
3. 为 `review_required` 和 `send_failed` 增加废弃草稿逻辑。
4. 记录操作日志，包含被断开引用的任务 ID、被重置草稿的任务 ID、被移除随信材料的任务 ID。
5. 删除材料接口可继续返回 `204`；如前端需要展示精确摘要，后续可改为返回删除结果 DTO。

### 事务与文件删除顺序

材料删除涉及数据库引用、材料记录和物理文件。实现时应避免出现「文件已删除但数据库回滚」或「数据库已删除但文件仍残留且无法追踪」的问题。

推荐顺序：

1. 在同一数据库事务内重新加载材料及引用任务。
2. 判断是否存在必须阻止删除的引用任务；如果存在，直接失败并回滚。
3. 清理 `primary_material_id` 和 `selected_material_ids`，废弃必要草稿。
4. 写入操作日志 metadata。
5. 删除材料数据库记录并提交事务。
6. 数据库提交成功后删除物理文件；如果物理文件删除失败，记录日志或错误信息，后续通过清理任务补偿。

如果继续采用提交前删除物理文件的实现，需要明确失败补偿策略，避免数据库回滚后材料记录指向不存在的文件。

### 外键与数据完整性

删除 `IdentityMaterial` 前，必须先清空所有引用它的外键和 JSON 数组引用：

- `IdentityProfile.current_primary_material_id`。
- `EmailTask.primary_material_id`。
- `EmailTask.selected_material_ids`。
- `BatchTask.primary_material_id`。
- `BatchTask.selected_material_ids`。

所有清理应在删除材料记录前完成，避免数据库外键约束或 ORM 关系导致删除失败。

### 操作日志 metadata

材料删除日志建议包含结构化 metadata，便于排查用户反馈：

```json
{
  "was_primary": true,
  "detached_primary_task_ids": [101, 102],
  "removed_attachment_task_ids": [103],
  "reset_draft_task_ids": [104],
  "detached_batch_task_ids": [201],
  "blocked_statuses": []
}
```

如果删除被阻止，也建议记录或返回阻止原因摘要，例如阻止状态、任务 ID 和引用类型。

### 并发保护

材料删除和任务状态流转可能并发发生。例如，任务在删除检查后从 `review_required` 变为 `approved`，或从 `approved` 进入 `sending`。

实现要求：

- 在同一事务内按当前状态重新判断引用任务。
- 对即将清理引用的任务使用数据库行锁或等价机制，避免检查和更新之间状态被并发修改。
- 如果提交前发现任务进入必须阻止删除的状态，应回滚整个删除操作。
- 删除操作不能对同一批引用任务做部分成功处理。

## 前端实现建议

第一阶段可以保持现有交互，只调整删除成功和失败提示：

- 删除按钮仍保留二次确认。
- 删除成功后刷新身份和材料列表。
- 如果后端仍返回 `204`，前端展示通用成功提示。
- 如果后续后端返回结果 DTO，前端展示自动清理摘要。

暂不新增「放弃草稿」按钮，避免让用户必须先定位具体任务再回到材料库删除。

如果用户正在工作区打开被重置的任务，删除材料后刷新工作区时应让用户理解状态变化。推荐提示：

- 「草稿已因关联材料删除而失效，请重新选择材料并生成草稿。」
- 「随信附件已移除，请重新确认草稿后再发送。」

第一阶段可以通过刷新后的任务状态和通用通知体现；后续如果返回删除结果 DTO，可以在工作区显示更精确的提示。

## 测试场景

后端测试应覆盖以下场景：

- `discovered` 任务引用默认材料时，删除材料成功，并清空 `primary_material_id`。
- `matched` 任务引用默认材料时，删除材料成功，并清空 `primary_material_id`。
- `review_required` 任务引用默认材料时，删除材料成功，清空草稿字段，并回退到 `matched` 或 `discovered`。
- `review_required` 任务只引用随信材料时，删除材料成功，移除 `selected_material_ids` 中的材料，并清空已审核字段。
- `send_failed` 任务引用默认材料或随信材料时，删除材料成功，并废弃已审核发送内容。
- `approved`、`scheduled`、`sending` 引用材料时，删除材料失败，并返回可理解的错误信息。
- `sent`、`reply_detected`、`canceled` 引用材料时，删除材料成功，并清理引用。
- 可继续批量任务引用材料时，删除材料仍被阻止。
- 不可继续批量任务引用材料时，删除材料成功，并清理引用。
- 已软删除任务处于允许软删除状态时，不应继续阻塞材料删除。
- 新增任何可软删除状态时，测试必须覆盖该状态下的材料引用断开行为。
- 同一材料同时作为默认材料和随信材料时，删除材料成功后两类引用都被清理，且状态落点符合默认材料命中的规则。
- 多个任务混合引用同一材料时，只要存在 `generating_draft`、`approved`、`scheduled` 或 `sending`，整个删除失败，不清理任何可断开任务的引用。
- 删除过程中如果任务状态并发变为必须阻止删除的状态，删除失败并回滚所有引用清理。

## 验收标准

- 用户删除只被 `discovered` 或 `matched` 任务引用的材料时，不再收到「当前材料仍被未完成任务作为默认材料使用」。
- 用户删除被 `review_required` 或 `send_failed` 引用的材料时，材料删除成功，相关任务不再保留可直接发送的旧审核内容。
- 用户删除被 `approved`、`scheduled` 或 `sending` 引用的材料时，系统继续阻止删除，并提示用户先处理发送链路。
- 删除材料后，任务表中不再存在指向已删除材料的 `primary_material_id` 或 `selected_material_ids`。
- 已进入回收站的任务不会继续造成材料删除阻塞；恢复任务后也不会指向已删除材料。
- 如果删除被阻止，数据库中的任务引用和材料记录保持原样，不出现部分清理。
- 操作日志能追踪材料删除及引用清理行为。

## 已确定口径与后续选项

- 删除随信材料但保留 `generated_*`：本设计确定保留正文草稿，但清空已审核内容，并让任务回到 `review_required` 重新确认。
- `draft_failed` 的 `last_error`：默认材料命中时清空；仅随信材料命中时可保留，除非错误信息明确来自该材料。
- 删除结果返回值：第一阶段可继续返回 `204`；如需要更精确的前端提示，后续再改为返回 DTO。
