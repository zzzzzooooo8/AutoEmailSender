# 发送日期占位符规格

## 背景

当前套磁信模板支持导师信息和发件人信息占位符，例如 `{{name}}`、`{{university}}`、`{{sender_name}}`。用户在写信时希望能在落款或正文中加入发送日期，但日期格式存在个人偏好，例如中文格式、斜杠格式或横线格式。

因此需要新增年、月、日三个独立占位符，让用户自行组合日期格式，同时确保日期值来自实际发送当天，而不是任务创建、模板编辑或草稿生成当天。

## 目标

- 在主题和正文占位符菜单中新增 `{{year}}`、`{{month}}`、`{{day}}`。
- 三个占位符只渲染数字，不自动附加“年”“月”“日”等单位。
- `month` 和 `day` 不补零，例如 5 月 8 日渲染为 `5` 和 `8`。
- 日期必须按北京时间 `Asia/Shanghai` 计算。
- 日期必须取实际发送当天，不能在创建任务、生成草稿或编辑模板时提前固化。
- 直接套用模板、AI 模板改写、工作区手动发送和测试发送等发送路径都应保持一致的日期语义。

## 非目标

- 本次不新增完整日期占位符，例如 `{{date}}`。
- 本次不提供日期格式选择器。
- 本次不支持补零格式，例如 `05`、`08`。
- 本次不支持用户自定义时区。
- 本次不调整现有导师信息、发件人信息占位符的含义。

## 占位符规格

新增占位符如下：

| 占位符 | 菜单文案 | 示例值 | 说明 |
| --- | --- | --- | --- |
| `{{year}}` | 发送年份 | `2026` | 北京时间实际发送当天的年份 |
| `{{month}}` | 发送月份 | `5` | 北京时间实际发送当天的月份，不补零 |
| `{{day}}` | 发送日期 | `20` | 北京时间实际发送当天的日期，不补零 |

用户可自行组合格式：

```text
{{year}}年{{month}}月{{day}}日
{{year}}/{{month}}/{{day}}
{{year}}-{{month}}-{{day}}
```

## 时间语义

### 实际发送当天

年、月、日必须在邮件实际发送前的最终渲染阶段计算。以下时间点不能作为最终日期来源：

- 创建任务时间。
- 保存身份模板时间。
- 导入模板文件时间。
- 生成 AI 草稿时间。
- 进入工作区预览时间。
- 审核或保存草稿时间。

如果任务在 2026-05-20 创建，但定时到 2026-05-22 发送，则最终邮件中的：

```text
{{year}}年{{month}}月{{day}}日
```

应渲染为：

```text
2026年5月22日
```

### 北京时间

日期必须按 `Asia/Shanghai` 计算。即使服务内部使用 UTC 存储或调度，在渲染占位符时也必须将当前发送时刻转换到北京时间后再取年月日。

示例：

- UTC 时间为 `2026-05-19 16:30:00Z`。
- 北京时间为 `2026-05-20 00:30:00+08:00`。
- 占位符应渲染为 `2026`、`5`、`20`。

## 前端行为规格

### 占位符菜单

- 主题输入框和正文富文本编辑器使用同一份占位符定义。
- 新增选项应同时出现在：
  - 身份默认模板编辑。
  - 创建任务模板编辑。
  - 工作区正文编辑。
  - 测试发信编辑。
- 插入后的视觉效果延续现有占位符 chip 样式。
- 保存到表单状态和提交给后端时，应继续保存为原始 token，例如 `{{year}}`，而不是菜单文案。

### 文案提示

涉及“支持占位符”的提示文案应加入发送日期示例，避免用户不知道新字段可用。

推荐示例：

```text
支持 {{name}}、{{university}}、{{sender_name}}、{{year}}、{{month}}、{{day}} 等占位符。
```

## 后端行为规格

### 模板上下文

后端最终发送模板上下文应新增：

- `year`
- `month`
- `day`

这些值由北京时间的当前发送时刻派生，全部为字符串数字：

```python
{
    "year": "2026",
    "month": "5",
    "day": "20",
}
```

### 渲染阶段

发送日期占位符必须在最终发送前渲染。实现时需要避免在早期生成阶段把日期提前替换进草稿。

早期模板渲染阶段可以将 `{{year}}`、`{{month}}`、`{{day}}` 原样保留，避免现有未知字段替换逻辑把它们清空。最终发送阶段必须使用真实发送时刻的北京时间数字覆盖这些 token。

直接套用模板路径中，如果创建任务时需要生成预览或 `approved_*` 初始内容，应避免将 `{{year}}`、`{{month}}`、`{{day}}` 固化为创建当天。最终发送阶段仍必须用实际发送当天重新渲染。

AI 模板改写路径中，供模型理解模板的上下文可以展示预览日期，但最终可发送内容不能依赖早期日期。若模型改写后保留日期 token，最终发送阶段应按实际发送当天替换。

### 测试发信

测试发信也应按触发测试发送时的北京时间计算年、月、日。测试场景没有导师实体时，导师字段继续使用现有测试值，日期字段使用真实发送时刻派生值。

## 数据兼容性

- 已保存模板不需要迁移。
- 历史任务不需要补写日期字段。
- 旧模板中的未知占位符保持现有处理逻辑。
- 新占位符作为纯文本 token 保存在现有 subject/body 字段中，不需要数据库 schema 变更。

## 测试规格

### 前端测试

- 占位符菜单中展示 `发送年份`、`发送月份`、`发送日期`。
- 在主题输入框插入 `发送年份` 后，提交值为 `{{year}}`。
- 在正文编辑器插入 `发送月份` 和 `发送日期` 后，序列化结果包含 `{{month}}` 和 `{{day}}`。
- 已保存的 `{{year}}年{{month}}月{{day}}日` 能重新渲染为 chip 加普通文本混排。

### 后端测试

- 模板渲染能将 `{{year}}年{{month}}月{{day}}日` 替换为北京时间日期，且月份和日期不补零。
- 使用固定 UTC 时间 `2026-05-19 16:30:00Z` 时，渲染结果应为 `2026年5月20日`。
- 定时任务最终发送阶段使用发送时刻日期，而不是任务创建时刻日期。
- 测试发信路径能渲染年、月、日占位符。
- 未知占位符继续沿用现有行为，不因新增日期字段改变。

## 验收标准

- 用户可以在写信占位符菜单中选择年、月、日。
- 插入占位符后，编辑器显示为现有样式的 chip。
- 保存后的模板内容包含 `{{year}}`、`{{month}}`、`{{day}}` token。
- 最终邮件发送时，三个 token 按北京时间实际发送当天渲染为不补零数字。
- 跨天定时发送时，邮件日期与北京时间发送日一致。
- 现有占位符、模板导入、AI 改写、直接套用模板和手动发送能力不被破坏。
