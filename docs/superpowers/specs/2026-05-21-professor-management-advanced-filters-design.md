# 导师管理页高级筛选设计

## 背景

导师管理页当前筛选能力偏简单：常驻关键词搜索，加上“职称 / 导师资格”和“学校 / 学院”两个单选下拉。这个设计适合导师数量较少时快速定位，但当导师库逐渐增大后，用户需要更细粒度地组合筛选，例如同时按多个学校、多个学院、系所和职称筛出一批导师，再进行选择、编辑、归档或创建后续任务。

首页已经具备“基础筛选 + 高级筛选”的交互模式。导师管理页需要借鉴首页高级筛选的能力，但不能把首页完整筛选条原样搬过来。首页服务于“筛选导师并创建任务”，导师管理页服务于“维护导师库”。两者共享部分筛选维度，但页面职责不同。

## 目标

- 将导师管理页筛选升级为“常驻基础筛选 + 可展开高级筛选”。
- 支持按学校、学院、系所、职称 / 导师资格进行多选筛选。
- 保留导师管理页现有的归档视图切换：`正常 / 已删除 / 全部`。
- 保持页面顶部轻量，避免把所有筛选条件默认铺开。
- 筛选、排序、分页和“选择全部筛选结果”语义保持一致。

## 非目标

- 不把首页筛选条完整搬到导师管理页。
- 不在本期加入首页任务流语义字段，例如导师关系状态、最低匹配度、发送次数。
- 不要求改后端接口或新增服务端筛选参数。
- 不改变导师导入、新增、编辑、归档、恢复和批量操作的业务逻辑。
- 不把归档视图切换纳入普通重置逻辑。

## 页面职责边界

导师管理页筛选应服务于数据维护：

- 找到需要补全、编辑、清理或归档的导师。
- 在大导师库中按学校、学院、系所、职称组合缩小范围。
- 为后续选择导师、创建任务或维护数据提供更稳定的列表操作基础。

首页筛选继续服务于任务流：

- 按匹配度、联系状态、发送次数等任务维度筛选导师。
- 发起匹配分析、创建邮件任务、打开工作区。

因此，导师管理页本期只迁移首页高级筛选的交互模式和通用字段，不迁移首页独有的任务流字段。

## 字段语义

- 关键词：匹配姓名、邮箱、学校、学院、系所、职称 / 导师资格、研究方向。
- 学校：对应 `ProfessorManagementItemDTO.university`。
- 学院：对应 `ProfessorManagementItemDTO.school`。
- 系所：对应 `ProfessorManagementItemDTO.department`。
- 职称 / 导师资格：对应 `ProfessorManagementItemDTO.title`，继续使用现有职称拆分规则。
- 归档视图：继续对应管理页现有 `archiveFilter`，取值为 `active`、`archived`、`all`。

`school` 和 `department` 在界面中必须分别显示为“学院”和“系所”，避免继续使用“院系”这类含混文案。

## 交互设计

### 常驻基础区

常驻区保留低频打扰、快速定位的控件：

- 搜索框。
- 排序下拉。
- “高级筛选”按钮。
- “重置”按钮。
- “刷新”按钮。

建议布局：

```text
[ 搜索姓名、邮箱、学校、学院、系所、职称或研究方向                  ]

[ 排序：最新导入 v ]  [ 高级筛选 2 ]  [ 重置 ]  [ 刷新 ]
```

搜索框可以继续独占一行，避免导师管理页表格区变得拥挤。第二行承载排序和操作按钮。

### 高级筛选入口

“高级筛选”按钮默认收起筛选面板，点击后展开，再次点击收起。

按钮沿用首页的筛选数量提示：

- 没有高级筛选条件时显示 `高级筛选`。
- 有高级筛选条件时显示 `高级筛选 N`。
- `N` 只统计高级筛选条件，不统计关键词、排序或归档视图。

计数规则建议按已选择的选项数量计算：

- 选择 2 个学校和 1 个职称，显示 `高级筛选 3`。
- 如果后续更希望按字段数计算，需要另起规格明确调整；本期优先与首页保持一致。

### 高级筛选面板

展开后展示：

```text
高级筛选
[ 学校 多选 ] [ 学院 多选 ] [ 系所 多选 ] [ 职称 / 导师资格 多选 ]
```

本期不新增单独的“清空高级筛选”按钮。页面只保留一个“重置”按钮，避免用户在“清空高级筛选”和“重置筛选”之间反复判断。

“重置”按钮职责：

- 清空关键词。
- 清空高级筛选。
- 恢复默认排序。
- 不切换 `正常 / 已删除 / 全部` 归档视图。

归档视图是导师管理页的主视图状态，不属于普通筛选条件。用户在“已删除”视图中点击重置时，应仍停留在“已删除”。

### 归档视图切换

`正常 / 已删除 / 全部` 保持在现有独立位置，不放进高级筛选面板。

原因：

- 它表达的是管理页主视图，不是普通字段过滤。
- 它影响可选择、可恢复、可归档等操作语义。
- 把它放入高级筛选会弱化“回收站 / 正常列表”的管理含义。

## 筛选规则

所有筛选继续在前端本地完成，作用于当前管理页已加载的导师列表。

关键词筛选：

- 使用大小写不敏感的模糊匹配。
- 匹配字段包括姓名、邮箱、学校、学院、系所、职称 / 导师资格、研究方向。
- 关键词为空时不过滤。

多选筛选：

- 同一筛选组内是“或”关系。
- 不同筛选组之间是“且”关系。
- 未选择某组时，该组不过滤。
- 空值不生成选项。

职称 / 导师资格筛选：

- 继续使用现有 `extractProfessorTitleTags` 和 `matchesProfessorTitleTag` 语义。
- 一个导师标题中包含多个标签时，命中任一已选标签即可通过职称组筛选。

学院与学校联动：

- 学院选项应随当前已选学校收窄。
- 当学校筛选变化后，已经选中的学院如果不再属于可选范围，应自动移除。
- 如果未选择学校，则学院选项来自全量导师列表。

分页与选择：

- 筛选条件、排序条件变化后，当前页回到第 1 页。
- 分页只影响展示，不影响“选择全部筛选结果”的范围。
- “选择全部筛选结果”仍作用于当前归档视图和当前筛选条件下的所有可选择导师。

## 排序设计

导师管理页可以新增排序下拉，但排序选项应贴合数据维护场景。

建议选项：

- 最新导入。
- 最近更新。
- 姓名 A-Z。
- 学校 A-Z。

不建议本期加入：

- 匹配度高到低。
- 发送次数高到低。

这些字段属于首页任务流语义，当前导师管理数据结构也不直接提供对应字段。

## 选项生成

高级筛选选项从 `ProfessorManagementItemDTO[]` 派生：

- 学校选项来自非空 `university`。
- 学院选项来自非空 `school`。
- 系所选项来自非空 `department`。
- 职称选项来自 `extractProfessorTitleTags(title)` 拆出的非空标签。

选项排序：

- 学校、学院、系所按 `localeCompare('zh-CN')` 排序。
- 职称优先沿用现有职称拆分后的排序语义；如实现成本更低，也可以先按中文排序，但需要保持测试覆盖。

## 状态模型建议

建议把导师管理页筛选状态收敛成一个对象，避免继续散落多个 `useState`：

```ts
type ProfessorManagementFilterState = {
  keyword: string;
  universities: string[];
  schools: string[];
  departments: string[];
  titles: string[];
};
```

排序状态单独维护：

```ts
type ProfessorManagementSortKey =
  | "latest"
  | "updatedAtDesc"
  | "nameAsc"
  | "universityAsc";
```

归档视图继续单独维护：

```ts
type ArchiveFilter = "active" | "archived" | "all";
```

这样可以清楚地区分三类状态：

- 普通筛选：关键词和高级筛选。
- 展示排序：排序下拉。
- 管理视图：归档状态。

## 实现边界

建议新增导师管理页筛选 helper，而不是把逻辑继续堆在 `ProfessorsPage.tsx` 中：

- `frontend/src/features/professor-management/client/filterManagementProfessors.ts`
- `frontend/src/features/professor-management/client/filterManagementProfessors.test.ts`
- 可选：`frontend/src/features/professor-management/client/sortManagementProfessors.ts`
- 可选：`frontend/src/features/professor-management/client/sortManagementProfessors.test.ts`

可复用现有组件：

- `frontend/src/components/molecules/MultiSelectFilter.tsx`
- `frontend/src/components/atoms/NativeSelectField.tsx`

预计主要改动：

- `frontend/src/pages/ProfessorsPage.tsx`

`ProfessorsPage.tsx` 应尽量只做页面编排：

- 维护筛选、排序、归档、分页和选中状态。
- 调用筛选 helper 得到 `filteredProfessors`。
- 调用排序 helper 得到排序后的结果。
- 基于排序结果分页展示。

## 测试与验证

自动化测试建议覆盖：

- 关键词可以匹配姓名、邮箱、学校、学院、系所、职称和研究方向。
- 学校、学院、系所、职称多选筛选正确。
- 同组“或”、不同组“且”的组合规则正确。
- 学校变化时，学院选项和已选学院能正确收窄。
- 重置清空关键词、高级筛选和排序，但不改变归档视图。
- 筛选或排序变化后页码回到第 1 页。
- “选择全部筛选结果”仍选中所有过滤后的可选导师，而不是当前页导师。

建议验证命令：

```bash
cd frontend
npm run test -- src/features/professor-management/client/filterManagementProfessors.test.ts
npm run test -- src/pages/SelectionControls.test.tsx
npm run lint
npm run build
```

人工验证：

- 默认进入导师管理页时，高级筛选收起，列表结果不变。
- 点击“高级筛选”后可以多选学校、学院、系所和职称。
- 高级筛选收起后，按钮能显示当前高级筛选数量。
- 点击“重置”后关键词、高级筛选和排序恢复默认，但仍停留在当前归档视图。
- 在“已删除”视图中筛选、重置、恢复导师时，页面语义不混乱。
- 多页筛选结果下，“选择全部筛选结果”仍跨页生效。

## 风险与约束

当前导师管理页仍是前端本地筛选和分页。当导师数量增长到数千以上时，本地筛选、排序和全量加载可能继续带来性能压力。那时应单独设计后端分页和服务端筛选。

高级筛选按钮显示数量会让用户知道隐藏条件正在生效，但也会让按钮文案略长。为了与首页保持一致，本期保留数量提示；如果后续视觉上觉得拥挤，可以改为仅使用选中态表达。

本期不引入任务流字段，因此用户无法在导师管理页按匹配度或联系状态筛选。这是刻意保留的页面职责边界，而不是能力遗漏。

## 验收标准

- 导师管理页不再常驻展示“职称 / 导师资格”和“学校 / 学院”两个单选下拉。
- 导师管理页提供可展开的高级筛选面板。
- 高级筛选支持学校、学院、系所、职称 / 导师资格多选。
- 页面只保留一个“重置”按钮负责清空普通筛选和排序。
- 归档视图切换不受重置影响。
- 现有导入、新增、编辑、归档、恢复、批量选择和分页行为不被破坏。
