# 统计面板总览页设计

## 背景

当前系统已经有首页导师看板、导师管理页、任务中心和个人中心。首页偏向导师筛选、匹配分析和创建任务；导师管理页偏向导入、编辑、归档和维护导师档案；任务中心偏向批量任务、爬虫任务和匹配分析任务管理；个人中心已经包含 Token 消耗记录中心。

用户需要一个独立的统计面板页，用于从更高层级观察系统运行情况。第一版统计面板不再重复建设 Token 模块，而是聚焦两个业务模块：

- 导师模块：观察导师池规模、资料质量、匹配覆盖率和高价值待推进导师。
- 邮件模块：观察邮件触达进度、回复效果、失败风险和待处理事项。

## 目标

- 在顶部导航中新增“统计面板”入口。
- 新增独立统计面板页面，用卡片化布局展示导师模块和邮件模块。
- 每个模块先展示模块标题，再展示一组指标卡、图表卡和行动列表卡。
- 指标卡视觉参考当前讨论中的统计面板卡片样式：白色卡片、浅色图标块、主数字、辅助说明。
- 导师模块展示导师总数、已匹配导师、高匹配导师、高分未联系、匹配分布、资料完整度、学校分布和待行动列表。
- 邮件模块展示已发送、已回复、回复率、发送失败、待审核、已排程、趋势图、状态漏斗、状态分布和待跟进列表。
- 统计逻辑集中在后端聚合接口中，前端只负责展示和跳转，避免在页面内拼装多个业务接口。

## 非目标

- 不在统计面板中新增 Token 消耗模块。Token 消耗仍保留在个人中心的 Token 消耗记录中心。
- 不接入第三方平台余额、真实账单、RPM、TPM 或模型费用计算。
- 不替代首页、导师管理页或任务中心的操作能力。统计面板只提供概览和关键跳转。
- 不新增复杂自定义报表能力，例如拖拽排序、任意字段分组、导出统计面板图片等。
- 不要求第一版支持跨身份、跨模型的复杂对比分析。

## 导航设计

顶部导航新增“统计面板”，建议顺序为：

```text
首页 ｜ 统计面板 ｜ 导师管理 ｜ 任务中心 ｜ 个人中心
```

路由建议使用：

```text
/dashboard
```

新增页面文件建议为：

```text
frontend/src/pages/DashboardPage.tsx
```

`frontend/src/App.tsx` 中新增懒加载路由，`frontend/src/components/organisms/TopNavBar.tsx` 中新增导航项。

## 页面整体布局

页面采用工作台式卡片布局，避免大屏展示风格。整体结构为：

```text
统计面板
触达与匹配数据基于当前身份和模型，导师档案数据基于当前导师库。

导师概览
[导师总数] [已匹配导师] [高匹配导师] [高分未联系]
[匹配分数分布] [资料完整度概览]
[学校分布 Top 10] [高分未联系导师]
[待补全导师]

邮件触达
[已发送] [已回复] [回复率] [发送失败] [待审核] [已排程]
[近 30 天发送/回复/失败趋势] [邮件状态漏斗]
[邮件任务状态分布] [高价值待跟进邮件]
```

响应式布局：

- 大屏：指标卡每行 4 到 6 个，图表卡两列。
- 中屏：指标卡每行 2 到 3 个，图表卡单列或两列自适应。
- 小屏：所有卡片单列堆叠，列表保持可读。

## 视觉设计

### 指标卡

指标卡使用统一结构：

- 左侧浅色图标块。
- 右侧小标题。
- 主数字使用更大字号。
- 辅助说明使用小号弱化文字，可显示比例、阈值或状态说明。

建议图标均来自 `lucide-react`：

- 导师类：`Users`、`BadgeCheck`、`Star`、`UserRoundCheck`、`ClipboardCheck`、`School`
- 邮件类：`Send`、`Reply`、`Percent`、`CircleAlert`、`FileCheck2`、`CalendarClock`
- 图表类：`BarChart3`、`LineChart`、`Funnel`、`Rows3`
- 行动列表类：`ListChecks`、`BellRing`、`ClipboardList`

### 图表卡

图表卡统一包含：

- 卡片标题。
- 右上角可选的简短说明或筛选标签。
- 主图表区域。
- 空状态说明。

第一版可以用 CSS 网格、横向条形图、简单 SVG 或轻量自绘组件实现。若后续需要更丰富交互，再评估是否引入图表库。

### 列表卡

行动列表卡应避免只显示统计数字，必须提供可执行上下文：

- 导师列表可跳转工作区或导师管理页。
- 邮件列表可跳转对应工作区或任务详情。
- 空状态要说明当前没有待处理事项。

## 导师模块

### 指标卡

| 卡片 | 图标 | 数据来源 | 计算方式 |
|---|---|---|---|
| 导师总数 | `Users` | `professors` | `archived_at is null` 的导师数 |
| 已匹配导师 | `BadgeCheck` | `email_tasks.match_score` | 当前身份和模型下，`match_score is not null` 的导师数 |
| 高匹配导师 | `Star` | `email_tasks.match_score` | `match_score >= 高分阈值` 的导师数 |
| 高分未联系 | `UserRoundCheck` | `email_tasks.status` + `match_score` | 高分且状态不属于已排程、发送中、已发送、已回复的导师数 |

高分阈值优先使用当前身份的 `match_threshold`。如果为空，默认使用 80。

### 匹配分数分布

图表形式：柱状图。

分桶：

- 未分析
- 0-59
- 60-69
- 70-79
- 80-89
- 90-100

数据口径：

- 当前身份和模型下有 `email_tasks.match_score` 的导师进入分数桶。
- 没有匹配任务或 `match_score is null` 的导师进入“未分析”。

用途：

- 判断导师池整体匹配质量。
- 判断是否还有大量导师没有跑匹配分析。

### 资料完整度概览

图表形式：横向进度条列表。

项目：

- 有邮箱
- 有研究方向
- 有近期论文
- 有主页链接
- 完整资料

完整资料建议定义为：

```text
有邮箱 + 有研究方向 +（有近期论文 或 有主页链接）
```

### 学校分布 Top 10

图表形式：横向条形图。

数据来源：

- `professors.university`

用途：

- 判断导师池是否过度集中在少数学校。
- 帮助用户规划补充导师来源。

### 高分未联系导师

组件形式：行动列表。

筛选条件：

- `match_score >= 高分阈值`
- 状态不属于 `scheduled`、`sending`、`sent`、`reply_detected`

列表字段：

- 导师姓名
- 学校 / 学院
- 匹配分
- 当前状态
- 操作：打开工作区

### 待补全导师

组件形式：行动列表。

缺失项：

- 缺邮箱
- 缺研究方向
- 缺近期论文
- 缺主页链接

列表字段：

- 导师姓名
- 学校 / 学院
- 缺失字段标签
- 操作：打开导师管理页或工作区

## 邮件模块

### 指标卡

| 卡片 | 图标 | 数据来源 | 计算方式 |
|---|---|---|---|
| 已发送邮件 | `Send` | `email_tasks` | `sent_at is not null` 或状态为 `sent` / `reply_detected` |
| 已回复 | `Reply` | `email_tasks` | `is_replied = true` 或状态为 `reply_detected` |
| 回复率 | `Percent` | `email_tasks` | 已回复数 / 已发送数 |
| 发送失败 | `CircleAlert` | `email_tasks.status` | `status = send_failed`，辅助说明显示失败率 |
| 待审核草稿 | `FileCheck2` | `email_tasks.status` | `status = review_required` |
| 已排程邮件 | `CalendarClock` | `email_tasks.status` | `status = scheduled` |

### 近 30 天发送 / 回复 / 失败趋势

图表形式：折线图或分组柱状图。

第一版建议使用分组柱状图，三组数据分别为：

- 发送
- 回复
- 失败

数据来源：

- 发送：`email_tasks.sent_at`
- 回复：`email_logs.direction = received` 或 `email_tasks.status = reply_detected`
- 失败：`email_tasks.last_send_attempt_at` 或 `updated_at`，状态为 `send_failed`

### 邮件状态漏斗

图表形式：漏斗图或竖向阶段卡。

阶段：

- 已生成草稿
- 待审核
- 已批准
- 已排程
- 已发送
- 已回复

### 邮件任务状态分布

图表形式：横向条形图。

状态：

- 待匹配 / 已匹配
- 草稿生成中
- 草稿失败
- 待审核
- 已批准
- 已排程
- 发送中
- 已发送
- 发送失败
- 已回复
- 已取消

### 高价值待跟进邮件

组件形式：行动列表。

第一版可先实现以下三类：

- 高分发送失败
- 高分已发送未回复
- 高分待审核

列表字段：

- 导师姓名
- 学校 / 学院
- 匹配分
- 待跟进原因
- 最近更新时间
- 操作：打开工作区

frontend/src/lib/api/dashboardApi.ts

## 数据口径

### 当前身份和模型

以下统计必须按当前身份和模型过滤：

- 已匹配导师
- 高匹配导师
- 高分未联系导师
- 匹配分数分布
- 邮件指标卡
- 邮件趋势
- 邮件漏斗
- 邮件状态分布
- 高价值待跟进邮件

### 全局导师档案

以下统计基于当前导师库，不按身份和模型过滤：

- 导师总数
- 资料完整度概览
- 学校分布 Top 10
- 待补全导师

过滤条件：

```text
professors.archived_at is null
```

## 空状态和错误处理

- 没有导师时，导师模块显示空状态：“暂无导师数据，请先导入或新增导师。”
- 没有邮件任务时，邮件模块显示空状态：“暂无邮件触达数据。”
- 当前身份或模型未选择时，页面显示配置提示，引导用户到个人中心配置身份和模型。
- API 请求失败时显示错误提示和重试按钮。
- 某个图表数据为空时，该卡片保留标题并显示局部空状态。

## 测试计划

### 后端测试

- 验证 `/api/dashboard/overview` 在有效身份和模型下返回完整结构。
- 验证导师总数排除已归档导师。
- 验证已匹配导师只统计当前身份和模型下有匹配分的任务。
- 验证高匹配导师使用身份阈值，阈值为空时使用默认 80。
- 验证匹配分数分布包含未分析桶和各分数桶。
- 验证资料完整度统计邮箱、研究方向、近期论文、主页链接和完整资料。
- 验证邮件发送数、回复数、回复率、失败数、待审核和已排程统计。
- 验证近 30 天趋势按日期分桶。
- 验证高分未联系导师和高价值待跟进邮件排序。

### 前端测试

- 验证导航中出现“统计面板”入口，并能跳转 `/dashboard`。
- 验证未选择身份或模型时显示配置提示。
- 验证统计面板加载后展示导师模块和邮件模块标题。
- 验证指标卡显示主数字和辅助说明。
- 验证匹配分布、资料完整度、学校分布、邮件趋势、漏斗和状态分布能根据 API 数据渲染。
- 验证空数据状态不崩溃。
- 验证行动列表中的“打开工作区”链接指向正确导师。

## 验收标准

- 顶部导航新增“统计面板”，当前路由高亮行为与其他导航项一致。
- `/dashboard` 页面可正常加载，并跟随当前身份和模型展示统计。
- 页面不包含 Token 模块。
- 导师模块至少包含 4 个指标卡、3 个图表卡和 2 个行动列表卡。
- 邮件模块至少包含 6 个指标卡、3 个图表卡和 1 个行动列表卡。
- 所有指标都有明确数据口径，前端不依赖多个业务接口临时拼装统计。
- API 空数据、无身份模型、请求失败时都有可理解的页面反馈。
- 新增后端和前端测试覆盖关键统计口径与页面渲染。
