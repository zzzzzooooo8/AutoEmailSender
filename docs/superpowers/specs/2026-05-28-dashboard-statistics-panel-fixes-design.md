# 统计面板视觉与筛选修复设计

## 背景

统计面板第一版已经提供导师概览、资料完整度、学校分布和邮件触达等统计内容。当前需要针对用户审查中发现的三个体验问题做一次聚焦修复：

- 饼图颜色数量有限，分类数超过当前色盘长度后会循环复用颜色，容易造成图例和扇区混淆。
- 统计面板中的下拉筛选仍使用原生 `select`，和首页已经统一过的下拉框体验不一致。
- 导师概览顶部区域中，“资料完整度概览”卡片偏窄，右侧图表区域没有充分承担概览视觉重心。

本规格只覆盖统计面板当前已讨论的三项视觉与交互修复，不扩展新的统计指标或后端聚合逻辑。

## 目标

- 将通用饼图色盘扩展到 36 色，并保证相邻扇区颜色不相同、不高度相似。
- 统计面板中的下拉筛选尽量复用首页现有下拉框组件和交互逻辑。
- 调整导师概览顶部布局，让“资料完整度概览”更宽，并在桌面端高度对齐左侧两行内容。
- 保持现有筛选数据流、图表数据结构和后端接口不变。

## 非目标

- 不引入随机色作为最终方案，避免同一分类在不同渲染中颜色不稳定。
- 不把统计面板的学校、学院筛选改成多选筛选。
- 不重做统计面板整体信息架构。
- 不调整邮件触达、匹配分布、学校分布等统计口径。
- 不新增第三方图表库。

## 问题一：饼图色盘扩展

### 当前问题

`DistributionPieChart` 当前使用固定色数组，并通过 `index % colors.length` 为每个扇区分配颜色。当前色盘数量较少，当分类数量超过色盘长度时，相同颜色会重复出现。重复颜色如果出现在相邻扇区或临近图例中，会削弱图表的可读性。

### 设计方案

采用固定 36 色色盘，而不是运行时随机生成颜色。色盘需要人工挑选并按视觉差异打散排序：

- 避免黑色、白色、接近白色、接近黑色和低饱和灰色。
- 覆盖青、蓝、紫、粉、红、橙、黄绿、绿等多个色相。
- 不把同色系颜色连续排列。
- 保持颜色在浅色页面和白色图表卡片上有足够辨识度。

在分配颜色时保留循环策略，但增加相邻色保护：

- 默认按打散后的 36 色色盘顺序取色。
- 如果当前颜色和前一个扇区颜色相同或过于相似，则继续向后查找下一个可用颜色。
- 生成完成后检查最后一个扇区和第一个扇区，因为饼图首尾也是相邻关系。
- 如果分类数超过 36，允许复用色盘，但仍要避免相邻重复或高度相似。

### 相似度规则

实现中建议使用轻量的色彩距离判断：

- 完全相同的 hex 颜色视为冲突。
- 将 hex 转换为 HSL 后，如果色相差距过小，并且亮度差距也较小，则视为高度相似。
- 阈值不需要追求色彩科学精确，目标是避免肉眼明显混淆的相邻扇区。

### 代码组织建议

- 将 36 色色盘作为 `DistributionPieChart` 附近的常量保留，或抽到 `frontend/src/lib/charting.ts` 中。
- 将颜色分配逻辑封装成纯函数，例如 `assignPieSliceColors(count)` 或 `resolvePieSliceColor(index, previousColor, firstColor)`。
- `DistributionPieChart` 只消费最终颜色，不在 reduce 中直接写 `colors[index % colors.length]`。

### 验收标准

- 10 个以内分类继续保持清晰、稳定的颜色展示。
- 11 到 36 个分类不会出现简单循环复用导致的重复色。
- 超过 36 个分类时允许复用颜色，但相邻扇区不能使用相同或高度相似颜色。
- 图例圆点颜色和饼图扇区颜色一致。
- 测试覆盖颜色数量超过旧色盘长度的场景，以及首尾相邻颜色检查场景。

## 问题二：统计面板下拉框复用首页逻辑

### 当前问题

首页已经使用 `NativeSelectField` 和 `MultiSelectFilter` 统一了下拉框体验。统计面板中 `MentorFilterBar` 和 `EmailOutreachFilterBar` 仍直接使用原生 `select`，导致样式、弹出层、选中态、禁用态和焦点态与首页不一致。

### 设计方案

统计面板的单选下拉统一复用 `NativeSelectField`。本次不使用 `MultiSelectFilter`，因为统计面板当前筛选是单选联动：

- 导师概览：学校、学院。
- 邮件触达：时间范围、学校、学院。

保留现有交互逻辑：

- 学校改变后清空学院。
- 未选择学校时，学院下拉禁用并显示“请先选择学校”。
- 学校有值时，学院下拉默认项显示“全部学院”。
- 清空筛选按钮继续清空当前模块的学校和学院筛选。
- 邮件触达时间范围仍保持单选 preset，不改变筛选语义。

### 复用原则

- 优先直接使用 `NativeSelectField`，不要在统计面板内复制首页下拉框样式。
- 如需要减少重复，可以在 `DashboardPage.tsx` 内部抽一个很薄的 `DashboardFilterSelect`，只负责传递 `label`、`ariaLabel`、`value`、`disabled`、`options` 和 `onChange`。
- 不新增第二套下拉框基础组件。
- 不修改 `NativeSelectField` 的公共行为，除非实现时发现统计面板暴露出组件本身的通用缺陷。

### 视觉与交互要求

- 下拉触发器使用首页同款 `ui-select-shell`。
- 打开后使用自定义 listbox 浮层、选中项对勾和旋转箭头。
- 禁用态、hover 态、focus 态与首页保持一致。
- 长学校名、学院名需要保持单行截断，不能撑破筛选条。
- 选项过多时沿用 `NativeSelectField` 的滚动菜单行为。

### 验收标准

- 统计面板不再直接渲染用于筛选的原生 `select`。
- 导师概览和邮件触达筛选的视觉与首页下拉框一致。
- 学校、学院联动逻辑保持不变。
- 禁用学院下拉时仍可读、不可点击，并符合现有无障碍状态。
- 现有统计面板筛选测试继续通过，并补充或调整测试以覆盖 `NativeSelectField` 交互。

## 问题三：资料完整度概览布局调整

### 当前问题

导师概览顶部区域采用桌面端四列布局：

```text
左侧三列：筛选条
左侧三列：三个指标卡片
右侧一列：资料完整度概览，跨两行
```

当前右侧“资料完整度概览”列偏窄，饼图和横向图例展示空间不足。用户希望左侧内容略微压缩，让右侧资料完整度卡片更宽，并且让该卡片高度和左侧两行内容总高度一致。

### 设计方案

保留当前信息结构，只调整桌面端网格比例和卡片拉伸方式：

- 左侧仍保留筛选条和三个指标卡片。
- 右侧仍放置“资料完整度概览”，并继续跨越两行。
- 桌面端降低左侧三列的相对宽度，提高右侧列的相对宽度。
- 右侧卡片设置为填满跨行高度，使其顶部与筛选条顶部对齐，底部与指标卡片底部对齐。

建议将当前四列比例从接近：

```text
1fr 1fr 1fr 1.05fr
```

调整为更偏向右侧图表的比例，例如：

```text
0.85fr 0.85fr 0.85fr minmax(22rem, 1.45fr)
```

实际实现时可以根据视觉效果微调，但方向是让右侧列明显宽于单个指标卡。

### 响应式要求

- 大屏：保持左三列、右一列的概览结构，右侧资料完整度跨两行并高度对齐。
- 中屏：如果空间不足，可以回落为两列或单列布局，避免指标卡被过度压缩。
- 小屏：继续单列堆叠，资料完整度不强制跨行。
- 不为了桌面端宽度牺牲移动端可读性。

### 验收标准

- 桌面端“资料完整度概览”明显比当前更宽。
- 右侧资料完整度卡片高度与左侧筛选条加指标卡片两行高度一致或视觉上齐平。
- 左侧指标卡内容不换行到难以阅读，不出现文字溢出。
- 资料完整度饼图与图例不会因为宽度不足显得拥挤。
- 统计面板小屏布局仍为自然单列，不产生横向滚动。

## 测试与验证

实现后建议运行：

```bash
cd frontend && npm run test -- DashboardPage DistributionPieChart
cd frontend && npm run lint
```

如本地测试命令不支持按文件名筛选，则运行完整前端测试：

```bash
cd frontend && npm run test
```

还需要用浏览器人工检查：

- 统计面板导师概览顶部布局在桌面宽度下是否对齐。
- 统计面板下拉框是否和首页下拉框视觉一致。
- 多分类饼图颜色是否稳定、清晰，相邻扇区是否容易区分。
- 移动端统计面板是否无横向滚动和内容重叠。

## 开放问题

- 36 色色盘是否需要完全沿用当前前 10 个颜色，还是允许为整体协调性轻微调整当前颜色顺序。
- 资料完整度右侧列的最终比例以代码中的固定建议为准，还是以浏览器截图微调后的视觉效果为准。
